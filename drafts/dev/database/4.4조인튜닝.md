> 서브쿼리는 문법적으론 "쿼리 안의 쿼리"지만, 옵티마이저 입장에서 보면 결국 **조인의 한 형태**로 풀리는 경우가 대부분이다.

---

## 4.4 서브쿼리 조인

### 4.4.1 서브쿼리의 3가지 유형

| 유형 | 위치 | 예시 |
|---|---|---|
| **인라인 뷰** | `FROM` 절 | `FROM (SELECT ... FROM emp) x` |
| **중첩 서브쿼리** | `WHERE` 절 | `WHERE dept_id IN (SELECT id FROM dept WHERE ...)` |
| **스칼라 서브쿼리** | `SELECT` 절 | `SELECT name, (SELECT ... FROM dept WHERE ...) FROM emp` |

### 4.4.2 서브쿼리 Unnesting

> **Unnesting(서브쿼리 풀어내기)** = 서브쿼리를 조인으로 **평탄화**하는 옵티마이저의 쿼리 변환 기법.

```sql
-- 변환 전: 중첩 서브쿼리
SELECT * FROM emp e
WHERE dept_id IN (SELECT id FROM dept WHERE loc = 'SEOUL');

-- 변환 후 (Unnesting): 조인으로 평탄화
SELECT e.* FROM emp e, dept d
WHERE e.dept_id = d.id AND d.loc = 'SEOUL';
```

**왜 풀까?** 조인으로 바꾸면 옵티마이저가 **조인 순서 / 방식 (NL/Hash/Merge)을 자유롭게 선택**할 수 있어서 훨씬 유리해.

### 4.4.3 Unnesting이 안 되면?

필터(Filter) 방식으로 서브쿼리가 **바깥 쿼리의 각 행마다 실행**된다.

```
Outer 쿼리 행 1 → 서브쿼리 실행
Outer 쿼리 행 2 → 서브쿼리 실행
Outer 쿼리 행 3 → 서브쿼리 실행
...
```

→ **NL 조인의 최악 버전** (외부 N건 × 서브쿼리 비용). 서브쿼리에 인덱스가 없으면 재앙 수준.

### 4.4.4 중첩 서브쿼리: EXISTS vs IN

| 방식 | 의미 | 특징 |
|---|---|---|
| `IN (subquery)` | 값이 집합 안에 있는가 | 서브쿼리 결과 전체를 확보하려 함 |
| `EXISTS (subquery)` | 존재 여부만 확인 | **하나라도 찾으면 즉시 true** → 부분범위 처리에 유리 |

> 대용량 조회에서 "존재만 확인하면 되는" 경우 **`EXISTS` 가 유리**하다. 옵티마이저가 Semi-Join으로 풀기 좋음.

### 4.4.5 스칼라 서브쿼리와 캐싱

```sql
SELECT e.name, (SELECT d.name FROM dept d WHERE d.id = e.dept_id) AS dept_name
FROM emp e;
```

- 스칼라 서브쿼리는 **결과 캐시(Scalar Subquery Cache)** 를 활용
- 같은 입력값에 대해선 다시 쿼리하지 않음 → **중복 값이 많을수록 효율적**
- 단, 캐시 용량을 넘으면 반복 실행 → **카디널리티가 많으면 오히려 독**

### 4.4.6 인라인 뷰와 뷰 병합(View Merging)

```sql
-- 인라인 뷰
SELECT * FROM (
  SELECT dept_id, COUNT(*) cnt FROM emp GROUP BY dept_id
) x
WHERE x.cnt > 10;
```

옵티마이저는 조건만 맞으면 **뷰를 본쿼리와 병합**해서 더 자유롭게 최적화해. 다만 **뷰 내부에 집계/정렬이 있으면 병합 불가**한 경우가 있음.

---

## 📝 정리

- 서브쿼리는 옵티마이저의 **Unnesting**으로 조인이 되는 게 정상 경로
- Unnesting이 막히면 **필터(Filter) 방식** → 성능 재앙
- `EXISTS`는 존재 확인, `IN`은 집합 비교 — 의미와 성능 모두 다름
- 스칼라 서브쿼리는 **결과 캐시** 덕에 효율적이지만, 키 종류가 많으면 역효과
- 인라인 뷰는 뷰 병합(View Merging)이 되도록 집계·정렬을 꼭 필요한 경우에만
