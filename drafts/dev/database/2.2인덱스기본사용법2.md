## 🚀 DB 인덱스 스캔 방식 한눈에 보기

인덱스를 어떻게 요리하느냐에 따라 쿼리 속도가 천차만별이죠. 상황별로 어떤 방식이 쓰이는지 확인해 보세요!

### 1. 기본 중의 기본 스캔들
* **Index Range Scan (정석)**
    * 가장 일반적인 방식이에요. 루트에서 리프까지 수직으로 내려간 뒤, 필요한 **범위(Range)만** 슥 긁어오는 거죠.
    * **주의:** 인덱스 선두 컬럼을 가공(`WHERE SUBSTR(이름, 1, 1) = ...` 등)하면 이 방식을 못 써요!
* **Index Full Scan (차선책)**
    * 수직 탐색 없이 리프 블록을 **처음부터 끝까지** 다 읽는 거예요. 보통 적절한 인덱스가 없을 때 "에휴, 이거라도 다 읽자" 하고 선택되는 방식입니다.
* **Index Unique Scan (초고속)**
    * PK나 Unique 인덱스를 `=` 조건으로 찾을 때 발생해요. "딱 하나"만 찾으면 끝나니까 스캔할 범위도 없어서 성능이 최고죠.



### 2. 효율을 극대화하는 스캔들
* **Index Skip Scan (점프 스캔)**
    * 선두 컬럼 조건이 없어도 인덱스를 탈 수 있게 해줘요. 선두 컬럼의 종류(Cardinality)가 적고 후행 컬럼 종류가 많을 때, 조건에 맞는 리프 블록만 **퐁당퐁당 건너뛰며** 읽는 똑똑한 방식입니다.
* **Index Fast Full Scan (속도광)**
    * Index Full Scan보다 훨씬 빨라요. 인덱스의 논리적 순서는 무시하고 **물리적으로 한꺼번에(Multiblock I/O)** 다 읽어버리거든요. 단, 결과가 정렬되지 않는다는 점 주의!
* **Index Range Scan Descending (역순 스캔)**
    * Range Scan이랑 똑같은데 뒤에서부터 읽어요. `ORDER BY ... DESC` 같은 조건이 있을 때 인덱스 덕분에 별도 정렬 없이 결과를 바로 얻을 수 있죠.

## 📊 MySQL 실행계획(Execution Plan) 체크 포인트

쿼리가 내가 의도한 대로 도는지 확인하려면 `EXPLAIN`을 꼭 열어보세요. 이 항목들만 봐도 답이 나옵니다.

| 항목 | 의미 | 체크 포인트 ✅ |
| :--- | :--- | :--- |
| **type** | 조인/스캔 방식 | `system` > `const` > `range` 순으로 좋고, `ALL`(풀스캔)은 피해야 해요. |
| **key** | 실제 사용 인덱스 | 내가 의도한 인덱스가 들어가 있는지 확인! |
| **rows** | 예측 탐색 행 수 | 이 숫자가 너무 크면 인덱스가 제대로 일을 안 하고 있다는 뜻이에요. |
| **Extra** | 추가 정보 | `Using filesort`나 `Using temporary`가 뜨면 성능 빨간불! 🚨 |

### 💡 꿀팁 요약
인덱스를 탄다고 무조건 빠른 게 아니야. **"얼마나 적게 스캔하고, 테이블에 얼마나 덜 접근하느냐"** 가 진짜 실력이지. 실행계획의 `type`이 `range`나 `const`로 나오도록 쿼리를 잘 짜는 게 핵심!
