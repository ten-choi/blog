> NL 조인(4.1)이 **"하나씩 대조"** 였다면, 소트 머지 조인은 **"양쪽 다 줄 세운 뒤 한 번에 합치기"** 다.

---

## 4.2 소트 머지 조인 (Sort Merge Join)

### 4.2.1 소트 머지 조인 메커니즘

두 테이블을 **각자 조인 키 기준으로 정렬**한 다음, 양쪽을 **순차적으로 훑으며 매칭**하는 방식이야.

**진행 순서**

1. **Sort 단계** — A 테이블, B 테이블을 각각 조인 키로 정렬 (PGA의 Sort Area 사용)
2. **Merge 단계** — 양쪽 커서를 동시에 전진시키며 조인 키가 같은 행을 매칭

```
Table A (sorted by key)      Table B (sorted by key)
[1, 3, 5, 7]                 [1, 2, 3, 5, 9]
       ↓ 두 커서를 동시에 진행 ↓
Match: 1, 3, 5
```

### 4.2.2 NL 조인과의 차이

| 구분 | NL 조인 | 소트 머지 조인 |
|---|---|---|
| **조인 방식** | 한 건씩 대조 (랜덤 액세스) | 정렬 후 순차 매칭 |
| **선행/후행** | 선행 테이블이 중요 | 선행 개념 약함 (양쪽 다 정렬) |
| **인덱스 의존** | 높음 (후행에 인덱스 필요) | 낮음 |
| **사전 비용** | 작음 | 큼 (정렬 비용) |

### 4.2.3 소트 머지 조인이 유리한 경우

- **조인할 데이터가 많고**, **후행 테이블에 적절한 인덱스가 없을 때**
- **이미 정렬되어 있는 집합**을 조인할 때 (Sort 생략 가능)
- **비등가(non-equi) 조인** — `BETWEEN`, `>`, `<` 같은 조인 조건
  → 해시 조인은 등가 조인만 지원하기 때문에 이 구간에선 소트 머지가 강점

### 4.2.4 소트 머지 조인의 한계

- **정렬 비용이 큼** — 데이터가 많을수록 PGA 부담 (최악의 경우 디스크 Temp 사용)
- **메모리에서 정렬이 끝나지 않으면** 디스크 소트로 전환 → 체감 성능 급락
- 요즘은 대부분의 경우 **해시 조인이 더 빠름** → 소트 머지는 "등가 아닌 조인" 같은 틈새용

### 4.2.5 실무 포인트

> MySQL은 전통적으로 **소트 머지 조인을 지원하지 않는다** (오라클/PostgreSQL 기준 개념).
> MySQL에서 비슷한 효과를 내려면 **인덱스가 이미 정렬 순서를 제공**하도록 설계해서 불필요한 정렬을 피하는 전략이 중요.

## 📝 정리

- 소트 머지 조인 = **"양쪽 정렬 → 한 번에 훑기"**
- 후행 인덱스가 없을 때, 비등가 조인일 때 유리
- 실무에선 해시 조인에 밀려 드물게 쓰이지만, **동작 원리를 알아야 해시 조인의 강점이 눈에 들어온다**
