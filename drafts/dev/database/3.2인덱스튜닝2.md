## 3.3 인덱스 스캔 효율화

### 스캔 범위를 줄인다 = 리프 블록을 덜 읽는다

인덱스 레인지 스캔의 비용은 결국 **"몇 개의 리프 블록을 훑어야 하느냐"** 로 수렴해.
같은 결과를 내더라도 **인덱스의 시작점과 끝점을 얼마나 좁혔는지**에 따라 성능이 갈린다.

### (1) 액세스 조건 vs 필터 조건 — 가장 중요한 구분

| 구분 | 역할 | 예시 |
|---|---|---|
| **액세스 조건 (Access Predicate)** | 인덱스 스캔의 **시작/끝 지점을 결정** | `WHERE 부서 = 'A' AND 입사일 >= '2020-01-01'` |
| **필터 조건 (Filter Predicate)** | 스캔 중에 조건 맞는지 **검사만** | `WHERE 이름 LIKE '김%'` (인덱스가 아닌 컬럼) |

> 🔑 같은 결과라도 **액세스 조건으로 작동하면 스캔 범위가 줄어들고**, 필터 조건은 스캔 범위를 줄이지 못해.

### (2) 결합 인덱스의 컬럼 순서

인덱스 `(A, B, C)` 가 있을 때 조건에 따라 스캔 범위가 달라져.

```sql
-- ✅ A, B, C 모두 액세스 조건 → 가장 좁은 범위
WHERE A = 10 AND B = 20 AND C = 30

-- ⚠️ A, B는 액세스, C는 필터 (B가 범위 조건일 때)
WHERE A = 10 AND B >= 20 AND C = 30

-- ❌ A만 액세스, B/C는 필터 (A가 범위 조건일 때)
WHERE A >= 10 AND B = 20 AND C = 30
```

> **원칙**: **`=` 조건 컬럼을 앞쪽**에, **범위 조건(`>=`, `BETWEEN`, `LIKE`) 컬럼을 뒤쪽**에 배치한다.

### (3) IN-List와 인덱스 스캔

`IN`은 내부적으로 `=` 여러 개로 분해돼 **인덱스를 여러 번 타는 것**과 같아.

```sql
WHERE 부서 IN ('A', 'B', 'C')
-- = 부서='A' 또는 부서='B' 또는 부서='C'
-- → 인덱스 스캔을 3번
```

선두 컬럼이 `IN`이면 선두 컬럼을 `=`처럼 쓸 수 있기 때문에 후행 컬럼까지 액세스 조건으로 활용 가능해.

### (4) Between vs Like, 그리고 범위 조건의 이동

범위 조건을 뒤로 밀수록 앞 컬럼이 더 많이 액세스 조건으로 쓰인다.

```sql
-- ❌ 범위 조건이 중간에
WHERE 부서 = 'A' AND 입사일 BETWEEN '...' AND '...' AND 직급 = '과장'
→ 인덱스 (부서, 입사일, 직급) 이면 직급은 필터 조건

-- ✅ 인덱스 순서를 (부서, 직급, 입사일) 로 바꾸면 3개 모두 액세스 조건
```

## 3.4 인덱스 설계

### 인덱스는 공짜가 아니다

인덱스가 많으면 **조회는 빨라져도 DML(Insert/Update/Delete)이 느려진다.**
→ 인덱스가 추가될 때마다 입력·갱신·삭제 비용이 선형으로 늘어나.

### (1) 설계 원칙 요약

1. **액세스가 빈번한 쿼리 위주로 설계한다** — 하루 100만 번 도는 쿼리 > 한 달 1번 도는 배치
2. **`=` 조건 컬럼을 선두에** 둔다
3. **범위 조건 컬럼을 뒤쪽**에 둔다
4. **정렬(ORDER BY) 컬럼을 반영**해 Sort Order By 연산을 생략할 수 있게 한다
5. **인덱스 개수는 최소화** — 중복된 인덱스 / 선두 컬럼이 겹치는 인덱스는 통합 검토

### (2) 결합 인덱스 하나가 여러 단일 인덱스보다 낫다

```
(A) 인덱스, (B) 인덱스, (C) 인덱스 따로 만들지 말고
→ (A, B, C) 하나로 통합하면 대부분의 쿼리를 커버
```

### (3) 인덱스 병합(Index Merge) vs 결합 인덱스

옵티마이저가 두 단일 인덱스를 합쳐서 쓰는 Index Merge도 있지만,
**잘 만든 결합 인덱스 하나가 대부분의 경우 더 빠르다.**

### (4) 인덱스 컬럼의 NULL 처리

- 오라클 B-Tree 인덱스는 **NULL 값을 인덱스에 저장하지 않음**
  → `WHERE col IS NULL` 은 인덱스를 못 탐
- NULL 많은 컬럼은 함수 기반 인덱스(`NVL`)나 `IS NOT NULL` 활용을 고려

## 📝 정리

- 인덱스 스캔의 비용 = **리프 블록 읽는 개수** → 스캔 범위를 줄이는 게 본질
- **액세스 조건 ≫ 필터 조건** — 같은 `WHERE`여도 역할이 다르다
- 결합 인덱스 설계: **`=` → 범위 → 정렬** 순서
- 인덱스는 **조회 최적화 ⇆ DML 페널티**의 트레이드오프 — 무작정 많이 만들지 말자
