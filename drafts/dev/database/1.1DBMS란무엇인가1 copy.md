> Explaining DBMS for junior developers


When you think about it simply, as developers we write code like `SELECT * FROM table WHERE a = b;` every day. But how exactly does this query end up retrieving rows from a table in the database? Let's find out.


---

# 2. Internal Components of a DBMS

A DBMS processes SQL through the cooperation of the following components. If you look closely, excluding the first (input) and the last (storage), the **three components in the middle are the ones that actually do the "work"**.

| Component | Role |
|-----------|------|
| **Interface** | Receives the user's SQL command |
| **Parser** | Checks whether the command is syntactically and semantically valid, then turns the SQL into a parse tree |
| **Optimizer** | Takes the parse tree and builds the execution plan — the strategy for retrieving data in the fastest way |
| **Execution Engine** | Actually retrieves the data according to the optimizer's strategy |
| **Storage Engine** | Writes data to and reads data from disk/SSD |


So the three that matter are the parser, the optimizer, and the execution engine.
A database query flows from the interface all the way down to the storage engine.

That part is easy enough so far, right?

The interface is just where you type the SQL, so let's now look at parsing — one of the "big three".

## 3. SQL Parsing

> **Parsing order**: Syntax/semantic check → Hashing → Plan cache lookup → Soft parse vs. Hard parse


### ① Syntax Check & Semantic Check

At this step, three basic things are verified. If you've done any development, these will feel obvious.

- **Syntax check**: Are the keywords in the correct order?
- **Semantic check**: Do the referenced tables/columns actually exist?
- **Permission check**: Does this user have the right to access this data?

### ② Hashing & Plan Cache Lookup 🔑

This is the key stage that **"does more work than you'd expect"**.

- **Hashing**: The DBMS converts the entire input SQL statement into a single unique **hash value**.
  - e.g. `SELECT * FROM emp` → `7a8b9c...`
- **Plan cache lookup**: The DBMS searches an in-memory area where previously compiled execution plans are kept (generally called the **plan cache**) to see whether a plan with the same hash value is already stored. If it is? It can **reuse the already-built plan as-is** — a huge win.

### ③ The Moment of Decision: Soft Parse vs. Hard Parse

If it's in the cache, it's a **soft parse** (reuse); if not, it's a **hard parse** (build everything from scratch). The names say it all.

| Category | Situation | What Happens | Performance |
|----------|-----------|--------------|-------------|
| **Soft Parse** | The same SQL exists in the cache | Pulls and reuses the already-built execution plan | Very fast |
| **Hard Parse** | The same SQL is not in the cache | Runs the full optimization pipeline and builds a new execution plan from start to finish | Very slow (high CPU load) |

---

## 4. Why Go Through All This Complexity?

> Because the optimizer's job of building an execution plan (optimization) is **an extremely CPU-heavy operation**.

### What Actually Happens

- During a **hard parse**, the optimizer explores table/index statistics, column distributions (histograms), and every possible join combination to **compute the execution plan with the lowest cost**. Even a 5-table join blows up into 120 join orderings × join methods × index-usage combinations — easily thousands to tens of thousands of candidates, so **CPU consumption is massive**.
- A **soft parse** skips this calculation entirely and simply **pulls the execution plan already stored in the plan cache**. Since it only needs a hash comparison and a permission check, its CPU cost is typically an order of magnitude or two lower than a hard parse.

---

## 5. The Importance of Bind Variables

Developers use **bind variables** to reduce hard parsing.

### ❌ Bad Example (Triggers Hard Parsing)

```sql
WHERE id = 1
WHERE id = 2
```

> The literals differ, so the hash values differ too — the optimizer has to compute a new execution plan every single time.

### ✅ Good Example (Encourages Soft Parsing)

```sql
WHERE id = :id
```

> The statement text is identical, so the hash value is identical, and the cached plan can be reused.

---

## 6. Summary

- Syntax/semantic checks and hashing all exist for one reason: **to decide "can we skip the heavy optimization step (soft parse) and reuse a stored plan?"**
- No matter how well your indexes are designed, **if hard parsing is excessive, the whole database slows down.**
- That's why in the real world, **using bind variables is considered the foundation of SQL tuning.**
