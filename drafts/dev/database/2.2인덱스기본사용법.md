> MySQL 인덱스의 핵심만 뽑아 **"인덱스 사용 설명서"** 느낌으로 정리했어. 외우는 게 아니라 **원리로 이해**하는 게 포인트야.

---

## 1. 인덱스 사용의 핵심: "시작점을 찾는가?"

인덱스를 효율적으로 쓴다는 건 결국 **Index Range Scan**을 할 수 있느냐의 문제야.

* **Index Range Scan (베스트!):** 루트 노드부터 타고 내려가서 딱 필요한 **시작 지점**을 찾고, 끝 지점에서 멈추는 방식이야. 아주 빠르지!
* **Index Full Scan (차선책):** 시작 지점을 못 찾아서 인덱스의 처음부터 끝까지 다 뒤지는 거야. 테이블 전체를 다 읽는(Table Full Scan) 것보다는 낫지만, 효율적이지는 않아.



> **💡 여기서 잠깐! 랜덤 I/O 주의보**
> 인덱스에서 찾은 주소로 실제 데이터 파일을 읽으러 갈 때 **랜덤 I/O**가 발생해. 그래서 읽어야 할 데이터가 전체의 **20~25%**를 넘어가면, 차라리 인덱스 없이 테이블 전체를 읽는 게 더 빨라질 수 있어.

---

## 2. 인덱스 레인지 스캔을 방해하는 조건들

인덱스가 만들어져 있어도, 조건절이 다음과 같은 형태이면 옵티마이저가 Range Scan 의 시작점을 결정할 수 없어 인덱스를 활용하지 못한다.

* **선두 컬럼이 실종됨:** (소속팀, 이름, 나이) 순으로 인덱스가 잡혀있는데 `이름`으로만 검색하면? 시작점을 못 찾아서 풀 스캔이야. **무조건 인덱스의 첫 번째 컬럼이 조건절에 있어야 해!**
* **OR 조건:** `A 혹은 B`를 찾으라고 하면 어디서부터 스캔을 시작할지 애매해져서 인덱스 활용이 힘들어져. (이럴 땐 `UNION ALL`로 나누면 인덱스를 탈 수 있어!)
* **IN 조건:** `IN`도 사실상 여러 개의 `=` 조건을 합친 거라, 내부적으로는 하나씩 끊어서 인덱스를 타게 돼.

---

## 3. 인덱스가 주는 보너스: "정렬(Sort) 생략"

인덱스는 이미 **정렬된 상태**로 저장되어 있어. 이걸 잘 이용하면 엄청난 이득을 봐.

* **ORDER BY 생략:** 인덱스 순서대로 데이터를 가져오면 따로 정렬할 필요가 없어서 CPU를 아낄 수 있어.
* **MIN/MAX 최적화:** 제일 왼쪽(최소)이나 제일 오른쪽(최대) 값만 슥 보면 끝나니까 광속으로 처리돼.

---

## 4. 인덱스 사용 시 주의할 "금기 사항"

이건 정말 자주 하는 실수니까 꼭 기억해둬!

1.  **컬럼 가공 금지:** `WHERE YEAR(birth_date) = 1994` 처럼 컬럼을 함수로 감싸버리면 인덱스를 못 써. 인덱스에는 원본 값이 들어있거든.
2.  **자동 형변환(Type Mismatch):** 문자형 컬럼을 숫자랑 비교하면(`WHERE string_col = 123`), MySQL이 내부적으로 컬럼을 숫자로 바꾸느라 인덱스를 포기해버려. **타입은 꼭 맞춰주자!**
3.  **ORDER BY 가공 금지:** 정렬할 때도 인덱스 컬럼을 가공해서 정렬하라고 시키면 인덱스 효과를 못 봐.

## 5. 추가로 조심해야 할 조건 유형

위 "금기 사항" 외에도 **실무에서 자주 인덱스를 못 타게 만드는 조건 유형**이 몇 가지 더 있어.

### 5.1 NULL 조건

```sql
-- 인덱스: (status)
SELECT * FROM orders WHERE status IS NULL;     -- ❌ 오라클 B-Tree → 인덱스 못 탐
SELECT * FROM orders WHERE status IS NOT NULL; -- ⚠️ 대부분 Full Scan 유도
```

- **오라클 B-Tree 인덱스는 NULL 을 저장하지 않아** → `IS NULL` 조건은 인덱스 무용
- **MySQL InnoDB 는 NULL 을 저장**하긴 하지만, NULL 비중이 크면 옵티마이저가 인덱스를 버리는 경우가 많아
- 해법:
  - NULL 대신 **디폴트 값 (`'N'`, `0` 등) 사용**
  - NULL 검색이 꼭 필요하면 **함수 기반 인덱스** (`NVL(status, 'X')`) 고려
  - MySQL 이라면 **NOT NULL 제약 + 기본값** 으로 설계하는 게 일반적으로 유리

### 5.2 LIKE 의 전방 일치 vs 후방 일치

```sql
-- 인덱스: (name)
SELECT * FROM users WHERE name LIKE '김%';    -- ✅ 전방 일치 → 인덱스 Range Scan
SELECT * FROM users WHERE name LIKE '%김';    -- ❌ 후방 일치 → 시작점 결정 불가 → Full Scan
SELECT * FROM users WHERE name LIKE '%김%';   -- ❌ 중간 일치 → 당연히 Full Scan
```

- 인덱스는 **왼쪽부터 정렬**되어 있어서 **시작 문자가 정해져야** Range Scan 가능
- `'%abc'` 나 `'%abc%'` 는 시작점을 모르니 인덱스 무용
- 후방/중간 일치가 필수라면:
  - **Reverse Index** (`name` 을 뒤집어서 인덱스 걸기) → `'%김'` 이 `'김%'` 처럼 동작
  - **전문 검색 (Full-Text Search)** — MySQL 의 FULLTEXT 인덱스, 오라클의 Oracle Text
  - **검색 엔진 분리** — Elasticsearch 같은 별도 스택

### 5.3 부정형 조건

```sql
-- 인덱스: (status)
SELECT * FROM orders WHERE status != 'DONE';    -- ❌ 인덱스 활용 어려움
SELECT * FROM orders WHERE status NOT IN (...);  -- ❌ 마찬가지
SELECT * FROM orders WHERE NOT EXISTS (...);     -- ⚠️ 상황에 따라 다름
```

- **부정형 조건**은 "대부분의 행이 조건을 만족"하는 경우가 많아서 **옵티마이저가 Full Scan 을 선호**해
- 예: `status != 'DONE'` 은 DONE 이 아닌 **모든 상태** 를 포함 → 인덱스로 좁혀봐야 이득 없음
- 해법:
  - **긍정형으로 변환** — `status IN ('PENDING', 'SHIPPING')` 처럼 남은 값들을 명시
  - 가능한 상태 값이 적다면 **값 목록을 IN 조건**으로 풀기

### 5.4 범위 조건의 후행 컬럼 영향

```sql
-- 인덱스: (dept, hiredate, sal)
SELECT * FROM emp
WHERE dept = 10
  AND hiredate BETWEEN '2020-01-01' AND '2020-12-31'
  AND sal >= 3000;
```

- `dept = 10` → **액세스 조건** (등치)
- `hiredate BETWEEN ...` → **액세스 조건** (범위)
- `sal >= 3000` → **필터 조건** (hiredate 가 범위라 그 뒤는 스캔 범위 결정에 못 씀)

> ⚠️ **범위 조건이 중간에 끼면 그 뒤 컬럼들은 액세스 조건이 아니라 필터 조건이 돼**. 결합 인덱스 설계 시 **`=` 컬럼 → 범위 컬럼** 순서를 지켜야 하는 이유야.
