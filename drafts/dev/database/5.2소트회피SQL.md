# 5.2 소트를 없애거나 줄이는 SQL 작성

[5.1](./5.1소트튜닝.md) 에서 **소트는 기본적으로 무거운 연산**이라고 봤어. 이번 편은 **SQL 레벨에서 소트를 없애거나 가볍게 만드는** 기법이야.

---

## 5.2.1 Union vs Union All

```sql
-- Union        → 중복 제거를 위해 내부적으로 정렬 발생
-- Union All    → 단순 결합, 정렬 없음
```

- **Union**
  - 옵티마이저가 상단/하단 두 집합의 **중복을 제거**하기 위해 정렬을 수행
  - 데이터가 많으면 바로 디스크 소트로 빠질 수도 있음
- **Union All**
  - 중복 확인 없이 **단순 결합**
  - 두 집합이 **상호 배타적인 게 보장**된다면 Union All 이 정답

> 💡 "UNION 으로 만들었지만 두 집합이 절대 겹치지 않는다"면 **UNION ALL 로 바꾸는 것만으로 소트가 사라져**.

---

## 5.2.2 Distinct vs Exists

```sql
-- Distinct — 전체 데이터를 읽어서 중복 제거
SELECT DISTINCT 주문ID FROM 주문 WHERE ...;

-- Exists — 존재 여부만 확인
SELECT 주문ID FROM 주문 WHERE EXISTS (...);
```

- **DISTINCT**
  - 조건에 맞는 데이터를 **전부 읽어서** 중복 제거
  - 내부적으로 정렬(또는 해시) 필요 → I/O 와 소트 둘 다 부담
- **EXISTS**
  - **하나라도 찾으면 true** 로 끝나는 부분범위 처리
  - 정렬 불필요 + I/O 최소

> 💡 중복 제거가 **존재 여부 확인 용도**라면 EXISTS 로 바꿀 수 있는지 먼저 검토해.

## 5.2.3 GROUP BY 와 해시 그룹핑

예전 옵티마이저는 `GROUP BY` 를 무조건 **Sort Group By** 방식으로 풀었어 — 그룹 키로 **정렬한 뒤** 같은 값끼리 묶는 방식이라 **정렬 비용이 필수**로 발생.

최근 옵티마이저는 **Hash Group By** 를 지원해 — 그룹 키를 **해시 테이블**에 쌓으면서 집계하므로 **정렬이 불필요**해.

```
오라클 실행 계획 예시:
  HASH GROUP BY          ← 정렬 없음 (빠름)
  SORT GROUP BY          ← 정렬 발생
```

### 어떤 경우에 Hash Group By 가 선택되나?

- **GROUP BY + ORDER BY** 가 **같은 컬럼** 이 아닐 때 (정렬된 결과가 필요 없을 때)
- 그룹 수가 적을 때 (해시 테이블이 메모리에 들어갈 때)
- `NO_USE_HASH_AGGREGATION` 같은 힌트로 막지 않았을 때

### MySQL 8.0+

```
EXPLAIN 에서 "Using temporary" 가 뜨면 임시 테이블 기반 그룹핑
힌트로 제어: /*+ GROUP_INDEX(...) */, /*+ SET_VAR(optimizer_switch='hash_aggregation=on') */
```

## 5.2.4 Window 함수의 정렬 생략 조건

Window 함수(`ROW_NUMBER()`, `RANK()`, `SUM() OVER(...)` 등) 는 `PARTITION BY` 와 `ORDER BY` 에 따라 내부 정렬이 필요해.

### 정렬을 피할 수 있는 경우

1. **Window 절의 정렬 키와 동일한 인덱스가 존재**하는 경우 → 옵티마이저가 인덱스 순서를 그대로 씀
2. **PARTITION BY 만 있고 ORDER BY 없음** + **순서에 무관한 함수 (COUNT, SUM)** → 정렬 생략 가능
3. **여러 Window 함수가 모두 같은 PARTITION BY / ORDER BY** 를 공유 → 정렬 한 번만 수행

### 안티패턴

```sql
-- Window 함수마다 PARTITION BY 가 다르게 쓰임
SELECT
  ROW_NUMBER() OVER (PARTITION BY dept ORDER BY sal),
  RANK()       OVER (PARTITION BY job  ORDER BY hiredate),
  ...
```
→ **정렬이 두 번 발생**. 가능하면 Window 스펙을 통일하자.

## 5.2.5 Sort Area 를 적게 쓰는 SQL 작성

소트가 **불가피하다면**, 최소한 **메모리 소트 안에서 끝나도록** 유도해야 해. 핵심은 **"정렬 대상 데이터의 크기를 줄이는 것"**.

### 기법 1 — 서브쿼리로 가공 타이밍 늦추기

```sql
-- ❌ 나쁨: 가공한 결과를 정렬 — Sort Area 에 가공 후 문자열이 쌓임
SELECT UPPER(name) || '-' || dept AS label
FROM   emp
ORDER BY UPPER(name) || '-' || dept;

-- ✅ 좋음: 먼저 정렬하고 가공은 출력 시점에
SELECT UPPER(name) || '-' || dept AS label
FROM (
  SELECT name, dept FROM emp ORDER BY name, dept
);
```

- 내부 서브쿼리에서 **가공하지 않은 원본** 으로 정렬 → **Sort Area 사용량 감소**
- 최종 출력 단계에서만 가공 연산 적용

### 기법 2 — SELECT 에 필요한 컬럼만

```sql
-- ❌ 나쁨
SELECT * FROM emp ORDER BY sal;

-- ✅ 좋음
SELECT empno, name, sal FROM emp ORDER BY sal;
```

- 정렬 시 **SELECT 컬럼 전체를 Sort Area 에 저장** 하기 때문에 `*` 는 Sort Area 를 많이 쓰게 돼
- **필요한 컬럼만 지정** 하면 Sort Area 부담 감소

### 기법 3 — 필터를 먼저 적용

```sql
-- ❌ 나쁨: 10만 건을 다 정렬한 뒤 상위 10건만
SELECT * FROM (SELECT * FROM logs ORDER BY created_at DESC) WHERE ROWNUM <= 10;

-- ✅ 좋음: 인덱스 활용해 최신 10건만 읽고 끝
SELECT /*+ INDEX_DESC(logs idx_created) */ *
FROM   logs
WHERE  ROWNUM <= 10
ORDER BY created_at DESC;
```

→ 인덱스 활용한 정렬 생략은 [5.3 편](./5.3인덱스소트생략.md) 에서 자세히 본다.

## 5.2.6 🔑 이번 편 핵심 정리

| 상황 | 처방 |
|---|---|
| 중복 없는 Union | **UNION ALL** |
| 존재 여부만 필요 | DISTINCT → **EXISTS** |
| 정렬된 결과 불필요한 GROUP BY | **HASH GROUP BY** (옵티마이저에 맡기거나 힌트) |
| Window 함수 여럿 | **Window 스펙 통일**로 정렬 횟수 최소화 |
| 소트가 불가피 | **컬럼 최소화 + 필터 먼저 + 가공 미루기** |

> 다음 편(5.3) 에선 **인덱스 자체가 이미 정렬된 상태**라는 점을 활용해, ORDER BY 연산을 아예 생략하는 기법을 본다.
