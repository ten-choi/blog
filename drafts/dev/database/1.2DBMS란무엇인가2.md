## 1. 복습: DBMS 의 내부 구성 요소

DBMS 는 아래 컴포넌트들이 협력해서 SQL 을 처리해. 이 중 첫 번째(입력)와 마지막(저장소)을 빼면 **실제로 "일"을 하는 건 가운데 3개**라고 볼 수 있어.

| 구성 요소 | 역할 |
|-----------|------|
| 인터페이스 | 사용자의 SQL 명령을 받음 |
| **파서 (Parser)** | 명령어가 문법에 맞는지 검사 |
| **옵티마이저 (Optimizer)** | 데이터를 가장 빠르게 찾을 전략을 수립 |
| **실행 엔진 (Execution Engine)** | 옵티마이저의 전략대로 실제 데이터를 조회 |
| 스토리지 엔진 | 디스크/SSD에 실제 데이터를 쓰고 읽음 |

---

## 2. 옵티마이저는 언제 동작하는가?

소프트 파싱에 성공한 경우엔 **옵티마이저가 동작하지 않아.** 라이브러리 캐시에 이미 저장되어 있는 실행 계획을 그대로 재사용하거든.

라이브러리 캐시에서 해시 값이 일치하는 실행 계획을 찾지 못했을 때 (**하드 파싱**) 만 옵티마이저가 호출돼서, 통계 정보와 쿼리 조건을 바탕으로 **새 실행 계획을 계산**해. 이 계산은 CPU 소비가 커서, 실제 SQL 성능에선 **하드 파싱 빈도가 꽤 중요한 지표**로 작용해.

---

## 3. 🛠️ 옵티마이저의 3단계 작업 공정

옵티마이저가 하드 파싱 단계에서 수행하는 핵심 작업은 크게 **3단계**로 나뉘어.

> **작업 순서**: 쿼리 변환 → 비용 예측 → 계획 생성

### ① 쿼리 변환 (Query Transformer)

사용자가 작성한 SQL 을 그대로 실행하는 게 아니라, **옵티마이저가 더 좋은 실행 계획을 만들 수 있는 형태로 SQL 문장 자체를 재작성**해.

**왜 재작성이 필요할까?**
- 사용자가 작성한 원본 SQL 형태로는 옵티마이저가 선택할 수 있는 실행 경로가 제한적일 때가 많아
- 논리적으로 동등하지만 구조가 다른 SQL 로 바꾸면 **더 많은 조인 순서·조인 방식·인덱스 활용 경우의 수**를 탐색할 수 있거든

**대표적인 변환 예시**

- 서브쿼리를 조인(JOIN) 형태로 변환
- 불필요한 조건 제거
- 뷰(View) 병합

---

### ② 비용 예측 (Estimator)

**통계 정보를 바탕으로** 각 후보 실행 계획이 소비할 자원량을 계산해.

- **Cardinality(카디널리티)**: 각 단계에서 **반환될 행의 개수 추정치**야. 이 추정이 틀리면 이후 조인 방식 선택까지 전부 틀어져.
- **Cost(비용)**: 해당 단계를 실행하는 데 드는 **논리적 I/O + 물리적 I/O + CPU** 를 하나의 정수값으로 환산한 것. 옵티마이저는 이 값이 **가장 낮은 후보**를 최종 선택해.

비용 계산 자체는 **Single Block I/O 시간, Multiblock I/O 시간, CPU 속도** 같은 **시스템 통계**를 기준으로 정량적으로 이뤄져.

**참고하는 데이터**

- 테이블의 행(row) 수
- 블록(block) 수
- 컬럼 값의 분포 (히스토그램)
- 인덱스 유무

---

### ③ 계획 생성 (Plan Generator)

실제로 실행 가능한 수많은 '경우의 수'를 조합해서 **최적의 경로를 최종 결정**해.

| 결정 항목 | 선택지 |
|-----------|--------|
| **조인 순서** | A 테이블을 먼저? B 테이블을 먼저? |
| **액세스 경로** | 인덱스 스캔? 풀 스캔(Full Scan)? |
| **조인 방식** | NL Join / Hash Join / Merge Join |

---

### ④ 로우 소스 생성 (Row Source Generator)

옵티마이저가 결정한 실행 계획은 아직 **추상적인 연산 트리** 상태야. 로우 소스 생성기는 이 연산 트리를 **실행 엔진이 직접 호출 가능한 연산자들의 집합 (Row Source Tree)** 으로 변환해.

- 각 노드는 `TABLE ACCESS BY INDEX ROWID`, `INDEX RANGE SCAN`, `HASH JOIN`, `SORT` 같은 **실제 연산자 단위**로 표현돼
- `EXPLAIN PLAN` 출력의 `Operation`, `Object Name`, `Options` 컬럼이 바로 이 단계에서 완성된 결과물이야
- 실행 엔진은 이 트리를 루트부터 순회하면서 각 연산자를 호출하고, 그 결과를 상위 연산자로 넘겨

---

## 4. 🧐 왜 "CPU를 많이 잡아먹는다"고 할까?

옵티마이저가 검토해야 할 실행 계획의 후보군이 **기하급수적으로 늘어나기** 때문이야.

### 예시: 테이블 5개를 조인한다면?

- 테이블을 읽는 순서의 가짓수: **5! = 120가지**
- 여기에 각 테이블마다 인덱스 사용 여부 × 조인 방식 조합
- → 후보군이 순식간에 **수천, 수만 개** 로 폭증

이 수만 개의 경우의 수마다 **비용 계산 로직**이 반복 수행되기 때문에, 하드 파싱 1회의 CPU 소비가 일반적인 SELECT 1회 실행보다 **수십~수백 배 높아**.

---

## 5. 💡 요약: 그래서 옵티마이저는 뭘 만드나?

결국 옵티마이저가 하는 일은 단 하나야.

> **비용이 가장 낮은 실행 계획(Execution Plan)을 선택하는 것.**

선택된 실행 계획은 로우 소스 생성기로 전달돼서, 실행 엔진이 호출할 수 있는 **연산자 트리**로 변환돼.

---

## 6. 📝 한 줄 노트: 로우 소스 생성기의 위치

우리가 SQL을 실행했을 때 실행계획에 나오는 `Operation`, `Object Name`, `Options` 등이 바로 **로우 소스 생성기가 완성한 산출물**이야.

엄밀히 따지면 로우 소스 생성기는 '옵티마이저'의 일부는 아니지만, **소프트 파싱을 해도 이 단계를 거친 정보를 가져다 쓰기 때문에** 파싱/최적화 흐름의 일부로 같이 묶어두는 게 이해하기 좋아.
