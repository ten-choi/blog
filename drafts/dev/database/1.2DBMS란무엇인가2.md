> 세상에서 가장 쉬운, Junior 개발자를 위한 SQL Tuning 기초 정리
>
> 오늘은 **옵티마이저가 하드 파싱 단계에서 구체적으로 어떤 일을 하는지** 알아봅니다.

## 1. 복습: DBMS의 내부 구성 요소

DBMS는 다음 컴포넌트들이 협력하여 SQL을 처리합니다. 이 중 첫 번째(입력)와 마지막(저장소)을 제외하면 **실제로 "일"을 하는 것은 가운데 3개**라고 볼 수 있죠.

| 구성 요소 | 역할 |
|-----------|------|
| 인터페이스 | 사용자의 SQL 명령을 받음 |
| **파서 (Parser)** | 명령어가 문법에 맞는지 검사 |
| **옵티마이저 (Optimizer)** | 데이터를 가장 빠르게 찾을 전략을 수립 |
| **실행 엔진 (Execution Engine)** | 옵티마이저의 전략대로 실제 데이터를 조회 |
| 스토리지 엔진 | 디스크/SSD에 실제 데이터를 쓰고 읽음 |

---

## 2. 옵티마이저는 언제 출근하는가?

> 소프트 파싱에 성공하면 옵티마이저는 그대로 **"휴가"** 를 갑니다. 할 일이 없거든요. (재사용, 개꿀!)

하지만 라이브러리 캐시에서 내비게이션 기록을 찾지 못했을 때(**하드 파싱**), 비로소 이 **"비싼 몸"** 인 옵티마이저가 출근해서 본격적인 두뇌 풀가동을 시작합니다.

---

## 3. 🛠️ 옵티마이저의 3단계 작업 공정

옵티마이저가 하드 파싱 단계에서 수행하는 핵심 작업은 크게 **3단계**로 나뉩니다.

> **작업 순서**: 쿼리 변환 → 비용 예측 → 계획 생성

### ① 쿼리 변환 (Query Transformer)

사용자가 작성한 SQL을 그대로 실행하는 것이 아니라, **DBMS가 처리하기 더 쉬운 형태로 문장을 다시 씁니다.**

> **💬 비유**
> 사용자가 *"나 어제 갔던 거기 갈래"* 라고 하면,
> 옵티마이저가 *"강남역 2번 출구 맛집 말씀이시죠?"* 라고 명확하게 바꾸는 과정.

**대표적인 변환 예시**

- 서브쿼리를 조인(JOIN) 형태로 변환
- 불필요한 조건 제거
- 뷰(View) 병합

---

### ② 비용 예측 (Estimator)

가장 머리 아픈 단계입니다. **통계 정보를 바탕으로** 각 단계에서 예상 비용을 계산합니다.

- **Cardinality(카디널리티)**: 얼마나 많은 데이터가 나올지
- **Cost(비용)**: 얼마나 많은 자원이 들지

> **💬 비유**
> *"이 길은 5km인데 신호등이 10개고, 저 길은 7km인데 고속도로야. 어디가 더 빠를까?"* 를 계산하는 것.

**참고하는 데이터**

- 테이블의 행(row) 수
- 블록(block) 수
- 컬럼 값의 분포 (히스토그램)
- 인덱스 유무

---

### ③ 계획 생성 (Plan Generator)

실제로 실행 가능한 수많은 '경우의 수'를 조합하여 **최적의 경로를 최종 결정**합니다.

| 결정 항목 | 선택지 |
|-----------|--------|
| **조인 순서** | A 테이블을 먼저? B 테이블을 먼저? |
| **액세스 경로** | 인덱스 스캔? 풀 스캔(Full Scan)? |
| **조인 방식** | NL Join / Hash Join / Merge Join |

---

### ④ 로우 소스 생성 (Row Source Generator)

옵티마이저가 결정한 실행 계획을 **실제로 실행 가능한 코드(Row Source Tree)** 로 변환하는 마지막 단계야.

> **💬 비유**
> 건축 설계도(실행 계획)를 받아서, 현장 작업자들이 바로 이해할 수 있는 **조립 설명서**로 바꿔주는 과정.

이 조립 설명서가 바로 다음 단계인 **실행 엔진(Execution Engine)** 이 한 줄씩 읽어가며 실제로 데이터를 퍼 올리는 근거가 돼.

---

## 4. 🧐 왜 "CPU를 많이 잡아먹는다"고 할까?

옵티마이저가 검토해야 할 실행 계획의 후보군이 **기하급수적으로 늘어나기** 때문입니다.

### 예시: 테이블 5개를 조인한다면?

- 테이블을 읽는 순서의 가짓수: **5! = 120가지**
- 여기에 각 테이블마다 인덱스 사용 여부 × 조인 방식 조합
- → 후보군이 순식간에 **수천, 수만 개** 로 폭증

이 수만 개의 경우의 수마다 **"비용 계산"** 을 일일이 수행해야 하니, CPU가 뜨거워질 수밖에 없는 거죠. 🔥

---

## 5. 💡 요약: 그래서 옵티마이저는 뭘 만드나?

결국 옵티마이저가 하는 일은 단 하나.

> **가장 싸고 빠른 '실행 계획(Execution Plan)' 이라는 설계도를 뽑아내는 것.**

이 설계도가 완성되면 다음 단계인 **'로우 소스 생성기(Row Source Generator)'** 로 넘겨서 실제로 데이터를 퍼 올릴 준비를 하게 됩니다.

---

## 6. 📝 한 줄 노트: 로우 소스 생성기의 위치

우리가 SQL을 실행했을 때 실행계획에 나오는 `Operation`, `Object Name`, `Options` 등이 바로 **로우 소스 생성기가 완성한 산출물**이야.

엄밀히 따지면 로우 소스 생성기는 '옵티마이저'의 일부는 아니지만, **소프트 파싱을 해도 이 단계를 거친 정보를 가져다 쓰기 때문에** 파싱/최적화 흐름의 일부로 같이 묶어두는 게 이해하기 좋아.
