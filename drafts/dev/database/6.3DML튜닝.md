# 6.3 파티션을 활용한 DML 튜닝


### 파티션이란
- 하나의 테이블 또는 인덱스를 **파티션 키 기준으로 쪼개서 별도 세그먼트에 저장**하는 거야
- 논리적으로는 하나의 테이블처럼 보이지만, **물리적으로는 여러 테이블에 나눠 담은 효과**를 내

### 파티션이 필요한 이유
- **관리적 측면**: 파티션 단위로 작업하면 같은 테이블이라도 **다른 파티션엔 영향 없음** → 가용성 향상
- **성능적 측면**: 경합 완화, 부하 분산

## 6.3.1 테이블 파티션

### (1) 파티션 테이블의 성능 향상 원리
* 파티션 테이블의 성능 향상 원리는 **파티션 pruning** 이야. SQL 파싱이나 실행 시점에 조건절을 분석해서 **해당 없는 파티션은 액세스 대상에서 아예 제외**해버려.
* 예를 들어 파티션이 없는 테이블에 1000만 건이 있다면 **1000만 건 전체에서 조건 필터링**을 해야 해.
* 반면 파티션이 적용된 경우엔 **조건에 해당하는 파티션만 스캔**하고 그 안에서 필터링하니까 성능 이득이 크지.

### (2) Range 파티션
* 날짜 컬럼처럼 **범위 값을 기준**으로 파티셔닝하는 방식
* **보관 주기 정책**에 따라 관리가 달라지는 경우에 효율적이야 (예: 1년만 보관하고 백업하는 로그 테이블)

### (3) 해시 파티션
* **파티션 개수**만 사용자가 정하고, 데이터 분산 알고리즘은 **오라클 내부 해시 함수**가 결정해
* 해시 알고리즘 특성상 **등치 조건(=) 또는 In-List 조건**에서만 파티션 Pruning 이 작동해

### (4) 리스트 파티션

* 해시 파티션이 "자동 분할" 이라면, 리스트 파티션은 **사용자가 정의한 논리적 그룹에 따라 분할**하는 방식이야
* 예를 들어 지역별로 파티션 키를 설정하면 '서울', '경기', '부산' 같은 **논리적 기준**으로 파티셔닝돼

### 인덱스 파티션

### (1) 로컬 파티션 인덱스
* **각 테이블 파티션과 1:1 로 대응**되는 인덱스 파티션이야
* 예를 들어 테이블이 월별로 파티셔닝되어 있다면, 각 파티션마다 그에 대응하는 인덱스가 만들어져
* 인덱스 파티션은 **테이블 파티션의 속성을 그대로 상속**받아. 따라서 `테이블 파티션 키 == 인덱스 파티션 키`가 돼.
* 파티션 테이블이 변경되면 로컬 파티션 인덱스도 **오라클이 자동으로 맞춰서 변경**해줘

> quiz
```
파티셔닝 되지 않은 테이블의 경우 로컬 파티션 인덱스가 존재 할까요?
```

> ✅ **정답: 존재할 수 없어.**
>
> 로컬 파티션 인덱스는 정의상 **"각 테이블 파티션과 1:1 로 대응되는 인덱스 파티션"** 이거든.
> 테이블이 파티셔닝되지 않았다면 **대응할 파티션 자체가 없으니까** 로컬 파티션 인덱스라는 개념이 성립하지 않아.
> 이 경우 만들 수 있는 건 **비파티션 인덱스(일반 인덱스)** 뿐이야.

### (2) 글로벌 파티션 인덱스

* 파티션을 **테이블과 다르게 구성**한 인덱스야. 로컬 인덱스가 아니면 **전부 글로벌 파티션 인덱스**라고 보면 돼.
* 파티션 유형이 다르거나, 파티션 키가 다르거나, 파티션 기준값 정의가 다른 경우가 여기 해당해.
* 테이블 파티션 구성을 변경하면 **Unusable 상태**로 바뀌어서 **재생성해야 해**.

### (3) 비파티션 인덱스

* 파티셔닝되지 않은 일반 인덱스
* 비파티션 인덱스가 파티션 테이블과 조합될 때는, **테이블에 변경이 있으면 인덱스를 재생성해야 해**.

## 6.3.3 파티션을 활용한 대량 update 튜닝

10억 건의 데이터가 저장된 테이블에서 **5% (5000만 건)** 의 데이터 변경이 있을 경우를 보자.

### 1) 5000만 건을 그냥 update
→ 5000만 건에 대한 select/update + **인덱스 수정 부담**까지 발생

### 2) index 를 drop 하고 5000만 건을 update
→ update 자체는 빨라지지만 **10억 건에 대한 인덱스 재생성 부담**이 생김

### 3) 파티션 exchange 를 이용
* **테이블이 파티셔닝되어 있고 인덱스도 로컬 파티션**으로 구성되어 있을 때 쓸 수 있어
```sql
(1) 임시 테이블을 만들고
(2) 임시 테이블에 update 될 데이터를 insert
(3) 임시 테이블에 인덱스를 생성
(4) 기존 파티션 테이블과 임시 테이블을 exchange
(5) 임시 테이블 drop
```

## 6.3.4 파티션을 활용한 대량 delete 튜닝

update 는 **변경 대상 컬럼을 포함하는 인덱스만 재생성**하면 되지만, delete 는 **모든 인덱스를 재생성해야 해서** 인덱스 재생성 비용이 훨씬 커.

### 1) 파티션 drop 을 이용한 대량 삭제

* 삭제 조건절 컬럼을 기준으로 파티셔닝되어 있고 로컬 인덱스인 경우, 아주 간단히 삭제 가능
* 예를 들어 월별로 파티셔닝되어 있고 1월 파티션을 통째로 삭제하는 경우
```sql
alter table 거래 drop partition p201412;
```

### 2) 파티션 truncate 를 이용한 대량 삭제
* 파티셔닝 기준으로 일괄 삭제하지 않고 **또 다른 삭제 조건**이 있을 때 쓰는 방식
* 지워야 할 데이터가 대다수라면, **"지울 데이터를 삭제"** 하는 것보다 **"전부 날리고 남길 데이터만 다시 insert"** 하는 게 빨라
```sql
(1) 임시 테이블 생성 후 남길 데이터만 insert
(2) 삭제 대상 테이블 파티션을 truncate
(3) 임시 테이블에 복제한 데이터를 원본 테이블로 insert (insert select 사용)
(4) 임시 테이블 drop
```

## 6.3.5 파티션을 활용한 대량 insert 튜닝

### 1) 비파티션 테이블일 때
* 대량 데이터를 insert 할 때 **인덱스를 unusable 시킨 뒤 재생성**하는 방식이 더 빠를 수 있어

### 2) 파티션 테이블일 때
* 테이블이 파티셔닝되어 있고 로컬 인덱스라면 **파티션 단위로 인덱스를 재생성**할 수 있어 → 전체 재생성 비용 회피
