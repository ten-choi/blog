## 3장. 인덱스 튜닝

### 3.1 테이블 액세스 최소화

- **인덱스 ROWID**

  - 인덱스 ROWID 는 **논리적 주소**야
  - 메모리 상의 **물리적 포인터와는 달라**
  - ROWID 를 이용한 테이블 액세스는 **생각보다 고비용 연산**이야

- 특정 컬럼 기준으로 **같은 값을 갖는 데이터가 물리적으로 가까이 모여 있을수록** (= 클러스터링 팩터가 좋은 컬럼에 인덱스를 만들면) 데이터를 찾는 속도가 빨라져

- 읽어야 할 데이터가 일정량을 넘는 순간 (= **인덱스 손익분기점**을 넘는 순간) **Table Full Scan 보다 Index Range Scan 이 오히려 느려져**

  - Table Full Scan 은 **시퀀셜 액세스**인데, 인덱스 ROWID 를 이용한 테이블 액세스는 **랜덤 액세스**이기 때문
  - Table Full Scan 은 **Multiblock I/O**인데, 인덱스 ROWID 를 이용한 테이블 액세스는 **Single Block I/O**이기 때문

- **인덱스 클러스터링 팩터 (CF)** 에 따라 인덱스 손익분기점이 달라져

  - CF 가 좋을수록 인덱스 손익분기점이 **뒤로 밀려**
  - 즉 **물리적으로 가까이 저장된 컬럼을 기준으로 인덱스를 만들면**, 많은 데이터를 Index Range Scan 으로 가져와도 Table Full Scan 보다 느려질 가능성이 적어져

- 기존 인덱스에 **컬럼을 추가**하면 테이블 액세스를 줄일 수 있어

  - 예를 들어 `(a, b)` 컬럼으로 이루어진 인덱스가 있는 테이블에서 아래 쿼리를 돌린다고 해보자

  - ```sql
    select *
    from table_name
    where a = 30 and c >= 2000;
    ```

  - 기존 인덱스에 `c` 컬럼을 추가해서 `(a, b, c)` 로 만들면 **인덱스 스캔 양은 그대로**지만 **테이블 액세스 양을 줄일 수 있어**

- 위 경우는 `where` 로 걸러지는 양이 많을 때의 튜닝이야. 그럼 **걸러지는 레코드가 거의 없는 경우**엔?

  - 예를 들어 아래 쿼리는 걸러지는 레코드가 거의 없어 (부서번호로 인덱스 생성되어 있다고 가정)

  - ```sql
    select 부서번호, sum(수량)
    from 판매집계
    where 부서번호 like '12%'
    group by 부서번호;
    ```

  - 이럴 땐 **쿼리에 사용된 컬럼(수량)까지 인덱스에 모두 포함**시켜서 테이블 액세스가 아예 발생하지 않게 튜닝할 수 있어

  - 이렇게 **모든 필요 컬럼이 인덱스에 포함되어 있어서 인덱스만 읽고 처리되는 쿼리**를 **Covered 쿼리**라고 하고, 그 인덱스를 **Covered 인덱스**라고 불러

  - 효과는 좋은데, **추가할 컬럼이 많아지면 현실적으로 어려워지는** 경우도 많아

### 3.2 부분범위 처리 활용

- **부분범위 처리**를 활용하면 인덱스로 액세스할 대상 레코드가 아무리 많아도 **아주 빠른 응답 속도**를 낼 수 있어
- 처음부터 전부 가져오지 않고, **일정량(array size)만 요청(fetch call)** 해서 처리하는 방식이야
- "멈출 수 있어야" 의미 있는 부분 범위 처리야
  - 전체를 읽어서 정렬해야 하는 작업이 있다면 **앞쪽 일부만 출력할 수 없거든**
  - **앞쪽 일부만 읽고 반환해도 되도록 인덱스를 구성**해야 돼

#### Top-N 쿼리 패턴

"**최근 / 최고 / 인기 N건**" 유형의 쿼리가 부분범위 처리의 전형적 활용 사례야.

```sql
-- 최근 주문 20건
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20;                                   -- MySQL / PostgreSQL

SELECT * FROM orders
ORDER BY created_at DESC
FETCH FIRST 20 ROWS ONLY;                   -- 오라클 12c+

SELECT * FROM (
  SELECT * FROM orders ORDER BY created_at DESC
) WHERE ROWNUM <= 20;                       -- 오라클 구버전 (Inline View 필수)
```

- 인덱스 `(created_at)` 이 있으면 옵티마이저는 **인덱스 역순 스캔** 으로 **앞 20건만 읽고 중단**
- 전체 건수가 1만 건이든 1억 건이든 **응답 시간이 거의 일정**

#### Pagination — OFFSET 안티패턴 주의

페이지네이션은 실무에서 제일 자주 틀리는 케이스야.

```sql
-- ❌ 안티패턴 (OFFSET 기반)
-- 100 페이지 (20건씩): OFFSET 2000 LIMIT 20
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 2000;
```

- OFFSET 2000 은 **앞 2000건을 읽어서 버린 뒤** 그 다음 20건을 반환
- 페이지가 깊어질수록 **읽어서 버리는 양이 선형으로 증가** → 뒷 페이지가 느려짐
- 100 페이지면 참을만해도 **1000 페이지면 재앙**

```sql
-- ✅ 커서 기반 페이징 (Keyset Pagination)
-- 다음 페이지 요청 시 "이전 페이지 마지막 created_at 값"을 조건으로
SELECT * FROM orders
WHERE  created_at < :last_seen_created_at
ORDER BY created_at DESC
LIMIT 20;
```

- 매 페이지가 **앞 건수와 무관하게 일정 성능**
- 인덱스 하나로 모든 페이지 커버 가능
- 단점: **임의의 N 페이지로 점프 불가** — 순차 이동만 됨 (무한 스크롤/피드에 최적)

#### 부분범위 처리가 작동하지 않는 경우

```sql
-- ❌ ORDER BY 로 전체 정렬이 필요 → 앞 N 건만 읽을 수 없음
SELECT * FROM orders ORDER BY total_amount DESC LIMIT 20;
-- (total_amount 에 인덱스가 없으면 전체를 정렬해야 함)

-- ❌ GROUP BY / 집계가 들어가면 앞에서 멈출 수 없음
SELECT dept, SUM(sal) FROM emp GROUP BY dept LIMIT 5;

-- ❌ DISTINCT 도 전체 스캔 필요
SELECT DISTINCT category FROM products LIMIT 10;
```

- 집계·정렬·중복 제거가 들어가면 **"앞만 보고 멈춘다"가 성립 안 됨**
- 이런 경우엔 **인덱스 설계**로 집계/정렬을 생략하거나, **결과 테이블을 별도로 유지**하는 전략이 필요
