# 5.1 소트 수행 원리

소트(Sort)는 **CPU 와 메모리를 많이 쓰는 무거운 연산**이야. SQL 튜닝에서 소트 관련 이슈가 나오면 보통 세 가지 중 하나를 의심하게 돼 — **"왜 소트가 생겼나"**, **"이걸 없앨 수 있나"**, **"없앨 수 없다면 어떻게 싸게 하나"**.

이번 편은 **맨 밑바닥 원리**를 보자. 어디서 일어나고, 어떻게 돌아가는지.

---

## 5.1.1 소트가 일어나는 순간

옵티마이저는 아래 같은 조건을 만나면 실행 계획에 **Sort 연산자**를 넣어:

- `ORDER BY`
- `GROUP BY` (해시 기반 그룹핑이 불가하거나 비효율적일 때)
- `DISTINCT`
- `UNION` (중복 제거용)
- **Sort Merge Join** 의 정렬 단계
- **Window 함수** 의 `PARTITION BY` / `ORDER BY`
- **Index Build** 시 (인덱스 생성할 때 키 순으로 정렬)

---

## 5.1.2 소트가 일어나는 장소 — PGA 의 Sort Area

소트는 기본적으로 **PGA(Program Global Area)** 에 할당된 **Sort Area** 에서 수행돼.

- **PGA 는 세션별 전용 메모리** — 다른 세션이 볼 수 없어
- Sort Area 는 세션이 정렬이 필요할 때 **일정 크기만큼 할당**받아서 사용
- 정렬이 끝나면 Sort Area 는 반환

### Sort Area 관련 파라미터

**오라클 (자동 PGA 관리)**

```
PGA_AGGREGATE_TARGET     -- 인스턴스 전체 PGA 총량 목표치
WORKAREA_SIZE_POLICY     -- AUTO(권장) / MANUAL
```

- `AUTO` 모드면 오라클이 세션마다 필요한 만큼 Sort Area 를 자동 배분 (보통 `PGA_AGGREGATE_TARGET` 의 5%)
- `MANUAL` 모드면 구시대 파라미터인 `SORT_AREA_SIZE` 를 직접 세팅 (요즘은 거의 안 씀)

**MySQL**

```
sort_buffer_size         -- 세션당 정렬용 메모리 크기 (기본 256KB)
tmp_table_size           -- 임시 테이블 메모리 크기
max_heap_table_size      -- 메모리 기반 임시 테이블 최대 크기
```

- MySQL 은 세션이 정렬 시작할 때 `sort_buffer_size` 만큼을 **즉시 통째로 할당**해. 너무 크게 잡으면 **동시 세션 × sort_buffer_size** 만큼 메모리를 먹어서 OOM 위험이 있어.

## 5.1.3 메모리 소트 vs 디스크 소트

### 메모리 소트 (In-Memory Sort)

정렬할 데이터가 **Sort Area 안에 전부 들어가는 경우**.

- 내부 알고리즘은 주로 **Quicksort / Tim Sort** 계열
- CPU 만 쓰고 디스크 I/O 는 발생 안 함 → 매우 빠름

### 디스크 소트 (External Sort)

정렬할 데이터가 **Sort Area 크기를 초과**하는 경우. 한 번에 메모리에 못 올리니까 이렇게 처리해:

1. 입력 데이터를 **Sort Area 크기만큼 조각(chunk)** 로 나눠 메모리에서 정렬
2. 정렬된 조각을 **Temp 테이블스페이스** (디스크) 에 임시 파일로 기록
3. 입력을 다 처리할 때까지 1~2 반복
4. 디스크에 쌓인 정렬된 조각들을 **Merge Pass** 로 합치며 최종 정렬 결과 생성

이게 **외부 정렬(External Merge Sort)** 알고리즘이야. 디스크에 읽고 쓰는 횟수만큼 성능이 떨어져.

### 전환 시점의 성능 차이

| 구분 | 메모리 소트 | 디스크 소트 |
|---|---|---|
| 공간 | PGA Sort Area | PGA + Temp 테이블스페이스 |
| I/O | 없음 | 디스크 쓰기 + 재읽기 |
| 속도 | 빠름 | **수십~수백 배 느림** |
| 트리거 | 기본 | Sort Area 초과 |

### 실행 계획에서 확인하는 법

**오라클**

```
Operation           Name        Sorts(Memory)  Sorts(Disk)
SORT ORDER BY                   1              1
```

- `Sorts(Disk) > 0` 이면 **디스크 소트가 발생**한 거야 → 튜닝 대상

**MySQL**

```
EXPLAIN 결과의 Extra 컬럼:
  Using filesort           → 정렬 발생
  Using filesort; Using temporary  → Temp 테이블 사용 (디스크 소트 가능성)
```

- `filesort` 라는 이름이지만 **메모리 안에서 끝나면 파일 생성 안 함**. 이름에 속지 마.
- **Performance Schema 의 `sort_merge_passes` 값이 크면** 디스크 스왑 발생 신호
