# 2. DBMS의 내부 구성 요소

DBMS는 다음 컴포넌트들이 협력해서 SQL을 처리해. 잘 보면 첫 번째(입력)와 마지막(저장소)을 제외하면 **실제로 "일"을 하는 건 가운데 3개**라고 볼 수 있어.

| 구성 요소 | 역할 |
|-----------|------|
| **인터페이스** | 사용자의 SQL 명령을 받음 |
| **파서 (Parser)** | 명령어가 문법에 맞는지 검사  SQL을 분석하고 실행 계획을 찾음|
| **옵티마이저 (Optimizer)** | 데이터를 가장 빠르게 찾을 전략을 수립 |
| **실행 엔진 (Execution Engine)** | 옵티마이저의 전략대로 실제 데이터를 조회 |
| **스토리지 엔진 (Storage Engine)** | 디스크/SSD에 실제 데이터를 쓰고 읽음 |


파서 옵티마이저 실행엔진 3개가 중요하다는소리지
인터페이스 부터 스토리지엔진까지의 흐름으로 db는 조회된다

여기까지는 이제 좀 쉽잖아 이해가 가지않나?

그럼 인터페이스는 그냥 sql적는거고 이제 3대장중 하나인 파싱을 봐보자

## 3. SQL 파싱 (Parsing) 

> **파싱 순서**: 구문 체크 → 해싱(Hashing) → 소프트 파싱 vs 하드 파싱 이렇게 볼수있어


### ① 구문 체크(Syntax) & 의미 체크(Semantic)

가장 기본적인 3가지를 검사해. 개발을 좀 해봤다면 당연하게 느껴질 항목들이야.

- **구문 체크**: 명령어 순서가 맞는가?
- **의미 체크**: 테이블/컬럼 이름이 실제로 존재하는가?
- **권한 체크**: 이 사용자가 이 데이터를 볼 권한이 있는가?

### ② 해싱(Hashing) & 라이브러리 캐시 확인 🔑

여기가 바로 **"생각보다 많은 일을 하는"** 핵심 구간이야.

- **해싱**: DBMS는 입력받은 SQL 문장 전체를 하나의 고유한 **해시 값(Hash Value)** 으로 변환해.
  - 예: `SELECT * FROM emp` → `7a8b9c...`
- **라이브러리 캐시 탐색**: 메모리 공간인 '라이브러리 캐시'를 뒤져서, 방금 만든 해시 값과 동일한 실행 계획이 이미 저장되어 있는지 확인해. 있으면? **이미 만들어둔 걸 그대로 재사용**하면 되니까 엄청난 이득이지.

### ③ 결정의 순간: 소프트 파싱 vs 하드 파싱

캐시에 있으면 **소프트 파싱**(재사용), 없으면 **하드 파싱**(처음부터 다 만들기). 이름 그대로야.

| 구분 | 상황 | 처리 내용 | 성능 |
|------|------|-----------|------|
| **소프트 파싱 (Soft Parse)** | 캐시에 동일 SQL이 있을 때 | 이미 만들어진 실행 계획을 바로 가져다 씀 | 매우 빠름 |
| **하드 파싱 (Hard Parse)** | 캐시에 동일 SQL이 없을 때 | 최적화와 로우 소스 생성 단계를 처음부터 끝까지 수행 | 매우 느림 (CPU 부하 ↑) |

## 4. 왜 이렇게 복잡하게 할까?

> 옵티마이저가 실행 계획을 세우는 과정(최적화)은 **CPU를 엄청나게 잡아먹는 무거운 작업**이기 때문이야.

### 실제로 일어나는 일

- **하드 파싱**에서 옵티마이저는 테이블·인덱스 통계, 컬럼 분포(히스토그램), 조인 가능한 모든 경우의 수를 탐색하며 **가장 비용이 낮은 실행 계획을 계산**한다. 테이블 5개 조인만 돼도 조인 순서 120가지 × 조인 방식 × 인덱스 사용 여부의 조합이 수천~수만 가지로 불어나므로 **CPU 소비가 매우 크다**.
- **소프트 파싱**은 이 계산을 건너뛰고 **라이브러리 캐시에 저장된 실행 계획을 그대로 꺼내 쓰는 것**뿐이다. 해시 비교와 권한 확인만 거치면 되므로 CPU 비용이 하드 파싱 대비 수십~수백 분의 1 수준이다.

## 5. 바인드 변수(Bind Variable)의 중요성

개발자들은 하드 파싱을 줄이기 위해 **바인드 변수**를 사용합니다.

### ❌ 나쁜 예 (하드 파싱 유발)

```sql
WHERE id = 1
WHERE id = 2
```

> 문자가 다르니 해시 값도 달라져서, 매번 새로 실행 계획을 계산해야 합니다.

### ✅ 좋은 예 (소프트 파싱 유도)

```sql
WHERE id = :id
```

> 문장이 같으니 해시 값도 같아서, 캐시된 계획을 재사용할 수 있습니다.

## 6. 정리

- 구문/의미 체크, 해싱 과정은 모두 **"어떻게든 무거운 최적화 단계를 건너뛰고(소프트 파싱), 저장된 계획을 재사용할 수 있을까?"** 를 판단하기 위한 필수 절차입니다.
- 아무리 인덱스를 잘 걸어도 **하드 파싱이 너무 많으면 DB 전체가 느려집니다.**
- 그래서 실무에서는 **바인드 변수 사용이 튜닝의 기본**으로 여겨집니다.
