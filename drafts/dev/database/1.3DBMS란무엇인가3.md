> 드디어 마지막 단계, 실제로 **"일(Action)"** 이 벌어지는 **실행 엔진(Execution Engine)** 차례야.

## 1. 실행 엔진의 역할

앞 단계들(파서 → 옵티마이저 → 로우 소스 생성기) 은 전부 **"어떻게 실행할지 결정"** 만 하는 단계였어. 실제로 데이터를 메모리로 끌어오고 연산을 수행하는 주체는 **실행 엔진**이야.

실행 엔진은 로우 소스 생성기가 만든 **Row Source Tree** 를 루트부터 순회하면서, 각 연산자(인덱스 스캔, 테이블 액세스, 조인, 정렬 등)를 차례로 호출하고 하위 연산자의 결과를 상위 연산자에 전달해서 최종 결과를 만들어.

---

## 2. 🏗️ 실행 엔진의 주요 역할 3가지

### ① 스토리지 엔진에 블록 요청

실행 엔진은 **디스크에 직접 접근하지 않아**. 필요한 블록의 주소(테이블/인덱스 식별자 + 블록 번호)를 스토리지 엔진에 전달하고, **"이 블록을 버퍼 캐시에 올려달라"** 고 지시해.

- 해당 블록이 이미 버퍼 캐시에 있음 → 스토리지 엔진은 그 블록을 바로 반환 (**논리적 I/O**)
- 버퍼 캐시에 없음 → 스토리지 엔진이 디스크에서 읽어 버퍼 캐시에 적재한 뒤 반환 (**물리적 I/O**)

---

### ② 블록을 행(Row) 단위로 해석

버퍼 캐시에 있는 블록은 **고정 크기의 바이트 덩어리**야. 실행 엔진은 블록 헤더와 로우 디렉터리를 읽어서 블록 내부에 몇 개의 행이 저장되어 있는지, 각 행의 시작 오프셋이 어디인지 계산한 뒤, 행을 한 건씩 꺼내서 **조건 평가·컬럼 추출 단계**로 넘겨.

---

### ③ 연산 수행 (필터링, 정렬, 조인)

Row Source Tree 의 각 연산자를 실제로 수행해.

| 연산 | 실행 엔진이 하는 일 |
|------|------|
| **Filtering** | 행 단위로 WHERE 조건을 평가해 조건을 만족하는 행만 상위 연산자로 전달 |
| **Sorting** | 입력 집합을 PGA의 Sort Area에 적재한 뒤 지정된 정렬 키 기준으로 정렬. Sort Area 를 초과하면 Temp 테이블스페이스로 스왑 |
| **Joining** | NL / Hash / Sort Merge 중 선택된 방식으로 양쪽 입력의 조인 키를 매칭 |

---

## 3. 실행 엔진이 사용하는 두 메모리 영역

실행 엔진은 두 종류의 메모리 영역을 번갈아 사용해. 두 영역은 **공유 범위**와 **용도**가 완전히 달라.

| 영역 | 공유 범위 | 용도 |
|------|------|------|
| **버퍼 캐시 (Buffer Cache)** | SGA 내에서 **모든 세션이 공유** | 디스크에서 읽어온 데이터 블록을 캐싱 |
| **PGA (Program Global Area)** | **세션별로 독립** | 정렬(Sort Area), 해시 조인용 해시 테이블(Hash Area), 커서 상태 저장 |

조건 평가까지는 **버퍼 캐시의 블록을 직접 참조**하면 돼. 근데 정렬·해시 조인처럼 **입력 집합을 재구성하는 연산**은 PGA 에 별도 공간을 확보해서 처리해.

### 블록 요청과 연산 수행의 일반 흐름

```
필요한 블록이 버퍼 캐시에 있나?
  ├─ 있음   → 해당 블록을 참조해 행 추출 → 연산자 수행
  └─ 없음   → 스토리지 엔진이 디스크에서 읽어 버퍼 캐시에 적재
              → 같은 흐름으로 행 추출 → 연산자 수행

정렬/해시 조인 등 재구성 연산이 필요한 경우
  → 버퍼 캐시에서 추출한 행을 PGA(Sort/Hash Area)로 복사해 처리
```

---

## 4. 🏁 드디어 완성된 SQL의 일생 (Final Review)

1편부터 3편까지 배운 내용을 한 번에 정리해보자.

| 단계 | 담당자 | 한 일 |
|------|--------|-------|
| **1** | 파서 | SQL 구문/의미/권한 검사, 해시 값으로 라이브러리 캐시 조회 (소프트 파싱 시도) |
| **2** | 옵티마이저 | 캐시 미스 시 쿼리 변환 → 비용 예측 → 후보 계획 생성 → 최저 비용 계획 선택 (하드 파싱) |
| **3** | 로우 소스 생성기 | 선택된 실행 계획을 실행 엔진이 호출 가능한 Row Source Tree로 변환 |
| **4** | **실행 엔진** | Row Source Tree를 순회하며 스토리지 엔진에 블록 요청, 행 단위 추출, 필터링·정렬·조인 연산 수행, 결과 반환 |

---

## 5. ⚠️ 여기서 튜닝 포인트!

> **"SQL 성능이 안 나와요"** 라고 할 때, 대부분의 시간은 바로 이 **실행 엔진 단계**에서 잡아먹혀.

### 성능 저하의 주요 원인

- **스토리지 엔진에 과도하게 많은 블록 요청을 보냄** → 논리적/물리적 I/O 과다
- **PGA 의 Sort Area 가 부족해 Temp 테이블스페이스로 스왑 발생** → 정렬 시 디스크 I/O 유발, 응답 시간 급증

### 그래서 다음 주제는?

다음 편의 주제인 **'I/O (입출력)'** 이 중요한 이유가 바로 여기 있어.

> **스토리지 엔진에 요청하는 블록 수를 줄이는 것** = 튜닝의 핵심

## 🎓 마무리

이제 SQL 한 줄이 어떤 컴포넌트들을 거쳐 결과가 되는지의 **전체 경로**가 정리됐어.

```
[파서] → [옵티마이저] → [로우 소스 생성기] → [실행 엔진] → [스토리지 엔진]
 문법·   비용 기반         실행 가능한            연산자 호출      블록 I/O
 해시 조회  계획 선택        연산자 트리 생성        / 연산 수행      / 버퍼 캐시 관리
```

여기까지 이해했다면 **'성능 저하의 주범, 디스크 I/O'** 에 대해 본격적으로 알아볼 준비가 끝난 거야. 🚀
