### 6.4 Lock 과 트랜잭션 동시성 제어

- 자신이 쓰는 데이터베이스의 **Lock 메커니즘과 트랜잭션 동시성 제어**는 필수 학습 주제야!

- 오라클 Lock 종류

  1. **DML Lock**
     - 애플리케이션 개발 측면에서 **가장 중요하게 다뤄야 할 Lock**
     - 다중 트랜잭션이 동시에 액세스하는 **사용자 데이터의 무결성을 보호**해줘
     - DML Lock 엔 **테이블 Lock** 과 **로우 Lock** 이 있어
  2. **DDL Lock**
  3. **래치 (Latch)**
     - SGA 에 공유된 각종 자료구조를 보호하기 위해 사용
  4. **버퍼 Lock**
     - 버퍼 블록에 대한 액세스를 **직렬화**하기 위해 사용
  5. **라이브러리 캐시 Lock / Pin**
     - 라이브러리 캐시에 공유된 **SQL 커서와 PL/SQL 프로그램**을 보호하기 위해 사용

- **DML 로우 Lock**

  - 두 개의 동시 트랜잭션이 **같은 로우를 변경하는 것을 방지**해줘
  - 같은 로우를 UPDATE / DELETE / INSERT 할 때
    - **UPDATE, DELETE**: 진행 중인 (아직 커밋하지 않은) 로우를 **다른 트랜잭션이 수정할 수 없어**
    - **INSERT**: unique 인덱스가 있고 같은 값을 입력하려고 할 때 **블로킹이 발생**해
      1. 선행 트랜잭션이 커밋할 때까지 후행 트랜잭션은 기다려
      2. 선행 트랜잭션이 커밋하면 후행 트랜잭션은 실패, 롤백하면 후행 트랜잭션은 성공
  - DML 로우 Lock 으로 인한 성능 저하를 막으려면 **Lock 을 너무 오래 유지하지 않도록 커밋 시점을 조절**해야 해

- **DML 테이블 Lock (= TM Lock)**

  - 현재 트랜잭션이 갱신 중인 테이블의 구조를 다른 트랜잭션이 변경하지 못하게, **로우 Lock 이전에 테이블 Lock 을 먼저 설정**해

  - 로우 Lock 은 **1가지 모드(배타적 모드)** 만 쓰는 반면, 테이블 Lock 에는 **여러 모드(RS, RX, S, SRX, X)** 가 있어

  - 테이블 Lock 이라는 이름 때문에 "테이블 전체를 잠근다" 고 오해하기 쉽지만, 실제로는 해당 트랜잭션이 **그 테이블에서 어떤 종류의 작업을 수행 중인지 DBMS 에 등록해 두는 메타 정보**에 가까워. 다른 트랜잭션은 자신의 작업 모드와 이 등록된 모드를 비교해서 **호환 가능하면 동시 진행, 충돌하면 대기 또는 포기**를 결정해.

    - Lock 이 해제될 때까지 기다릴지 / 일정 시간만 기다리다 포기할지 / 아예 기다리지 않고 바로 포기할지 선택 가능

  - 예) DDL 로 테이블 구조를 변경하려 할 때

    - 테이블 구조를 변경하려는 트랜잭션은 해당 테이블에 **TM Lock 이 걸려 있는지 먼저 확인**해
    - **RX 모드**로 설정한 트랜잭션이 하나라도 있다면 작업을 멈춰
    - 반대로 DDL 이 먼저 수행 중이라면 **DML 을 수행하려는 트랜잭션이 기다려**

- **블로킹 vs 교착상태**

  - **블로킹**: 선행 트랜잭션이 설정한 Lock 때문에 후행 트랜잭션이 **기다리고 있는 상태**
  - **교착상태 (Deadlock)**: 두 트랜잭션이 각각 특정 리소스에 Lock 을 설정한 상태에서, **서로 상대방이 가진 리소스에 Lock 을 걸려고** 해서 아무도 진행하지 못하는 상황
    - 오라클은 교착상태를 감지하면, **먼저 인지한 트랜잭션이 문장 수준 롤백을 진행한 뒤 에러를 던져줘**

- **트랜잭션 동시성 제어**는 **비관적 동시성 제어**와 **낙관적 동시성 제어**로 나뉘어

  1. **비관적 동시성 제어**
     - "사용자들이 같은 데이터를 동시에 수정할 것"이라고 **가정**하고, 데이터를 읽는 시점에 Lock 을 걸어서 조회/갱신이 끝날 때까지 유지해
     - `select ... for update` 로 비관적 Lock 을 걸 수 있어
     - 동시성을 크게 떨어뜨릴 우려가 있지만, `for update wait ...`, `for update nowait` 옵션으로 **동시성을 일부 회복**할 수 있어
  2. **낙관적 동시성 제어**
     - "사용자들이 같은 데이터를 동시에 수정하지 않을 것"이라고 **가정**하고, 읽을 때는 Lock 을 걸지 않아. 대신 **수정 시점에 변경 여부를 반드시 검사**해야 해.
     - 방법은 두 가지: **모든 컬럼의 변경 여부를 확인**하거나, **수정 시간 컬럼**을 따로 두고 수정 시간이 바뀌었는지 확인

- **데이터 품질과 동시성을 함께 잡으려면**

  - `for update` 사용을 **두려워하지 마**
  - `for update` 가 필요한 상황이면 정확히 쓰고, 번거롭더라도 **wait / nowait 옵션으로 예외처리에 신경써야 해**
  - 불필요하게 Lock 을 오래 유지하지 말고, 트랜잭션의 원자성을 보장하는 범위 내에서 **가급적 빨리 커밋**하자

- **채번 방식**

  - **채번**: 신규 데이터를 입력할 때 **PK 중복을 방지하기 위한 과정**

  - 채번 방식 종류

    1. **채번 테이블**

       - 채번 전용 테이블을 별도로 관리하는 방식
       - 채번 레코드를 읽어서 **+1 한 값으로 업데이트**하고, 그 값을 새 레코드의 PK 로 사용해
       - **두 트랜잭션이 중복 값을 채번할 가능성을 원천 차단**해줘
       - **장점**
         - **범용성**이 좋아
         - 중복 레코드 발생에 대비하지 않아도 돼
         - INSERT 과정에서 **결번 방지** 가능
         - **PK 가 복합컬럼일 때도 사용 가능**
       - **단점**
         - 채번 레코드를 변경하기 위한 **로우 Lock 경합** 때문에 성능이 떨어져 (그래서 동시 INSERT 가 아주 많으면 쓰기 어려워)
    2. **시퀀스 오브젝트**

       - 쉽게 말하면 **오라클이 내부적으로 관리해주는 채번 테이블**이야

       - **장점**
         - **성능이 빠름**
         - 중복 레코드 발생에 대비하지 않아도 됨

       - **단점**
         - 테이블마다 시퀀스 오브젝트를 **생성·관리하는 부담**이 있음
         - 시퀀스 채번 과정에서 Lock 에 의한 성능 이슈가 있음 (단, **캐시 사이즈를 적절히 설정**하면 빠른 성능을 유지할 수 있어)
         - **PK 가 단일컬럼일 때만** 사용 가능
         - **결번이 생길 수 있음**
    3. **MAX + 1 조회**
       - 예: `select max(거래일련번호) + 1 from 거래` 의 값을 새로 insert 할 row 의 PK 로 사용
       - **장점**
         - 시퀀스나 채번 테이블을 **관리할 필요 없음**
         - 동시 트랜잭션 충돌이 적으면 **성능이 매우 좋음**
         - **PK 가 복합컬럼이어도 사용 가능**
       - **단점**
         - 레코드 중복에 대비한 **세밀한 예외처리 필요**
         - 다중 트랜잭션의 동시 채번이 심하면 **시퀀스보다 훨씬 느려질 수도** 있음
