# 4.1 NL 조인
조인 메소드에는 **NL 조인, 해시 조인, 소트 머지 조인** 이렇게 3가지가 있어.

MySQL 은 버전에 따라 지원하는 조인 메소드가 달라.

- **5.7 까지** → NL 조인만 지원
- **8.0 부터** → 해시 조인도 지원

## 4.1.1 NL 조인 메커니즘
NL 조인 (Nested Loop Join) 은 이름 그대로 **중첩 루프** 로 두 테이블을 매칭하는 방식이야.

조인 과정은 이렇게 흘러가.

- (1) Table A 에서 row 를 하나씩 스캔해
    - 이때 Table A 를 **Driving Table** (또는 Outer Table) 이라고 불러


- (2) Outer Table 의 row 를 하나 꺼낼 때마다 Table B 를 스캔해서 조인 조건에 맞는 row 를 반환해
    - 이때 Table B 를 **Inner Table** 이라고 불러


- (3) Outer Table 의 row 가 끝날 때까지 1~2 과정을 반복해

## 4.1.3 올바른 조인 메소드 선택
조인을 고를 때는 이 순서로 접근하면 돼.

1. 일단 **NL 조인부터 고려**해. 소량 조인에선 거의 최적이거든.
2. 성능이 안 나오면 NL 조인 튜닝 포인트를 따라 **과도한 랜덤 액세스가 발생하는 지점**을 먼저 찾아.
3. 그래도 안 되면 **소트 머지 조인이나 해시 조인**을 검토해.

## 4.1.4 NL 조인 특징 요약
1. **랜덤 액세스 위주**의 조인 방식이야. 그래서 대량 데이터를 조인할 때는 효율이 떨어져.
2. 한 레코드씩 순차적으로 처리하기 때문에, **소량 조인**이거나 **부분 범위 처리**가 가능한 상황에선 아주 빠른 응답 속도를 낼 수 있어.














# 공용 메모리 vs 전용 메모리

두 메모리의 핵심 차이는 **"누가 쓰느냐"** 와 **"무엇을 담느냐"** 입니다.

## 1. 공용 메모리 (Shared Memory)

**모든 세션이 함께 사용하는 메모리**입니다. DB 인스턴스가 시작될 때 한 번 할당되고, 접속한 모든 사용자가 이 공간을 공유합니다.

여기에는 주로 **디스크에서 읽어온 데이터 페이지(블록)** 가 캐싱됩니다. 예를 들어 A 세션이 `회원` 테이블을 조회해서 그 데이터가 공용 메모리(Buffer Pool)에 올라왔다면, B 세션이 같은 테이블을 조회할 때 디스크를 다시 읽지 않고 메모리에서 바로 꺼내 쓸 수 있습니다. 이게 공용 메모리의 핵심 효용입니다.

이외에도 실행계획 캐시, 메타데이터(딕셔너리) 캐시, 트랜잭션 로그 버퍼 등 **"여러 세션이 공통으로 참조하면 이득인 것들"** 이 여기에 들어갑니다.

## 2. 전용 메모리 (Private Memory)

**각 세션(또는 쿼리)이 혼자 쓰는 메모리**입니다. 세션이 접속하면 할당되고, 세션이 끊어지면 반환됩니다.

여기에는 **그 세션만의 작업 데이터**가 담깁니다. 대표적으로 정렬(Sort) 중인 데이터, 해시 조인을 위한 해시 테이블, 커서 정보, 세션 변수, 임시 결과 집합 같은 것들이죠. 다른 세션이 볼 필요도, 봐서도 안 되는 데이터들입니다.

## 쉬운 비유

도서관을 떠올리면 이해가 빠릅니다.

**공용 메모리 = 도서관의 책장**입니다. 모든 이용자가 공유하는 공간이고, 누군가 꺼내 본 책은 다음 사람도 볼 수 있습니다. 책(데이터)을 보관하는 곳이죠.

**전용 메모리 = 각 이용자의 개인 열람석**입니다. 내가 책장에서 책을 가져와 내 자리에서 노트 필기하고, 메모하고, 정리하는 공간입니다. 옆자리 사람과 섞이지 않고, 내가 자리를 비우면 치워집니다.

## 구체적인 예시: SELECT 쿼리 한 번의 흐름

`SELECT * FROM orders ORDER BY amount DESC` 같은 쿼리가 실행될 때를 보겠습니다.

먼저 `orders` 테이블의 데이터 블록들을 디스크에서 읽어 **공용 메모리(Buffer Pool)** 에 올립니다. 이 데이터는 다른 세션이 같은 테이블을 조회할 때도 재사용할 수 있습니다.

그다음 이 데이터를 `amount` 기준으로 정렬해야 하는데, 이 **정렬 작업은 내 세션의 전용 메모리(Sort Area / work_mem)** 에서 이루어집니다. 다른 세션은 내가 뭘 정렬하는지 알 필요도 없고, 알면 안 되죠. 정렬이 끝나 결과를 반환하면 이 전용 메모리는 비워집니다.

## 왜 나눠 놓았을까?

**성능과 안전성** 때문입니다.

공용 메모리에 모든 걸 다 넣으면 세션마다 중구난방으로 데이터를 쓰고 지워서 **동기화 비용(락 경합)** 이 폭증하고, 한 세션의 오류가 다른 세션에 영향을 줄 수 있습니다. 반대로 모든 걸 전용 메모리로만 처리하면 같은 데이터를 세션마다 중복으로 읽어야 해서 **디스크 I/O가 폭증**합니다.

그래서 **"공유해서 이득인 것은 공용에, 공유하면 오히려 손해거나 위험한 것은 전용에"** 라는 원칙으로 나눠 놓은 것입니다.

## 정리

간단히 표로 정리하면 이렇습니다.

| 항목 | 공용 메모리 | 전용 메모리 |
|------|-------------|-------------|
| 사용 주체 | 모든 세션이 공유 | 각 세션이 독점 |
| 담기는 것 | 데이터 블록, 실행계획, 메타데이터 | 정렬·해시 작업 데이터, 커서, 세션 변수 |
| 생명주기 | 인스턴스 시작~종료 | 세션 시작~종료 |
| 핵심 효용 | 디스크 I/O 절감 (캐싱) | 세션별 연산 격리 |

앞서 나온 Sort Merge Join 이야기로 돌아가면, **"조인 대상 데이터는 공용 메모리에서 읽어오고, 그걸 정렬하는 작업은 전용 메모리(PGA)에서 한다"** 는 흐름이 됩니다. 그래서 정렬이 메모리 안에서 끝나면 빠르고, 데이터가 너무 커서 디스크로 넘치면 느려지는 것이죠.
