## 1. 인덱스가 필요한 이유

테이블은 기본적으로 **입력된 순서대로 저장**되어 있을 뿐, 어떤 컬럼 값이 어디 있는지에 대한 순서 정보를 갖지 않는다. 특정 값을 찾으려면 **모든 블록을 순차적으로 읽고 각 행의 해당 컬럼을 비교**해야 한다. 이것이 **Table Full Scan** 이다.


블럭은 대충 8kb정도의 단위로 dbms에서 조회하는것을 말한다 이블록은 왜있냐
블록 단위 관리는 "디스크 물리 특성 + 지역성 + 캐시 효율 + 동시성 + 복구"가 얽힌 종합적인 선택입니다.
대충 넣고 뺄때 이정도 는 가져와야 그주변도 좀 서치하고 캐싱도 하고 단일 1개 행 가져오는거랑 데이터를 가져올때 접근비용이 비싼만큼 어느정도 묶어서 가져온다


인덱스는 **특정 컬럼의 값을 미리 정렬해서, 각 값이 테이블의 어느 위치(ROWID)에 있는지를 기록해 둔 별도의 자료구조**다. 조건에 해당하는 값이 인덱스의 어디에 있는지 빠르게 찾고, 거기서 ROWID를 꺼내 테이블의 필요한 블록만 읽으면 되므로 전체 블록을 읽지 않아도 된다.

ROWID란?ROWID는 **"이 행이 디스크 어디에 저장되어 있는가"**를 나타내는 값입니다. Oracle에서는 모든 행이 자동으로 ROWID를 가지며, 이 값을 통해 DB는 행의 실제 위치를 즉시 찾아갈 수 있습니다.Oracle의 ROWID는 다음 정보를 담고 있습니다:

지금은 블록까지는 좀더 큰 데이터 단계라고 볼수있다 
AAASPmAAFAAAABSAAA
│     │    │      │
│     │    │      └─ Row Number (블록 내 몇 번째 행인지)
│     │    └──────── Block Number (파일 내 몇 번째 블록인지)
│     └───────────── File Number (어느 데이터 파일인지)
└─────────────────── Object Number (어느 테이블/객체인지)

## 2. B-Tree 인덱스의 3층 구조

그냥 다 비트리 하면된다고 보면된다 

DB 인덱스는 대부분 **B-Tree**(Balanced Tree) 구조로 되어 있어. 이름 그대로 **균형 잡힌** 트리.

```
                 [ 루트 블록 ]               ← 1층
                /      |      \
         [브랜치] [브랜치] [브랜치]         ← 2층 (여러 단계일 수도)
          /  |  \    |    /  |  \
       [리프][리프][리프][리프][리프]        ← 3층 (최하단)
        ↓    ↓    ↓    ↓    ↓
     (정렬된 키 + ROWID)
```

| 노드 | 역할 |
|---|---|
| **루트 (Root)** | 트리의 시작점. 1개만 존재 |
| **브랜치 (Branch)** | 중간 노드. 더 작은 범위로 내려가는 이정표 |
| **리프 (Leaf)** | 실제 **키 값 + ROWID**(테이블 위치) 저장. 옆 리프와 연결(Linked List) |

### 핵심 포인트

- **리프 블록은 키 값 순으로 정렬**되어 있음 → 범위 검색이 효율적
- **리프 블록끼리 양방향 연결**되어 있음 → 이웃 값으로 이동이 쉬움
- **모든 리프의 높이가 같음**(Balanced) → 어떤 값이든 같은 비용으로 접근 가능

## 3. 인덱스 탐색 = "Vertical Traversal + Horizontal Traversal"

인덱스를 탄다는 건 사실 **두 단계의 탐색**을 한다는 뜻이야.

### ① Vertical Traversal

> 루트 → 브랜치 → 리프까지 **내려가는** 과정.
> 찾고자 하는 값의 **시작 위치(리프 블록)** 를 찾는 단계.

- 트리 깊이(BLEVEL)만큼 블록을 읽음
- 100만 건의 인덱스라도 보통 **3~4단계**면 리프 도달 (로그 시간 복잡도)

### ② Horizontal Traversal

> 리프 블록에서 **옆으로 쭉 훑으며** 데이터를 읽는 과정.
> 진짜 데이터 조회의 본체 — **스캔 범위의 크기가 성능을 결정**한다.

```
루트 → 브랜치 → 리프 도착!  ← 여기까지가 Vertical Traversal
                  │
                  ↓ 옆으로 ─→ ─→ ─→ ─→ ─→  ← Horizontal Traversal
                  (필요한 만큼)
```

### 🔑 어디서 시간이 더 걸리나?

| 탐색 | 비용 | 메모 |
|---|---|---|
| Vertical Traversal | 거의 상수 (3~4블록) | 트리 깊이가 낮음 |
| **Horizontal Traversal** | **스캔 범위에 비례** | **튜닝의 핵심** |

> 그래서 튜닝은 결국 **"Horizontal Traversal 범위를 줄이는 일"** 이다.
> 선두 컬럼 조건, 결합 인덱스 순서, 액세스 vs 필터 조건 — 전부 **Horizontal Traversal을 좁히기 위한 기법**이야.

## 4. 인덱스로 테이블 데이터 찾기 — ROWID

리프에는 **키 값 + ROWID**만 있지, 실제 레코드는 없어.

```
[리프 블록]
 (홍길동, ROWID=0x12AB3...)
 (김철수, ROWID=0x12AB8...)
 (이영희, ROWID=0x12AC1...)
       ↓ ROWID로 테이블 블록을 찾아가야 함 (랜덤 액세스)
[테이블 블록]
 실제 전체 행 데이터
```

- ROWID = **테이블 내 물리적 위치 주소**
- 이 ROWID로 테이블 블록을 찾아가는 게 **"테이블 랜덤 액세스"** 🎯
- 읽을 데이터가 많아지면 이 랜덤 액세스 비용이 커져 → 손익분기점을 넘으면 **Full Scan이 더 빠름**

## 5. 결합 인덱스(Composite Index)의 탐색

여러 컬럼으로 구성된 결합 인덱스도 똑같은 B-Tree 구조야. 다만 **정렬 기준**이 복합키라는 차이.

```
인덱스 (부서, 이름)

[리프 블록들 — 부서 → 이름 순 정렬]
 (영업, 김철수)
 (영업, 박영희)
 (영업, 이민수)
 (인사, 강지훈)    ← 부서가 바뀌면 이름은 다시 처음부터
 (인사, 김민정)
 (인사, 최정민)
```

### 왜 "선두 컬럼"이 중요한가?

- **부서 = '영업'** → 시작점 찾기 OK ✅
- **이름 = '김민정'** 만 주면? → 어느 부서의 김민정인지 모름. 전체 리프를 다 뒤져야 함 ❌

이게 바로 **"선두 컬럼이 조건절에 없으면 인덱스를 못 탄다"** 의 진짜 이유.

---

## 6. 📝 정리 — 왜 B-Tree인가?

1. **균형 트리라 모든 값의 탐색 비용이 거의 일정**
2. **리프가 정렬되어 있고 연결되어 있어 범위 검색이 효율적**
3. **Vertical Traversal은 O(log N), Horizontal Traversal은 범위에 비례** → 튜닝 지점이 명확
4. **ROWID로 테이블 액세스** → 이 부분의 랜덤 I/O가 튜닝의 핵심 주제
