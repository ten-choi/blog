> 7.1에서 옵티마이저가 **"비용 기반으로 판단"** 하는 걸 봤다. 그런데 그 판단이 틀릴 때가 있다.
> 그럴 때 개발자가 옵티마이저에게 **"이렇게 해줘"** 라고 찍어주는 수단이 **힌트(Hint)** 다.

---

## 7.2 옵티마이저에 대한 이해와 힌트

### 7.2.1 힌트를 쓰는 이유

옵티마이저는 대부분 **최적의 실행계획**을 찾아주지만, 이런 경우엔 틀린다:

- **통계정보가 낡았거나 부정확**할 때
- **데이터 분포가 옵티마이저 가정과 다를 때** (히스토그램이 없거나 부족)
- **바인드 변수 때문에 카디널리티 예측이 빗나갈 때** (Bind Peeking 이슈)
- **복잡한 조인에서 경우의 수가 너무 많아** 옵티마이저가 포기할 때

이럴 때 힌트로 **실행계획을 고정**해준다.

### 7.2.2 힌트 문법

```sql
SELECT /*+ 힌트_이름(옵션) */ 컬럼 FROM 테이블 ...
```

- `/*+ ... */` 형태로 **SELECT/INSERT/UPDATE/DELETE 바로 뒤**에 씀
- 오타나 잘못된 힌트는 **에러가 나지 않고 그냥 무시**됨 (조용히 실패) → **검증이 중요**

### 7.2.3 자주 쓰는 힌트 (오라클 기준)

**① 액세스 경로 제어**

| 힌트 | 의미 |
|---|---|
| `FULL(t)` | 테이블 t를 Full Scan하도록 강제 |
| `INDEX(t idx)` | 테이블 t에 인덱스 idx를 쓰도록 |
| `INDEX_DESC(t idx)` | 인덱스를 역순 스캔 |
| `NO_INDEX(t idx)` | 특정 인덱스를 쓰지 말도록 |

**② 조인 방식 제어**

| 힌트 | 의미 |
|---|---|
| `USE_NL(t)` | t를 후행으로 NL 조인 |
| `USE_HASH(t)` | t를 후행으로 해시 조인 |
| `USE_MERGE(t)` | 소트 머지 조인 |

**③ 조인 순서 제어**

| 힌트 | 의미 |
|---|---|
| `LEADING(t1 t2 t3)` | 조인 순서를 t1 → t2 → t3으로 고정 |
| `ORDERED` | FROM 절 순서대로 조인 (구식) |

**④ 쿼리 변환 제어**

| 힌트 | 의미 |
|---|---|
| `UNNEST` / `NO_UNNEST` | 서브쿼리 Unnesting 허용/차단 |
| `MERGE` / `NO_MERGE` | 뷰 병합 허용/차단 |
| `PUSH_PRED` | 조건을 뷰 내부로 밀어 넣기 |

**⑤ 기타**

| 힌트 | 의미 |
|---|---|
| `APPEND` | Direct Path INSERT |
| `PARALLEL(t, 4)` | 병렬 처리 (DOP = 4) |
| `GATHER_PLAN_STATISTICS` | 실행 시점의 실제 통계 수집 |

### 7.2.4 MySQL의 힌트

MySQL은 오라클보다 힌트가 단순하지만 핵심은 비슷해.

```sql
-- 인덱스 강제
SELECT * FROM emp USE INDEX (idx_dept) WHERE dept_id = 10;
SELECT * FROM emp FORCE INDEX (idx_dept) WHERE dept_id = 10;
SELECT * FROM emp IGNORE INDEX (idx_dept) WHERE dept_id = 10;

-- 옵티마이저 힌트 (MySQL 5.7+)
SELECT /*+ INDEX(emp idx_dept) */ * FROM emp WHERE dept_id = 10;
SELECT /*+ NO_INDEX_MERGE(emp) */ * FROM emp WHERE ...;

-- 조인 순서
SELECT /*+ JOIN_ORDER(a, b) */ * FROM a, b WHERE ...;
```

### 7.2.5 힌트를 쓰면 안 되는 경우

> 💬 힌트는 **"마지막 수단"** 이야. 먼저 이 순서로 접근하자.

1. **통계 먼저 갱신** — `DBMS_STATS.GATHER_TABLE_STATS` / MySQL의 `ANALYZE TABLE`
2. **쿼리 재작성** — 서브쿼리/조인을 옵티마이저가 잘 풀도록 다듬기
3. **인덱스 재설계** — 진짜 필요한 인덱스가 없을 수도 있음
4. **그래도 안 되면 힌트**

### 7.2.6 힌트의 위험성

- **데이터 분포가 바뀌면 잘못된 선택이 됨** — 힌트는 고정이지만 데이터는 움직임
- **옵티마이저의 미래 개선을 차단** — DB 버전 업에도 옛날 플랜 유지
- **"유령 힌트" 문제** — 오타로 무시돼도 에러가 안 남 → 실행계획으로 반드시 검증

### 7.2.7 힌트 적용 후 확인하는 법

```sql
-- 오라클
EXPLAIN PLAN FOR SELECT /*+ ... */ ...;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- MySQL
EXPLAIN SELECT /*+ ... */ ...;
EXPLAIN FORMAT=JSON SELECT /*+ ... */ ...;
```

> 힌트를 넣었으면 **반드시 실행계획으로 반영 여부 확인**. 조용히 무시됐을 수 있다.

## 📝 정리

- 힌트 = 옵티마이저에게 **"내 판단을 따라달라"** 고 찍어주는 주석
- 통계/쿼리/인덱스를 먼저 고치고, **그래도 안 되면** 힌트를 쓴다
- 오타는 에러 없이 무시됨 → **실행계획으로 반드시 검증**
- 힌트는 고정, 데이터는 변함 → 장기적으론 부채가 될 수 있음
