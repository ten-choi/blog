## 오디블로 영어 공부를 하려고 했는데

영어를 공부하는 사람이라면 한 번쯤 생각해봤을 것 같다. 오디블을 들으면서 영어 공부를 해야겠다고 마음먹고 구독하는 것을.

몇 번 들어본 결과, 생각보다 비효율적이었다. 듣기만으로는 머리에 잘 남지 않는다.

문제는 이런 패턴이었다.

1. 듣는다
2. 모르는 단어가 나온다
3. 찾아보고 다시 듣던 곳으로 돌아간다
4. 그 구간을 반복해서 듣고 싶다

그래서 앱과 웹 모두 찾아봤는데, 구간반복을 지원하는 기능이 없었다. 앱에서는 구현이 어려워 보였지만 웹이라면 가능할 것 같았다. 크롬 익스텐션으로 직접 만들어 보기로 했다.

## 원하는 기능

1. 구간반복
2. 구간반복 저장
3. 저장된 구간에서 몇 초 이동하는 버튼
4. 진행률을 보여주는 상단 바

## 개발 과정

Claude와 함께 핵심 기능인 구간반복부터 개발했다. 기능 자체는 금방 만들었다.

### 핵심 문제: 구간반복이 중간에 멈추는 현상

가장 기록에 남길 만한 문제는 **구간반복이 두 번째 반복쯤부터 멈춰버리는 것**이었다.

단순하게 생각하면 `audio.currentTime = 시작지점` 으로 seek 하면 거기서부터 다시 재생될 것 같지만, 실제로는 그렇지 않았다.

#### Audible의 스트리밍 구조


DRM (Digital Rights Management)
DASH (Dynamic Adaptive Streaming over HTTP)


HTMLMedia (HTMLMediaElement)
우리가 웹사이트에서 보는 <video>나 <audio> 태그 그 자체를 말합니다.

2. MSE (Media Source Extensions)
이게 핵심입니다. 원래 <video> 태그는 mp4 같은 완성된 파일 하나만 재생할 수 있었는데요, MSE가 등장하면서 데이터 조각을 실시간으로 이어 붙이는 게 가능해졌습니다.



Audible은 일반 mp3 파일 하나를 통째로 재생하는 방식이 아니다.

- **DASH 스트리밍** 방식
- **20초 단위 세그먼트**로 분할
- **DRM(CENC)** 적용

이 구조에서 seek가 발생하면 플레이어 내부는 이렇게 동작해야 한다.

1. `seeking` 이벤트 발생
2. 새 위치에 해당하는 세그먼트를 네트워크로 fetch
3. 준비되면 `seeked` 이벤트 발생
4. 재생 계속

그런데 반복 재생을 여러 번 하다 보면, seek 후에 **플레이어 내부 fetch가 멈춰버리는** 경우가 생겼다. 이 상태에서는 `readyState`가 1에 머물고, `seeked` 이벤트도 오지 않아 재생이 재개되지 않는다.

#### `readyState` 값의 의미

| 값 | 상태 |
|---|---|
| 0 | 아무 정보 없음 |
| 1 | 메타데이터만 있음 |
| 2 | 현재 프레임 재생 가능 |
| 3, 4 | 안정적으로 재생 가능 |

`readyState`가 1에 계속 머문다는 것은, seek는 걸렸는데 실제 재생에 필요한 세그먼트를 받아오지 못한 채 멈춰 있다는 의미다.

#### 해결 방법

`seeked` 이벤트를 기다리면서 동시에 `readyState`를 감시한다. `readyState <= 1` 상태가 일정 시간 이상 지속되면 `currentTime`을 다시 설정해 seek를 재트리거한다.

> **한 줄 요약:** Audible의 DASH 플레이어는 반복 seek를 여러 번 하다 보면 내부 세그먼트 로딩이 멈출 수 있다. `seeked`만 믿지 말고 `readyState`도 감시하면서, 막히면 `currentTime`을 다시 설정해 seek를 재시도해야 한다.
>
> 


targetAudio.currentTime = loopStart + (retryCount * 0.001);

사실상:

“같은 시간으로 seek하면 플레이어가 새 seek로 안 볼 수도 있으니,
아주 미세하게 다른 값으로 넣어서 내부에서 새 요청으로 인식하게 만들자”

### 나머지 기능들

구간반복 저장(2번)과 초 이동 버튼(3번)은 비교적 간단하게 개발했다.

진행률 상단 바(4번)는 아쉬운 점이 있었다. 유튜브처럼 항상 화면 상단에 진행률 바를 표시하고 싶었는데, 크롬 익스텐션 권한으로는 불가능했다. 브라우저가 아닌 데스크톱 애플리케이션 수준의 권한이 필요한 부분이었다.

대신 별도의 작은 웹창을 따로 띄워서 우회했다. 모니터를 3개 쓰고 있어서 구석에 띄워두면 간간이 확인하기에는 충분했다.

## 배포는 안 하기로

검색해보면 생각보다 이 기능을 원하는 사람이 꽤 있었다. 배포도 고민했지만, Audible 관련 익스텐션을 배포했을 때 법적으로 문제가 될 수 있을 것 같아 개인용으로만 쓰기로 했다.

## 앞으로 해보고 싶은 것

킨들과 오디블을 따로 구매하는 게 번거로워서, 오디블과 같은 책의 텍스트를 준비해두고 오디블 음성을 STT로 변환해 자막처럼 보여주는 것도 시도해봤다. 그런데 STT 결과를 실시간으로 텍스트에 맞춰 따라가는 게 생각보다 매끄럽지 않았다.

제대로 만들려면 이런 기능들이 필요하다.

- STT로 읽는 속도에 맞춰 북 리더기 스크롤 연동
- 모르는 영단어 저장 및 후리가나 형태로 플로팅 표시

기술적으로 STT 결과와 텍스트를 자연스럽게 동기화하는 부분이 가장 어렵다. 좀 더 고민해봐야 할 것 같다.
