> 세상에서 가장 쉬운, Junior 개발자를 위한 SQL Tuning 기초 정리
>
> 드디어 마지막 단계, 실제로 **"일(Action)"** 이 벌어지는 **실행 엔진(Execution Engine)** 입니다.

## 1. 실행 엔진은 "공사 현장의 작업반장"

앞선 단계들이 설계도를 그리고 자재를 준비하는 과정이었다면,
**실행 엔진은 실제로 장비를 돌려 흙을 파고 건물을 올리는 역할**을 합니다.

로우 소스 생성기가 넘겨준 **'조립 설명서(Row Source Tree)'** 를 한 줄 한 줄 읽으면서 작업을 수행하죠.

---

## 2. 🏗️ 실행 엔진의 주요 역할 3가지

### ① 스토리지 엔진에 데이터 요청 (가장 중요!)

실행 엔진은 **스스로 하드디스크에 가서 데이터를 꺼내올 힘이 없습니다.**
대신 스토리지 엔진에게 명령을 내립니다.

> **💬 명령 예시**
> *"야, 가서 `customer` 테이블의 10번 블록 좀 메모리(Buffer Cache)로 올려줘!"*

- 메모리에 이미 데이터가 있다면 → 바로 가져옴
- 없다면 → 디스크에서 읽어오게 시킴

---

### ② 로우(Row) 단위로 데이터 조립

스토리지 엔진이 **블록 단위**로 데이터를 가져다주면, 실행 엔진은 그걸 쪼개서 우리가 아는 **행(Row) 단위**로 만듭니다.

> *"이 블록 안에 데이터가 5줄 있네? 하나씩 꺼내서 조건에 맞는지 확인해 보자."*

---

### ③ 연산 수행 (필터링, 정렬, 조인)

가져온 데이터를 가지고 사용자가 요청한 복잡한 계산을 처리합니다.

| 연산 | 예시 |
|------|------|
| **Filtering** | "나이가 20세 이상인 사람만 남기고 나머지는 버려!" |
| **Sorting** | "이름 순서대로 줄 세워!" |
| **Joining** | "A 테이블의 이 사람이랑 B 테이블의 이 정보랑 합쳐!" |

---

## 3. 💡 실행 엔진이 일하는 두 개의 작업 공간

실행 엔진은 일을 할 때 **두 곳의 장소**를 바쁘게 왔다 갔다 합니다.

| 장소 | 역할 | 비유 |
|------|------|------|
| **버퍼 캐시 (Buffer Cache)** | 데이터가 임시로 올라와 있는 공유 메모리 공간 | 작업대 (책상 위) 📋 |
| **PGA (Private Global Area)** | 정렬(Sort)이나 해시 조인을 할 때 사용하는 개별 작업 공간 | 개인 공구함 🧰 |

### 실행 엔진의 일상 루틴

```
"버퍼 캐시에 데이터 있나?"
  ↓ 없음
"스토리지 엔진아, 디스크에서 가져와서 버퍼 캐시에 올려줘."
  ↓
"자, 이제 내 책상(PGA)으로 가져와서 정렬(Sort) 시작!"
```

---

## 4. 🏁 드디어 완성된 SQL의 일생 (Final Review)

1편부터 3편까지 배운 내용을 한 번에 정리해봅시다.

| 단계 | 담당자 | 한 일 |
|------|--------|-------|
| **1** | 파서 | "이 SQL 문법 맞네? 예전에 해본 적 있나 해시값으로 찾아보자." (소프트 파싱 시도) |
| **2** | 옵티마이저 | "처음 보는 SQL이네. 쿼리 변환하고 비용 계산해서 제일 싼 계획 짜야지." (하드 파싱) |
| **3** | 로우 소스 생성기 | "계획 나왔다! 실행 엔진이 이해하게 트리 구조로 코드 짜줄게." |
| **4** | **실행 엔진** | "좋아, 설명서대로 스토리지 엔진 시켜서 데이터 가져오고, 내가 직접 정렬해서 사용자한테 줄게!" ✅ |

---

## 5. ⚠️ 여기서 튜닝 포인트!

> **"SQL 성능이 안 나와요"** 라고 할 때, 대부분의 시간은 바로 이 **실행 엔진 단계**에서 잡아먹힙니다.

### 성능 저하의 주요 원인

- 🔥 **스토리지 엔진에 일을 너무 많이 시킴** → I/O 과다
- 🔥 **실행 엔진이 메모리(PGA)가 부족해서 쩔쩔맴** → Sort 부하

### 그래서 다음 주제는?

다음 공부 주제인 **'I/O(입출력)'** 가 중요한 이유가 여기 있습니다.

> 👉 **실행 엔진이 스토리지 엔진을 얼마나 적게 부려 먹느냐 = 튜닝의 핵심**

## 🎓 마무리

이제 SQL이 실행되는 **전체 메커니즘**이 머릿속에 한 폭의 그림처럼 그려지시나요?

```
[파서] → [옵티마이저] → [로우 소스 생성기] → [실행 엔진] → [스토리지 엔진]
  문법       전략          조립 설명서          작업반장        창고지기
```

여기까지 이해하셨다면, 이제 **'성능 저하의 주범, 디스크 I/O'** 에 대해 알아볼 준비가 완벽히 되신 겁니다! 🚀
