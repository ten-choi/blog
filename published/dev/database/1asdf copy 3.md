# [요약] SQL 파싱과 최적화 (SQL Tuning 기초)

## 1. SQL 처리 과정 (Three Steps)
사용자가 SQL을 실행하면 DBMS 내부에서는 다음의 세 단계를 거쳐 결과를 반환합니다.



## 1. DBMS 는 무엇니냐

DBMS의 주요 내부 구성 요소로 이렇게 찾는다고 함 
인터페이스에서 sql을 받고 그다음 문법검사하고 옵티마이저에서 어떻게 조회할지 정리하고 실행엔진으로 실행해서 스토리지에 접근해서 데이터를 가져온다


인터페이스: 사용자의 SQL 명령을 받음.

파서(Parser): 명령어가 문법에 맞는지 검사함.

옵티마이저(Optimizer): (여기가 바로 그곳!) 어떻게 데이터를 찾는 게 가장 빠를지 전략을 짬.

실행 엔진(Execution Engine): 옵티마이저의 전략대로 실제 데이터를 찾아옴.

스토리지 엔진(Storage Engine): 하드디스크나 SSD에 실제로 데이터를 쓰고 읽음.




### ① SQL 파싱 (Parsing)


① 구문 체크 (Syntax) & 의미 체크 (Semantic)가장 기본입니다."명령어 순서가 맞는가?""테이블 이름은 존재하는가?""이 사용자가 이 데이터를 볼 권한이 있는가?" 를 확인합니다.② 해싱(Hashing) & 라이브러리 캐시 확인 (핵심!)여기가 바로 사용자가 말씀하신 "생각보다 많은 일을 하는" 지점입니다.해싱: DBMS는 입력받은 SQL 문장 전체를 하나의 **고유한 값(Hash Value)**으로 변환합니다. (예: SELECT * FROM emp → 7a8b9c...)라이브러리 캐시(Library Cache) 탐색: 메모리 공간인 '라이브러리 캐시'를 뒤져서, 방금 만든 해시 값과 똑같은 값을 가진 실행 계획이 이미 저장되어 있는지 확인합니다.③ 결정의 순간: 소프트 파싱 vs 하드 파싱구분상황처리 내용성능소프트 파싱 (Soft Parse)캐시에 동일 SQL이 있을 때이미 만들어진 실행 계획을 바로 가져다 씀.매우 빠름하드 파싱 (Hard Parse)캐시에 동일 SQL이 없을 때**최적화(3번)**와 로우 소스 생성(4번) 단계를 처음부터 끝까지 다 수행함.매우 느림 (CPU 부하)왜 이렇게 복잡하게 할까?옵티마이저가 실행 계획을 세우는 과정(최적화)은 CPU를 엄청나게 잡아먹는 무거운 작업이기 때문입니다.비유하자면 이렇습니다:하드 파싱: 목적지까지 가는 길을 찾기 위해 지도를 펼치고, 실시간 교통량을 분석하고, 최단 거리를 매번 새로 계산하는 것.소프트 파싱: "아, 아까 갔던 길이지?" 하고 내비게이션 최근 목적지 목록에서 바로 눌러서 출발하는 것.그래서 개발자들은 하드 파싱을 줄이기 위해 바인드 변수를 사용합니다.나쁜 예 (하드 파싱 유발): WHERE id = 1, WHERE id = 2 (문자가 다르니 해시 값도 달라서 매번 새로 계산)좋은 예 (소프트 파싱 유도): WHERE id = :id (문장이 같으니 해시 값도 같아서 캐시된 계획 재사용)요약하자면사용자가 말씀하신 구문/의미/해싱 과정은 모두 **"어떻게든 무거운 '최적화' 단계를 건너뛰고(소프트 파싱), 저장된 계획을 재사용할 수 있을까?"**를 판단하기 위한 필수 절차인 셈입니다.공부하고 계신 문서에서 이 부분을 강조하는 이유는, 아무리 인덱스를 잘 걸어도 하드 파싱이 너무 많으면 DB 전체가 느려지기 때문이에요.혹시 바인드 변수가 어떻게 생겼는지, 혹은 라이브러리 캐시를 직접 확인하는 방법도 궁금하신가요?
