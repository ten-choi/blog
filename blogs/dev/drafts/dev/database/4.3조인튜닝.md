> NL 조인은 **"한 건씩"**, 소트 머지는 **"양쪽 정렬"**,
> 해시 조인은 **"한쪽을 해시 테이블로 올려놓고 다른 쪽을 훑기"** — 대량 조인의 현대적 해답이다.

---

## 4.3 해시 조인 (Hash Join)

### 4.3.1 해시 조인 메커니즘

두 단계로 진행돼.

**① Build 단계**

- 둘 중 **작은 집합(Build Input)** 을 골라 조인 키를 해시 함수로 돌려서
- **메모리(PGA의 Hash Area)** 에 해시 테이블을 쌓는다.

**② Probe 단계**

- 큰 집합(Probe Input)을 **한 행씩** 훑으면서
- 각 행의 조인 키를 같은 해시 함수로 돌려 → 해시 테이블에서 매칭 확인

```
Build (작은 쪽)         Probe (큰 쪽)
[dept]                  [emp]
 ↓ 해시 테이블 구축       ↓ 한 행씩 스캔하며 해시 조회
[h(10) → row]           for each row:
[h(20) → row]             bucket = h(row.dept_id)
[h(30) → row]             if bucket in hash table → join
```

### 4.3.2 왜 빠른가

- **해시 테이블 조회는 O(1)** — NL 조인의 인덱스 탐색보다 저렴
- **랜덤 I/O가 없음** — 테이블은 각각 **한 번씩 Full Scan**만 하면 끝
- **인덱스 없어도 된다** — 대량 조인에서 인덱스 유무에 의존하지 않음

### 4.3.3 해시 조인이 유리한 경우

- **양쪽 데이터가 많고** NL 조인으로는 비용이 폭발할 때
- **후행 테이블에 인덱스가 없을 때**
- **한쪽이 작아서 메모리(Hash Area)에 올릴 수 있을 때**

### 4.3.4 해시 조인의 한계

- **등가 조인(equi-join)만 지원** — `=` 이 아닌 조건은 사용 불가
- **Hash Area가 부족하면 디스크 소트 발생** (Grace Hash Join, In-Memory Hash Join 구분)
  → 이때는 해시 조인의 장점이 많이 사라짐
- **Build 입력이 너무 크면** 해시 테이블 자체가 부담

### 4.3.5 NL vs Hash vs Sort Merge — 한눈에 비교

| 기준 | NL 조인 | 소트 머지 | 해시 조인 |
|---|---|---|---|
| **소량 조인** | ✅ 최고 | ❌ 과함 | ❌ 과함 |
| **대량 조인** | ❌ 느림 | ⚠️ 정렬 부담 | ✅ 강함 |
| **후행 인덱스 없음** | ❌ 느림 | ✅ OK | ✅ OK |
| **비등가 조인** | ✅ 가능 | ✅ 가능 | ❌ 불가 |
| **응답 속도 (첫 행)** | ✅ 즉시 | ❌ 전체 정렬 후 | ❌ Build 후 |

### 4.3.6 MySQL의 해시 조인

- **MySQL 8.0.18부터 해시 조인 지원**
- 8.0.20 이후로는 NL 조인보다 해시 조인을 기본 선택하는 케이스가 늘어남
- [MySQL 공식 문서 — Hash Join](https://dev.mysql.com/doc/refman/8.0/en/hash-joins.html)

## 📝 정리

- 해시 조인 = **"작은 쪽 해시 테이블 → 큰 쪽 Probe"**
- **대량 조인 + 인덱스 없음** 상황의 표준 해법
- 단, **등가 조인만 가능**, **Hash Area 부족 시 디스크로 떨어짐**
- MySQL 8.0.18+ 에서 사용 가능 — 버전 확인 필수
- 다음 편: **서브쿼리 조인** — 서브쿼리가 언제 조인으로 풀리는지
