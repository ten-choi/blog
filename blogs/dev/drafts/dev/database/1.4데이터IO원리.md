## 1. DBMS는 왜 "블록 단위"로 읽을까?

디스크에는 우리가 입력한 **한 줄(row)** 이 그대로 저장되어 있지 않아. DBMS는 데이터를 **블록(Block) 단위**로 묶어서 저장하고 읽어.

- **블록 크기**: 오라클 기본 8KB, MySQL InnoDB 기본 16KB
- 하나의 블록 안에 **여러 개의 행(row)** 이 들어있다

### 왜 한 줄씩 안 읽을까?

- 디스크와 파일시스템은 **고정 크기 단위**로만 I/O 를 처리한다. HDD 는 섹터, SSD 는 페이지 단위로 동작하고, DBMS 는 이를 묶어 블록이라는 단위를 만든다. 1바이트만 필요해도 그 1바이트가 속한 **블록 전체**가 전송된다.
- 하나의 I/O 요청에는 **블록 크기와 무관한 고정 비용**(커맨드 발행, 커널/드라이버 레이어 통과, HDD 면 헤드 이동(seek)과 회전 지연, SSD 면 컨트롤러의 매핑 조회)이 붙는다. 따라서 같은 데이터 양을 **여러 번 나눠 요청**하면 이 고정 비용이 매번 반복된다.


- 한 블록에는 여러 행(row)이 함께 들어있다. 블록 하나를 버퍼 캐시에 올려두면, 같은 블록 안에 있는 **다른 행을 읽는 다음 요청은 추가 디스크 I/O 없이** 메모리에서 처리된다.
- 운영체제 역시 파일을 페이지 단위로 캐싱하기 때문에, DBMS 블록 크기가 OS 페이지 크기와 정렬되면 **불필요한 복사·분할 비용**이 제거된다.

## 1.5 I/O 를 수행하는 두 컴포넌트

I/O 는 실행 엔진과 스토리지 엔진이 **역할을 나눠서** 처리해.

### ① 실행 엔진 (Execution Engine) — I/O 요청자

옵티마이저가 만든 실행 계획을 실행 엔진이 받아서, **"어떤 인덱스/테이블의 몇 번 블록이 필요한지"** 를 스토리지 엔진에 전달해. 실행 엔진은 **디스크에 직접 접근하지 않아** — 필요한 블록의 식별자만 알고 요청을 보낼 뿐이야.

### ② 스토리지 엔진 (Storage Engine) — I/O 수행 주체

실제 I/O 를 처리하고 **버퍼 캐시를 관리**하는 컴포넌트야.

- **논리적 I/O**: 스토리지 엔진이 관리하는 **버퍼 캐시(메모리)** 에서 요청된 블록을 찾는 경우. 디스크 접근은 발생하지 않아.
- **물리적 I/O**: 버퍼 캐시에 해당 블록이 없을 때, 스토리지 엔진이 **디스크/SSD** 에서 블록을 읽어 버퍼 캐시에 적재한 뒤 반환하는 경우.



## 2. 논리적 I/O vs 물리적 I/O

이 구분이 **튜닝의 핵심 용어**야. 외워두자.
논리적이란 표현은  ㅈ같다 logical io 와 physical io 가 좋다 한글로하면 뭔가 뉘앙스가 이상하다


| 구분 | 의미 | 발생 위치 | 속도 |
|---|---|---|---|
| **논리적 I/O (Logical I/O)** | **메모리(버퍼 캐시)** 에서 블록을 읽음 | RAM | 빠름 |          
| **물리적 I/O (Physical I/O)** | **디스크**에서 블록을 읽어서 메모리로 올림 | 디스크 + RAM | 느림 (수백~수천 배) |

### 흐름





```
SQL 요청
  ↓
"이 블록이 버퍼 캐시에 있나?"
  ├─ YES → 논리적 I/O (Cache Hit)  ✅ 빠름
  └─ NO  → 물리적 I/O (Cache Miss) → 디스크에서 읽어서 버퍼 캐시에 올림
              ↓
           이후 요청은 논리적 I/O로 처리
```

### Buffer Cache Hit Ratio

> **BCHR = 논리적 I/O 중 물리적 I/O 없이 해결된 비율**
>
> ```
> BCHR = (논리적 I/O - 물리적 I/O) / 논리적 I/O × 100
> ```

높을수록 좋지만, **"BCHR 99%라고 무조건 좋은 튜닝"은 아니다** — 애초에 논리적 I/O 자체를 줄이는 게 더 본질적인 튜닝이야.

### 🔑 튜닝 관점

- 대부분의 SQL 성능 저하는 **논리적 I/O가 많아서** 생김
- 즉 **"몇 블록을 읽느냐"** 가 가장 중요 → 이건 인덱스/조인 설계로 결정됨
- 물리적 I/O는 **캐시 효율**과 **I/O 서브시스템**의 문제

## 3. Single Block I/O vs Multiblock I/O

디스크에서 블록을 읽어올 때도 **"한 번에 몇 블록을 읽느냐"** 에 따라 속도가 완전히 달라져.

| 방식 | 설명 | 언제 쓰이나 |
|---|---|---|
| **Single Block I/O** | 한 번에 **1블록**씩 읽음 | **인덱스 스캔**, **ROWID로 테이블 액세스** |
| **Multiblock I/O** | 한 번에 **여러 블록**(8~128)을 묶어서 읽음 | **Table Full Scan**, **Index Fast Full Scan** |

### 두 방식의 실제 동작 차이

- **Single Block I/O** 는 한 번의 I/O 요청이 **1개 블록만 반환**한다. 100 블록이 필요하면 디스크와의 **요청-응답 왕복이 100번** 발생한다.
- **Multiblock I/O** 는 한 번의 I/O 요청으로 **연속된 여러 블록**(오라클 기본 최대 128개)을 한 번에 전송받는다. 100 블록이 필요하고 한 번에 32 블록씩 읽도록 설정됐다면 왕복은 약 **4번**으로 줄어든다.
- 요청 1회마다 드는 **고정 비용**(시스템 콜, 커맨드 발행, 커널/드라이버 전환, 디스크 준비)은 읽는 블록 수와 무관하게 거의 일정하다. 따라서 한 번에 많은 블록을 받을수록 **블록당 실비용이 내려간다**.
- Multiblock I/O 는 **물리적으로 연속된 블록**일 때 이득이 크다. 블록이 디스크 전역에 흩어져 있으면 내부적으로 또 다시 Single Block 요청들로 쪼개져 처리돼, Multiblock 으로 선언만 됐을 뿐 실제 이득은 사라진다.

### 성능 의미

- **전체의 20~25% 이상** 을 읽어야 한다면?
  → Single Block I/O를 많이 하느니 **Multiblock I/O로 전체를 긁는 Full Scan이 더 빠르다.**
  → 이게 바로 **"인덱스 손익분기점"** 의 정체

- 클러스터링 팩터(CF)가 좋으면 Single Block I/O의 효율이 올라가고, 손익분기점이 뒤로 밀린다

### 오라클 기준 파라미터

```
DB_BLOCK_SIZE             -- 블록 크기 (기본 8KB)
DB_FILE_MULTIBLOCK_READ_COUNT  -- Multiblock I/O 시 한 번에 읽을 블록 수 (기본 128)
```

## 4. 시퀀셜 액세스 vs 랜덤 액세스

I/O 효율을 결정하는 또 하나의 중요한 축이야.

| 구분 | 설명 | 예시 |
|---|---|---|
| **시퀀셜 액세스 (Sequential)** | **연속된 블록**을 순서대로 읽음 | Table Full Scan, Index Leaf Scan |
| **랜덤 액세스 (Random)** | 여기저기 **흩어진 블록**을 골라 읽음 | 인덱스 ROWID로 테이블 블록 찾기 |

### SSD 시대에도 유효한가?

> 과거 HDD에선 **디스크 헤드 이동(seek time)** 때문에 랜덤 액세스가 수십~수백 배 느렸음.
> SSD/NVMe에선 그 차이가 많이 줄었지만, **여전히 시퀀셜이 더 빠르고** DBMS의 옵티마이저도 그 가정으로 비용을 계산해.

## 5. 🧠 이게 튜닝에 왜 중요한가?

모든 SQL 튜닝의 목표는 결국 이 한 줄로 수렴한다.

> **"논리적 I/O를 줄여라. 특히 불필요한 Single Block 랜덤 액세스를 줄여라."**

이걸 염두에 두고 뒤에서 배울 주제들을 보면 왜 그런 기법들이 있는지 바로 보여.

| 튜닝 기법 | I/O 관점에서 하는 일 |
|---|---|
| **인덱스 컬럼 추가(Covered Index)** | 테이블 랜덤 액세스 자체를 제거 |
| **파티션 Pruning** | 읽을 블록 자체를 줄임 |
| **해시 조인** | 랜덤 액세스를 Full Scan(시퀀셜)로 전환 |
| **바인드 변수** | 하드 파싱 CPU 비용 절감 (I/O와는 다른 축) |
| **Direct Path I/O** | 버퍼 캐시 우회 + Multiblock I/O 활용 |

---

## 📝 정리

- DBMS는 **블록 단위**로 I/O를 한다 (오라클 8KB, MySQL 16KB)
- **논리적 I/O(메모리)** ≫ **물리적 I/O(디스크)** — 속도 차이 수백~수천 배
- **Single Block**(Index, ROWID) vs **Multiblock**(Full Scan) — 상황에 맞는 선택이 중요
- **시퀀셜**이 **랜덤**보다 여전히 빠름 (SSD에서도 유효)
- 튜닝 = **"읽어야 할 블록 수를 줄이는 일"**

> 다음 편은 **2장 인덱스** — 블록 I/O를 최소화하는 가장 강력한 도구, B-Tree 구조부터 본다.
