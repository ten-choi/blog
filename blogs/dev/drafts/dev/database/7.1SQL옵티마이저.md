 # 7.1 통계정보와 비용 계산 원리

## 7.1.1 선택도와 카디널리티


**선택도(Selectivity)**: 전체 레코드 중 조건절에 의해 **선택되는 레코드의 비율**이야.


'=' 조건으로 검색할 때는 이렇게 계산해.

`선택도 = 1 / NDV`

**NDV**: 컬럼 값 종류의 개수 (성별이라면 남/여 → 2)


**카디널리티(Cardinality)**: 전체 레코드 중 조건절에 의해 **선택되는 레코드 개수**야.

`카디널리티 = 총 로우 수 × 선택도 = 총 로우 수 / NDV`

> 💡 옵티마이저는 **카디널리티로 비용을 계산**하고, 그 비용을 기준으로 **어떤 실행 계획을 쓸지 결정**해. 그러니까 카디널리티의 기반이 되는 **선택도 값이 얼마나 정확한지가 튜닝의 핵심**이야.


## 7.1.2 통계정보

### 오브젝트 통계

#### 테이블 통계

테이블 통계 조회
```sql
select 
       num_rows -- 총 레코드 계수
     ,blocks -- 테이블 블록 수 = '사용된' 익스텐트에 속한 총 블록 수
     ,avg_row_len -- 레코드당 평균 길이
     ,sample_size -- 샘플링한 레코드 수
     ,last_analyzed -- 통계정보 수집일시
from all_tables
where owner='SCOTT'
and table_name = 'EMP'
;
```

#### 인덱스 통계


인덱스 통계 조회
```sql
select blevel -- 브랜치 레벨. 인덱스 루트에서 블록에 도달하기 직전까지 읽게되는 블록 수
     ,leaf_blocks  -- 인덱스 리프 블록 총 개수
     ,num_rows -- 인덱스에 저장된 레코드 개수
     ,distinct_keys -- 인덱스 키값의 조합으로 만들어지는 값의 종류 개수. c1(3) * c2(4) = 12. 선택도 계산에 사용.
     ,avg_leaf_blocks_per_key -- 인덱스 키값을 모두 '=' 조건으로 조회할 때 읽게 될 리프 블록 개수
     ,avg_data_blocks_per_key -- 인덱스 키값을 모두 '=' 조건으로 조회할 때 읽게 될 테이블 블록 개수
     ,clustering_factor -- 인덱스 키값 기준으로 테이블 데이터가 모여 있는 정도. 인덱스를 스캔하면서 테이블 레코드를 찾을 때 읽게 될 테이블 블록 개수를 미리 계산 한 수치
     ,sample_size
     ,last_analyzed
from all_indexes
where owner = 'SCOTT'
and table_name = 'EMP'
and index_name = 'EMP_X01'
;
    
    
```

#### 컬럼 통계

컬럼 통계 조회

```sql
select num_distinct -- 컬럼 값의 종류 개수(NDV)
     , density -- '=' 조건일 때 선택도. 1/NDV
     , avg_col_len -- 컬럼 평균 길이(bytes)
     , low_value -- 최소값
     , high_value -- 최대값
     , num_nulls -- null 인 레코드 수
     , last_analyzed
     , sample_size
from all_tab_columns
where owner = 'SCOTT'
and table_name = 'EMP'
and column_name = 'DEPTNO'
;
```

**컬럼 히스토그램**

'=' 조건의 선택도는 보통 `1/NDV` 나 `Density` 값을 활용해.

근데 **데이터 분포가 균일하지 않으면** 그대로 쓰기 어려워. 이런 경우 옵티마이저는 통계 외에 **히스토그램**을 써.

**히스토그램**: 컬럼 값별로 **데이터 비중이나 빈도를 미리 계산해놓은 통계 정보**. 실제 데이터를 샘플링해서 계산해둔 값이야.

- 히스토그램 유형
  - **도수분포(Frequency)** : 값별로 빈도수 저장
  - **높이균형(Height-Balanced)** : 각 버킷의 높이가 동일하도록 데이터 분포 관리
  - **상위도수분포(Top-Frequency)** : 많은 레코드를 가진 상위 n개 값에 대한 빈도수 저장
  - **하이브리드(Hybrid)** : 도수분포와 높이균형 특성 결합


히스토그램 조회

```sql
select endpoint_value, endpoint_number
from all_histograms
where owner = 'SCOTT'
and table_name = 'EMP'
and column_name = 'DEPTNO'
order by endpoint_value
;
```

### 시스템 통계

**시스템 통계**: 애플리케이션과 **하드웨어 성능 특성의 측정값**이야.

- CPU 속도
- 평균 Single Block I/O 속도
- 평균 Multiblock I/O 속도
- 평균 Multiblock I/O 개수
- I/O 서브시스템의 최대 처리량
- 병렬 Slave 의 평균 처리량

## 7.1.3 비용 계산 원리

단일 테이블을 인덱스로 액세스할 때의 비용 계산 원리를 볼게.

**'=' 조건 검색 시**

``` 
비용 = BLEVEL                     -- 인덱스 Vertical Traversal 비용
      + AVG_LEAF_BLOCKS_PER_KEY  -- 인덱스 Horizontal Traversal 비용
      + AVG_DATA_BLOCKS_PER_KEY  -- 테이블 랜덤 액세스 비용
```      

- blevel : 브랜치 레벨. 인덱스 루트에서 블록에 도달하기 직전까지 읽게되는 블록 수
- avg_leaf_blocks_per_key : 인덱스 키값을 모두 '=' 조건으로 조회할 때 읽게 될 리프 블록 개수
- avg_data_blocks_per_key : 인덱스 키값을 모두 '=' 조건으로 조회할 때 읽게 될 테이블 블록 개수


**'=' 조건이 아닐 때 (컬럼 통계까지 활용)**

```   
 비용 = BLEVEL                                  -- 인덱스 Vertical Traversal 비용
      + LEAF_BLOCKS       * 유효 인덱스 선택도    -- 인덱스 Horizontal Traversal 비용
      + CLUSERING_FACTOR  * 유효 테이블 선택도    -- 테이블 랜덤 액세스 비용
```

- leaf_blocks  : 인덱스 리프 블록 총 개수
- clustering_factor : 인덱스 키값 기준으로 테이블 데이터가 모여 있는 정도. 인덱스를 스캔하면서 테이블 레코드를 찾을 때 읽게 될 테이블 블록 개수를 미리 계산 한 수치
- 유효 인덱스 선택도 : 전체 인덱스 레코드 중 액세스 조건에 의해 선택될 것으로 예상되는 레코드 비중
- 유효 테이블 선택도 : 전체 인덱스 레코드 중 인덱스 컬럼에 대한 모든 조건절에 의해 선택 될 것으로 예상되는 레코드 비중


## 7.1.4 통계 정보 수집 및 관리

옵티마이저의 비용 계산은 전부 **통계 정보가 정확하다는 전제**에서 돌아가. 통계가 낡았거나 없으면, 좋은 원리도 **틀린 실행 계획**을 만들어낼 수밖에 없어.

### 수동 수집 명령

**오라클 — `DBMS_STATS` 패키지**

```sql
-- 특정 테이블의 통계 수집
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'SCOTT',
    tabname          => 'EMP',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,  -- 자동 샘플링 크기
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',  -- 히스토그램 자동
    cascade          => TRUE                          -- 관련 인덱스 통계도 수집
  );
END;
/

-- 스키마 전체
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(ownname => 'SCOTT');
END;
/
```

**MySQL**

```sql
ANALYZE TABLE emp;               -- 테이블 통계 갱신
ANALYZE TABLE emp, dept;         -- 여러 테이블 동시에

-- 히스토그램 (8.0+)
ANALYZE TABLE emp UPDATE HISTOGRAM ON sal WITH 32 BUCKETS;
ANALYZE TABLE emp DROP HISTOGRAM ON sal;
```

### 자동 수집 스케줄

**오라클**

- 기본적으로 **밤 10시 ~ 아침 2시, 주말 전일** 에 `auto optimizer stats collection` 자동 작업이 실행
- 내부적으로는 **통계가 부정확해진(Stale)** 테이블만 골라서 `DBMS_STATS` 를 돌림
- 확인:
  ```sql
  SELECT client_name, status FROM dba_autotask_client;
  ```

**MySQL (InnoDB)**

- `innodb_stats_auto_recalc = ON` (기본값) 이면 **테이블의 10% 이상 데이터가 변경**될 때마다 자동 재계산
- `innodb_stats_persistent = ON` (기본값) 으로 통계를 디스크에 영구 저장 (재시작해도 유지)

### Stale Statistics — "낡은 통계"의 위험

```
문제 시나리오:
1. 어제까지 테이블 1만 건, 통계도 1만 건으로 저장
2. 오늘 아침 배치로 1000만 건 insert
3. 쿼리 실행 — 옵티마이저는 여전히 "1만 건짜리 테이블" 로 계산
4. 실제론 큰 테이블인데 NL 조인 선택 → 대참사
```

- **"통계는 과거를 본다, 데이터는 현재를 산다"** — 이 간극이 실행계획 오류의 제1원인
- 수동 갱신을 해야 할 타이밍:
  - 대량 INSERT/UPDATE/DELETE 직후
  - 파티션 추가/삭제 직후
  - 인덱스 생성/삭제 직후
  - 평소엔 잘 돌던 쿼리가 갑자기 느려졌을 때 — **가장 먼저 통계부터 의심**

### Dynamic Sampling — 통계가 없을 때의 대안

통계가 아예 수집된 적 없거나 너무 낡은 테이블에 쿼리가 날아오면, 오라클은 **Dynamic Sampling (= Dynamic Statistics)** 을 수행할 수 있어.

- 실행 계획 수립 직전에 **쿼리 조건에 해당하는 데이터를 몇 개 블록 샘플링** 해서 임시 통계를 만듦
- 장점: 통계가 없어도 옵티마이저가 **완전히 틀린 계획을 세우지는 않음**
- 단점: **파싱 시점에 I/O 가 추가 발생** → 파싱 비용이 커짐
- 파라미터: `OPTIMIZER_DYNAMIC_SAMPLING` (0~11 레벨)

- [ ] 통계 수집 자체가 **서비스 시간대에 부하를 주지 않도록** 스케줄이 걸려있는가?
- [ ] 파티션 테이블이라면 **글로벌 + 파티션 양쪽 통계** 모두 수집하고 있는가?
