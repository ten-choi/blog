# 5.3 인덱스를 이용한 정렬 연산 생략

**인덱스는 이미 정렬된 자료구조**야. 이걸 그대로 읽으면 **추가 정렬 없이** 정렬된 결과를 얻을 수 있어. 이번 편은 이 성질을 적극적으로 활용하는 튜닝 기법들이야.

---

## 5.3.1 왜 "정렬 생략"이 중요한가?

OLTP 환경에서 흔히 요구되는 패턴을 떠올려봐:

- 최신 주문 20건
- 조회수 상위 10건
- 로그인 이력 최근 5건

이런 쿼리에 정렬을 매번 수행하면, **1000만 건을 다 읽고 정렬한 뒤 앞 10건만 반환**하는 식이 돼. 말도 안 되게 비효율적이야.

그런데 **적절한 인덱스가 있으면** 옵티마이저는 **"인덱스 순서대로 앞 N건만 읽고 멈춘다"** 는 **부분범위 처리**를 할 수 있어. 1000만 건이 1억 건이든 성능이 거의 일정하게 나와.

---

## 5.3.2 Sort Order By 생략 조건

옵티마이저가 `ORDER BY` 를 생략하려면 **정렬 키 순서가 인덱스 키 순서와 일치**해야 해.

### 기본 케이스

```sql
-- 인덱스: (created_at)
SELECT * FROM logs ORDER BY created_at;
```

→ 인덱스 리프 블록을 **순서대로 읽기만** 하면 정렬된 결과. **SORT ORDER BY 연산자 없음**.

### 결합 인덱스 케이스

```sql
-- 인덱스: (dept, sal)
SELECT * FROM emp WHERE dept = 10 ORDER BY sal;  -- ✅ 정렬 생략 가능
SELECT * FROM emp WHERE dept = 10 ORDER BY name; -- ❌ 인덱스 키 아님 → 정렬 필요
SELECT * FROM emp ORDER BY sal;                   -- ❌ 선두 컬럼 조건 없음
```

**규칙**:
- `WHERE` 의 등치 조건 컬럼과 `ORDER BY` 컬럼을 **합친 순서**가 인덱스 순서와 일치해야 함
- `ORDER BY` 에 인덱스에 없는 컬럼이 섞이면 생략 불가

---

## 5.3.3 ORDER BY + LIMIT / ROWNUM 조합 — 진짜 강력한 패턴

```sql
-- 인덱스: (created_at DESC) 또는 (created_at) 를 DESC 스캔
-- 최신 로그 20건
SELECT * FROM logs ORDER BY created_at DESC LIMIT 20;           -- MySQL / PostgreSQL
SELECT * FROM logs WHERE ROWNUM <= 20 ORDER BY created_at DESC; -- 오라클 (부적절 — 아래 참고)
SELECT * FROM logs ORDER BY created_at DESC FETCH FIRST 20 ROWS ONLY; -- 오라클 12c+
```

**옵티마이저가 하는 일:**
1. 인덱스 역순으로 리프 블록을 스캔 (Index Range Scan Descending)
2. 첫 20건을 모으는 순간 **스캔 중단**
3. 테이블에는 20건만 랜덤 액세스

→ 전체 건수와 **무관하게 일정 성능**. 이게 부분범위 처리의 위력.

### ⚠️ 오라클의 ROWNUM 함정

```sql
-- ❌ 잘못된 방법: 20건만 읽는 게 아니라 전체 정렬 후 20건 자르기
SELECT * FROM logs WHERE ROWNUM <= 20 ORDER BY created_at DESC;
```

- `WHERE ROWNUM` 은 **ORDER BY 보다 먼저 적용**돼
- 결국 **임의의 20건을 가져온 뒤 그걸 정렬**하는 꼴이 됨 (틀린 결과)

**올바른 방법**:

```sql
-- Inline View 로 감싸서 ORDER BY 가 먼저 적용되게
SELECT * FROM (
  SELECT * FROM logs ORDER BY created_at DESC
) WHERE ROWNUM <= 20;

-- 또는 12c+ 의 FETCH FIRST
SELECT * FROM logs ORDER BY created_at DESC FETCH FIRST 20 ROWS ONLY;
```

## 5.3.4 DESC 정렬과 인덱스

```sql
-- 인덱스: (created_at ASC)
SELECT * FROM logs ORDER BY created_at DESC;
```

옵티마이저는 **Index Range Scan Descending** 으로 **오름차순 인덱스를 역순으로 읽어서** 정렬 없이 DESC 결과를 반환할 수 있어.

### 복합 정렬의 경우

```sql
-- 인덱스: (created_at, user_id)
SELECT * FROM logs ORDER BY created_at DESC, user_id DESC; -- ✅ 역순 스캔으로 해결
SELECT * FROM logs ORDER BY created_at DESC, user_id ASC;  -- ❌ 방향이 섞임 → 정렬 필요
```

- **모든 정렬 컬럼의 방향이 동일**하면 정렬 생략 가능
- 방향이 섞이면 별도 정렬 필요 → **Descending Index 를 고려** (오라클 / MySQL 8.0+)

## 5.3.5 MIN / MAX 최적화

```sql
-- 인덱스: (created_at)
SELECT MIN(created_at) FROM logs;  -- 인덱스의 '첫 리프' 1건만 읽음
SELECT MAX(created_at) FROM logs;  -- 인덱스의 '마지막 리프' 1건만 읽음
```

**실행 계획**:
```
INDEX FULL SCAN (MIN/MAX)
  -- 또는
FIRST ROW / LAST ROW 최적화
```

- 인덱스의 **첫 번째 또는 마지막 리프 블록만 읽고 즉시 반환**
- 테이블은 전혀 안 읽음 → **O(1)** 수준의 성능

### ⚠️ 함수가 씌워지면 망가짐

```sql
SELECT MIN(YEAR(created_at)) FROM logs;  -- ❌ 함수 때문에 Full Scan
SELECT YEAR(MIN(created_at)) FROM logs;  -- ✅ MIN 먼저 → 결과에 함수 적용
```

## 5.3.6 Pagination 튜닝 — OFFSET 안티패턴

```sql
-- ❌ 안티패턴: OFFSET 100000 LIMIT 20
SELECT * FROM logs ORDER BY created_at DESC LIMIT 20 OFFSET 100000;
```

- OFFSET 100,000 은 **앞의 10만 건을 읽고 버린 뒤** 그 다음 20건을 반환
- 페이지가 깊어질수록 **성능이 선형으로 나빠짐**

### ✅ 커서 기반 페이징 (Keyset Pagination)

```sql
-- 첫 페이지
SELECT * FROM logs ORDER BY created_at DESC LIMIT 20;

-- 다음 페이지: 마지막 created_at 값을 조건으로
SELECT * FROM logs
WHERE  created_at < :last_created_at
ORDER BY created_at DESC
LIMIT 20;
```

- 매 페이지가 **앞의 건수와 무관하게 일정 성능**
- 단점: **임의의 N 페이지로 점프 불가** (순차 이동만 가능)
- 무한 스크롤이나 피드에 가장 잘 맞는 패턴

---

## 5.3.7 3-Tier 환경에서의 주의

- **클라이언트 - WAS - DB** 로 구성된 현대 서버 구조에선, 수많은 클라이언트가 서버 리소스를 공유해
- 따라서 특정 클라이언트가 **DB 커넥션을 오래 잡고 있는 상황**은 피해야 함
- **거대한 결과셋을 전체 가져와 클라이언트에서 정렬** 하는 건 최악 — 커넥션 풀 고갈의 원인
- **DB 에서 정렬을 끝내고 상위 N 건만 전달** 하는 게 정답. 이때 **인덱스로 정렬 생략 + LIMIT** 이 빛을 발해

---

## 🔑 이번 편 핵심

- 인덱스는 **이미 정렬된 자료구조** → `SORT ORDER BY` 연산자를 제거 가능
- 정렬 생략 조건: **WHERE 등치 컬럼 + ORDER BY 컬럼** 이 **인덱스 순서와 정확히 일치**
- **ORDER BY + LIMIT** 은 부분범위 처리의 대표 패턴 — 전체 건수와 무관한 일정 성능
- **MIN/MAX** 는 인덱스의 양 끝 블록만 읽고 끝 — O(1) 수준
- **OFFSET 페이징** 은 안티패턴. **커서 기반 페이징**으로 대체
- 오라클의 **ROWNUM + ORDER BY** 는 순서에 주의 (Inline View 또는 `FETCH FIRST` 사용)
