## 🚀 DB 인덱스 스캔 방식 한눈에 보기

인덱스를 어떻게 타느냐에 따라 쿼리 속도가 천차만별이야. 상황별로 어떤 방식이 쓰이는지 보자.

### 1. 기본 중의 기본 스캔들
* **Index Range Scan (정석)**
    * 가장 일반적인 방식이야. 루트에서 리프까지 수직으로 내려간 뒤, 필요한 **범위(Range) 만** 슥 긁어오는 거지.
    * **주의**: 인덱스 선두 컬럼을 가공(`WHERE SUBSTR(이름, 1, 1) = ...` 등)하면 이 방식을 못 써!
* **Index Full Scan (차선책)**
    * Vertical Traversal 없이 **리프 블록을 처음부터 끝까지** 다 읽어. 보통 적절한 인덱스가 없을 때 "그래도 테이블 풀스캔보단 낫겠지" 하고 선택되는 방식이야.
* **Index Unique Scan (초고속)**
    * PK 나 Unique 인덱스를 `=` 조건으로 찾을 때 발생해. "딱 하나만" 찾으면 끝이니까 스캔 범위 자체가 없어서 성능이 최고야.



### 2. 효율을 극대화하는 스캔들
* **Index Skip Scan (점프 스캔)**
    * 선두 컬럼 조건이 없어도 인덱스를 탈 수 있게 해주는 방식이야. 선두 컬럼의 종류(Cardinality)가 적고 후행 컬럼 종류가 많을 때, **조건에 맞는 리프 블록만 건너뛰며** 읽는 방식이야.
* **Index Fast Full Scan (속도 위주)**
    * Index Full Scan 보다 훨씬 빨라. 인덱스의 논리적 순서는 무시하고 **물리적으로 한꺼번에 (Multiblock I/O)** 다 읽어버리거든. 단, **결과가 정렬되지 않는다는 점** 주의!
* **Index Range Scan Descending (역순 스캔)**
    * Range Scan 이랑 동일하지만 **뒤에서부터** 읽어. `ORDER BY ... DESC` 같은 조건이 있을 때 별도 정렬 없이 결과를 바로 얻을 수 있지.

## 📊 MySQL 실행계획(Execution Plan) 체크 포인트

쿼리가 의도한 대로 도는지 확인하려면 `EXPLAIN` 을 꼭 열어봐. 이 항목들만 봐도 답이 나와.

| 항목 | 의미 | 체크 포인트 ✅ |
| :--- | :--- | :--- |
| **type** | 조인/스캔 방식 | `system` > `const` > `range` 순으로 좋고, `ALL` (풀스캔) 은 피해야 해 |
| **key** | 실제 사용 인덱스 | 내가 의도한 인덱스가 들어가 있는지 확인! |
| **rows** | 예측 탐색 행 수 | 이 숫자가 너무 크면 인덱스가 제대로 일을 안 하고 있다는 신호야 |
| **Extra** | 추가 정보 | `Using filesort` 나 `Using temporary` 가 뜨면 성능 빨간불 🚨 |

---

### 💡 꿀팁 요약
인덱스를 탄다고 무조건 빠른 게 아니야. **"얼마나 적게 스캔하고, 테이블에 얼마나 덜 접근하느냐"** 가 진짜 실력이지. 실행계획의 `type`이 `range`나 `const`로 나오도록 쿼리를 잘 짜는 게 핵심!

## 🧩 그 외 "확장 인덱스" 기능들

위에서 본 건 **"어떻게 스캔하느냐"** 에 대한 방식이었고, 지금부터는 **"어떤 인덱스를 만들 수 있느냐"** 에 대한 확장 기능들이야.

### 1. 함수 기반 인덱스 (Function-Based Index)

컬럼을 **가공한 결과 자체를 인덱스**로 만들어버리는 기법이야.

```sql
-- ❌ 일반 인덱스로는 못 탐
SELECT * FROM users WHERE UPPER(email) = 'FOO@BAR.COM';

-- ✅ 함수 기반 인덱스 생성
CREATE INDEX idx_users_upper_email ON users (UPPER(email));

-- 이제 옵티마이저가 인덱스를 쓸 수 있음
SELECT * FROM users WHERE UPPER(email) = 'FOO@BAR.COM';
```

- **언제 쓰나**: `UPPER`, `LOWER`, `SUBSTR`, 날짜 포맷 같이 **조건절에서 자주 쓰는 함수 결과**
- **주의**: 인덱스 컬럼이 실제 저장 컬럼이 아니라 **"함수 적용 결과"** 이기 때문에, DML 시 매번 함수 평가가 일어나 **DML 비용이 일반 인덱스보다 큼**
- **MySQL**: `CREATE INDEX ... ((UPPER(email)))` 구문 지원 (8.0.13+) 또는 **Generated Column + 인덱스** 조합

### 2. Invisible Index (무효화 인덱스)

인덱스를 **"옵티마이저에게만 안 보이게"** 숨기는 기능. 실제 인덱스는 존재하지만 쿼리 최적화에서는 무시돼.

```sql
-- 오라클
ALTER INDEX idx_foo INVISIBLE;
ALTER INDEX idx_foo VISIBLE;

-- MySQL 8.0+
ALTER TABLE users ALTER INDEX idx_foo INVISIBLE;
ALTER TABLE users ALTER INDEX idx_foo VISIBLE;
```

- **언제 쓰나**:
  - 인덱스를 **삭제하기 전에 "정말 안 써도 되나?" 테스트** — invisible 로 바꿔놓고 모니터링하다 문제 없으면 DROP
  - **특정 세션에서만 쓰고 싶을 때** — 세션 파라미터로 invisible 인덱스도 보이게 설정 가능
- **장점**: 롤백이 즉시 가능. DROP 했다가 다시 만드는 건 대용량 테이블에서 매우 비싼 작업

### 3. Descending Index (내림차순 인덱스)

인덱스를 **DESC 방향으로 저장**하는 기능.

```sql
-- 최신순 조회가 많은 컬럼을 아예 DESC 로 저장
CREATE INDEX idx_logs_created_desc ON logs (created_at DESC);
```

- **MySQL**: 8.0 부터 "진짜 DESC 인덱스" 지원. 그 이전은 DESC 키워드를 무시하고 그냥 ASC 로 만들었음
- **오라클**: 오래 전부터 지원
- **언제 유용한가**: **ORDER BY ... DESC + LIMIT** 패턴이 빈번할 때. 이미 DESC 로 저장되어 있으니 역순 스캔조차 불필요

### 4. Partial / Filtered Index

테이블의 **일부 행에만 인덱스를 만드는** 기능. 인덱스 크기를 줄이고 유지 비용도 낮춤.

```sql
-- PostgreSQL / SQLite
CREATE INDEX idx_active_users ON users (email) WHERE status = 'ACTIVE';

-- SQL Server (Filtered Index)
CREATE INDEX idx_active_users ON users (email) WHERE status = 'ACTIVE';
```

- **언제 쓰나**:
  - 테이블 1억 건 중 **"활성 사용자"는 100만 건뿐** 이고, 쿼리는 거의 활성 사용자만 찾는 경우
  - → 인덱스 크기가 **1/100 로 줄어들고** DML 부담도 그만큼 감소
- **MySQL**: 직접 지원은 없고, **Generated Column + 인덱스** 로 우회 가능
- **오라클**: 공식 Partial Index 는 없지만 **Function-Based Index 로 유사 효과** (`NVL(CASE WHEN status='ACTIVE' THEN email END, ...)`)

### 5. 다중 값 인덱스 (Multi-Value Index, MySQL 8.0.17+)

JSON 배열 안의 값에 인덱스를 거는 기능.

```sql
CREATE INDEX idx_tags ON articles ( (CAST(tags AS CHAR(20) ARRAY)) );

-- 활용
SELECT * FROM articles WHERE 'javascript' MEMBER OF (tags);
```

- JSON 필드를 자주 검색하는 현대 애플리케이션에 유용

### 6. 공간 인덱스 (Spatial Index) — R-Tree

좌표 / 영역 검색에 특화된 인덱스.

```sql
CREATE SPATIAL INDEX idx_location ON places (geom);
SELECT * FROM places WHERE ST_Contains(ST_GeomFromText('POLYGON(...)'), geom);
```

- B-Tree 와 다른 **R-Tree** 구조
- 위치 기반 서비스, GIS 에 필수
