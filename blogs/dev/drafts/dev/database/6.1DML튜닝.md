### 인덱스와 DML 성능
#### (1) 입력
테이블에 레코드를 입력하면, 인덱스에도 입력해야 한다.

인덱스는 정렬된 자료구조이므로, Vertical Traversal을 통해 입력할 블록을 찾아야 한다.

#### (2) DELETE
테이블에서 레코드 하나를 삭제하면, 인덱스 레코드를 모두 찾아서 삭제해야한다.

#### (3) UPDATE
변경된 컬럼을 참조하는 인덱스만 찾아서 변경

그대신 테이블에서 한 건 변경할 때마다 인덱스에는 두 개의 오퍼레이션 발생

### 무결성 제약과 DML 성능
PK, FK, Check, Not Null 같은 제약은 애플리케이션에서 직접 구현할 수도 있지만, **DBMS 에서 설정하는 쪽이 훨씬 완벽하게 데이터 무결성을 보장**해.

다만 성능과 **트레이드오프**하는 조건이라는 점은 꼭 기억해둬야 해.

PK, FK 는 검증할 때 **실제 데이터를 조회해야 하기 때문**에 추가 비용이 발생하거든.


<br>

### 조건절과 DML 성능
UPDATE / DELETE 는 결국 **"대상 레코드를 먼저 찾은 뒤 변경하는"** 두 단계 작업이야.

- 변경할 레코드를 찾는 비용은 **SELECT 성능**과 똑같이 **조건절 + 인덱스**에 좌우돼.
- 즉 `WHERE` 조건이 인덱스를 못 타면 변경 자체가 아니라 **"찾는 단계"** 부터 이미 느려져 있는 거야.
- 대량 UPDATE/DELETE 에선 **변경 대상을 찾는 I/O 비용**이 전체 성능의 절반 이상을 먹는 경우가 흔해.

> 💡 **튜닝 포인트**: UPDATE 가 느리다고 UPDATE 만 보지 말고, **`SELECT ... WHERE 조건`을 먼저 빠르게 만드는 게** 대부분의 해법이야.


<br>

### 서브쿼리와 DML 성능
`INSERT ... SELECT`, `UPDATE ... WHERE 컬럼 IN (SELECT ...)`, `MERGE` 처럼 DML 안에 서브쿼리가 들어가는 경우가 정말 많아.

- 서브쿼리의 실행 계획이 **DML 전체 성능을 결정**한다고 봐야 해.
- 서브쿼리가 매 행마다 재실행되는 **필터(Filter) 방식**으로 풀리면 재앙이야 → 조인 형태로 풀리도록 **Unnesting 을 유도**해야 돼.
- `INSERT ... SELECT` 같은 대량 적재 상황에선 서브쿼리 쪽이 대부분의 I/O 를 차지하니까, **서브쿼리 튜닝이 곧 DML 튜닝**이야.

> 💡 서브쿼리 조인 최적화는 [조인 튜닝 (4) — 서브쿼리 조인](./4.4조인튜닝.md) 편에서 자세히 다룰게.


<br>

### Redo 로깅과 DML 성능
#### Redo 로그
오라클은 데이터 파일과 컨트롤 파일에서 발생하는 **모든 변경사항을 Redo 로그에 기록**해.

Redo 로그는 트랜잭션 데이터가 유실됐을 때 **트랜잭션을 재현해서 유실 이전 상태로 복구**하는 데 써.

DML 을 수행할 때마다 Redo 로그를 생성해야 하니까 성능에 영향을 미쳐.

단, INSERT 작업에 한해서는 Redo 로깅을 **생략**할 수도 있어 (Direct Path Insert 등).

<br>

#### Redo 로그 목적
**(1) Database Recovery**

물리적으로 디스크가 깨지는 등의 media fail 이 발생했을 때 복구하는 용도로 써.

**(2) Cache Recovery**

버퍼 캐시는 I/O 성능을 올리기 위한 공간이지만 **휘발성**이야.

인스턴스가 종료되면 버퍼 캐시가 날아가니까, 이걸 복구하기 위해 Redo 로그를 써.

**(3) Fast Commit**

변경사항을 일단 **Append 방식**으로 빠르게 로그 파일에 기록만 해두고,

실제 메모리 버퍼 블록과 데이터 파일 블록 간 동기화는 **DBWR(DB Writer)·Checkpoint 프로세스**가 나중에 배치로 일괄 처리해.

> 💡 변경된 메모리 버퍼 블록을 디스크 데이터 블록에 반영하는 건 **랜덤 액세스**라서 매우 느려. 그래서 Redo 로그는 일단 **Append(순차 쓰기)** 로 빠르게 기록만 해두고, 실제 반영은 뒤로 미루는 방식을 쓰는 거야.

<br>

### Undo 로깅과 DML 성능
Undo 는 **트랜잭션을 롤백해서 현재를 과거 상태로 되돌리는 데** 쓰는 로그야.


<img width="800" src="https://user-images.githubusercontent.com/60383031/149776429-55b098c8-7cde-426e-b177-1f493167b2a7.png">

오라클은 데이터를 **입력·수정·삭제**할 때마다 Undo 세그먼트에 기록을 남겨둬.

<br>

#### Undo 로그 목적
**(1) Transaction RollBack**

트랜잭션에 의한 변경사항을 최종 커밋하지 않고 **롤백할 때** 써.

**(2) Transaction Recovery**

Redo 를 이용한 roll forward 단계가 끝나면 **커밋되지 않은 변경사항까지 모두 복구**돼 있어.

그래서 셧다운 시점에 아직 커밋되지 않았던 트랜잭션들을 다시 롤백해야 하는데, **이때 Undo 를 사용**해.


**(3) Read Consistency**

**데이터 읽기 일관성**을 확보하기 위해 사용돼 (MVCC 의 핵심).

<br>

#### MVCC 모델 (Multi-Version Concurrency Control Model)
**(1) Current 모드**

디스크에서 캐시로 적재된 **원본(current) 블록을 현재 상태 그대로** 읽는 방식이야.


**(2) Consistent 모드**

다른 트랜잭션이 변경한 블록을 만나면, **원본 블록으로부터 복사본 블록**을 만들고 거기에 **Undo 데이터를 적용**해서 "과거 시점의 모습"을 재현해.


<img width="800" src="https://user-images.githubusercontent.com/60383031/149789962-6452c001-b8c5-4a55-87db-ca3960454cc7.png">


**※ SCN (System Commit Number)**

마지막 커밋이 발생한 시점 정보를 담고 있는 **Global 변수 값**이야.

**※ 블록 SCN**

각 블록이 마지막으로 변경된 시점을 추적하기 위해 **모든 블록 헤더에 SCN 을 기록**해둬.

**※ 쿼리 SCN**

모든 쿼리는 시작할 때 **Global SCN 값을 먼저 확인**하고 읽기 작업을 시작해. 이걸 "쿼리 SCN" 이라고 불러.

**※ 정리하면**

Consistent 모드는 **쿼리 SCN 과 블록 SCN 을 비교**해서, 쿼리가 도는 도중에 블록이 변경됐는지 확인하며 데이터를 읽는 방식이야.

읽다가 **블록 SCN 이 쿼리 SCN 보다 더 큰 블록**을 만나면 → 복사본 블록을 만들고 Undo 를 적용해서 **쿼리 시작 시점의 모습**으로 되돌려 읽어.

### 커밋과 DML 성능
커밋(Commit)은 단순히 "변경을 확정"하는 명령이 아니라, **물리적 I/O 와 로그 쓰기를 동반하는 무거운 작업**이야.

- 커밋 시점에 **Redo 로그 버퍼를 디스크 Redo 로그 파일로 플러시** (LGWR) → **디스크 동기 I/O 발생**
- 커밋 시점에 **Undo 세그먼트 정리** 및 블록 헤더의 SCN 갱신
- 너무 자주 커밋하면?
  - Redo 로그 파일 쓰기가 반복돼서 **"log file sync" 대기 이벤트**로 전체 처리량 하락
  - 예: 100만 건 루프에서 **1건마다 커밋** → 파일 I/O 100만 번 발생
- 너무 드물게 커밋하면?
  - **Undo 세그먼트 팽창** 및 **ORA-01555 (Snapshot too old)** 에러
  - 장시간 트랜잭션 → 다른 세션의 **블로킹** 유발

> 💡 **실무 가이드**
> - 대량 배치는 **수천 ~ 수만 건 단위로 묶어서** 커밋 (배치 커밋)
> - OLTP 트랜잭션은 **비즈니스 단위**로만 커밋 (원자성 유지)
> - "빠르다고 1건마다 커밋" — 가장 자주 만나는 **안티패턴**
