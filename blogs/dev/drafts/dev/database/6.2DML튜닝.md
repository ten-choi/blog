> 대량 INSERT/UPDATE/DELETE를 할 때 **가장 먼저 만나는 벽**은 **버퍼 캐시 경합**과 **Redo/Undo 로깅 부하**다.
> Direct Path I/O는 이 둘을 한 번에 회피하는 대량 처리의 표준 기법이다.

---

## 6.2 Direct Path I/O

### 6.2.1 일반 I/O vs Direct Path I/O

**일반 I/O (Conventional Path)**

```
사용자 SQL → 버퍼 캐시 → (필요 시) 디스크
  - 버퍼 캐시 경합 발생
  - Redo 로깅 수행
  - Undo 로깅 수행
  - Free List 탐색 (빈 블록 찾기)
```

**Direct Path I/O**

```
사용자 SQL → 디스크에 직접 쓰기 (버퍼 캐시 우회)
  - 버퍼 캐시 건너뜀
  - Redo 로깅 최소화 or 생략
  - 테이블 High Water Mark(HWM) 위에 새 블록을 바로 append
```

### 6.2.2 어떤 상황에서 쓰나?

- 수백만 ~ 수억 건의 **대량 배치 INSERT**
- **CTAS** (`CREATE TABLE AS SELECT`)
- **SQL*Loader Direct Path** 모드
- **Parallel DML**
- 정렬 결과가 PGA를 초과해 **Temp 테이블 스페이스**에 쓸 때

### 6.2.3 오라클 기준 활성화 방법

```sql
-- INSERT ... SELECT 를 Direct Path로
INSERT /*+ APPEND */ INTO target_table
SELECT * FROM source_table;

-- Redo 로깅까지 끄려면 (테이블 속성 변경)
ALTER TABLE target_table NOLOGGING;
```

### 6.2.4 Direct Path의 "비용"

공짜는 아니야 — 대가가 있다.

| 얻는 것 | 잃는 것 |
|---|---|
| 버퍼 캐시 경합 회피 | **해당 세션에서 같은 테이블 읽기 불가** (커밋 전) |
| Redo/Undo 부하 급감 | **Media Recovery 불가** (nologging 시) |
| HWM 위 append로 빠름 | **블록 내부 빈 공간 재활용 안 함** → 테이블 크기 팽창 |
| 병렬 처리와 궁합 좋음 | **FK/Trigger 있으면 Direct Path 거부** |

### 6.2.5 주의사항

- **Direct Path INSERT 후에는 반드시 COMMIT** 해야 해당 테이블을 다시 조회 가능
- **NOLOGGING은 복구 불가 리스크** — 배치 직후 **백업** 필요
- **인덱스가 있으면 여전히 비용 발생** — 대량 적재 전에 **인덱스 DROP → 적재 → 재생성** 전략 고려
- MySQL에는 Direct Path I/O라는 이름의 기능은 없지만, 유사한 목적으로 **`LOAD DATA INFILE`**, **`INSERT ... SELECT`의 `DISABLE KEYS`** 등을 사용

### 6.2.6 체크리스트

대량 적재 전에 스스로 물어보자.

- [ ] 건수가 **100만 건 이상**인가? (그 이하면 일반 INSERT로 충분)
- [ ] 대상 테이블에 **FK/Trigger**가 있는가? → Direct Path 불가
- [ ] 적재 중 같은 테이블을 **다른 세션이 조회**해야 하는가?
- [ ] **복구 정책상 NOLOGGING**이 허용되는가?
- [ ] 인덱스를 **일시적으로 내려도** 서비스에 영향이 없는가?

---

## 📝 정리

- Direct Path = **버퍼 캐시 우회 + Redo/Undo 최소화**
- 대량 배치의 표준 해법이지만 **복구·동시성 제약**이 따라옴
- `/*+ APPEND */` + `NOLOGGING` + 인덱스 drop/rebuild 조합이 전통적 레시피
- MySQL에선 `LOAD DATA INFILE` 같은 벌크 로딩 도구로 비슷한 효과
