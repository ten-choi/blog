## 1. Why do we need an index?

By default, a table just stores rows **in the order they were inserted**. It keeps no ordering information about where any particular column value lives. So to find a specific value, the database has to **read every block in sequence and compare the target column of every row**. That brute-force read is a **Table Full Scan**.

> **A quick word on "blocks."**
> A DBMS doesn't read the disk one row at a time — it reads in fixed-size chunks called **blocks** (commonly around 8 KB). Why blocks at all? Because *reaching* data on disk is expensive, so once you've paid that cost it's worth grabbing a chunk of neighboring data at once — that helps with locality, caching, concurrency, and recovery all in one design choice. For now, just think of a block as "the smallest unit the DBMS reads from or writes to storage."

An **index** is a separate data structure that **pre-sorts the values of a specific column and records where each value physically lives in the table (its ROWID)**. Instead of scanning every block, the database quickly locates where the matching value sits inside the index, pulls out its ROWID, and reads **only the table blocks it actually needs**.

> **What is a ROWID?**
> A ROWID answers the question **"where on disk is this row stored?"** In Oracle every row automatically carries a ROWID, and the database uses it to jump straight to the row's physical location. An Oracle ROWID encodes:
>
> ```
> AAASPmAAFAAAABSAAA
> │     │    │      │
> │     │    │      └─ Row Number   (which row inside the block)
> │     │    └──────── Block Number  (which block inside the file)
> │     └───────────── File Number   (which data file)
> └─────────────────── Object Number (which table / object)
> ```

## 2. The 3-layer structure of a B-Tree index

Almost every database index is a **B-Tree** (Balanced Tree). Just as the name says, it's a **balanced** tree.

```
                 [ Root Block ]               ← Level 1
                /      |      \
         [Branch] [Branch] [Branch]           ← Level 2 (can be several levels)
          /  |  \    |    /  |  \
       [Leaf][Leaf][Leaf][Leaf][Leaf]         ← Level 3 (bottom)
        ↓    ↓    ↓    ↓    ↓
     (sorted keys + ROWIDs)
```

| Node | Role |
|---|---|
| **Root** | The entry point of the tree. There is exactly one |
| **Branch** | Intermediate node. A signpost that points down into a narrower range |
| **Leaf** | Stores the actual **key value + ROWID** (table location). Linked to neighboring leaves (a linked list) |

Because of this balanced shape, a B-Tree (more precisely, the B+Tree that databases use) searches in roughly **logₘ(n)** time.

### Key points

- **Leaf blocks are sorted by key value** → range searches are efficient.
- **Leaf blocks are doubly linked to one another** → moving to the neighboring value is cheap.
- **Every leaf sits at the same depth** (balanced) → any value can be reached at the same cost.

## 3. Using an index = "Vertical Traversal + Horizontal Traversal"

"Using an index" really means performing **two phases of search**.

### ① Vertical Traversal

> The process of **descending** root → branch → leaf.
> This step finds the **starting position (the leaf block)** of the value you're looking for.

- Reads one block per tree level (BLEVEL).
- Even an index over a million rows usually reaches a leaf in **3–4 levels** (logarithmic time complexity).

### ② Horizontal Traversal

> The process of **scanning sideways** across leaf blocks to read the data.
> This is the real body of the query — **the size of the scan range decides performance**.

```
Root → Branch → Leaf reached!   ← Vertical Traversal ends here
                  │
                  ↓ sideways ─→ ─→ ─→ ─→ ─→   ← Horizontal Traversal
                  (as far as needed)
```

### 🔑 Where does the time actually go?

| Traversal | Cost | Note |
|---|---|---|
| Vertical Traversal | Nearly constant (3–4 blocks) | The tree is shallow |
| **Horizontal Traversal** | **Proportional to the scan range** | **The heart of tuning** |

> So tuning ultimately comes down to **"shrinking the Horizontal Traversal range."**
> Leading-column conditions, composite-index column order, access conditions vs. filter conditions — every one of these techniques exists to **narrow the Horizontal Traversal**.

## 4. Finding the table data from an index — ROWID

A leaf holds only the **key value + ROWID** — never the actual record.

```
[Leaf Block]
 (James Smith,  ROWID=0x12AB3...)
 (Mary Johnson, ROWID=0x12AB8...)
 (Robert Brown, ROWID=0x12AC1...)
       ↓ follow the ROWID to the table block (random access)
[Table Block]
 the full row data
```

- A ROWID is the **physical location address inside the table**.
- Following that ROWID to the table block is **"table random access."** 🎯
- The more rows you have to read, the more this random-access cost piles up — past a breakeven point, a **Full Scan becomes faster**.

---

## 5. Searching a composite index

A composite (multi-column) index is the exact same B-Tree structure. The only difference is that its **sort key is a compound key**.

```
Index (dept, name)

[Leaf blocks — sorted by dept → then name]
 (Sales, Alice Brown)
 (Sales, Bob Carter)
 (Sales, David Lee)
 (HR,    Emma Davis)    ← when dept changes, name restarts from the top
 (HR,    Frank Miller)
 (HR,    Grace Wilson)
```

### Why does the "leading column" matter?

- **dept = 'Sales'** → can find the starting point ✅
- Given only **name = 'Frank Miller'**? → no way to know which dept's Frank Miller it is. You'd have to scan every leaf ❌

This is the real reason behind the rule **"if the leading column isn't in the WHERE clause, the index can't be used."**

---

## 6. 📝 Summary — why a B-Tree?

1. **It's a balanced tree, so the search cost is nearly constant for any value.**
2. **Leaves are sorted and linked, so range searches are efficient.**
3. **Vertical Traversal is O(log N); Horizontal Traversal is proportional to the range** → the tuning point is obvious.
4. **Table access goes through the ROWID** → the random I/O in this step is the central subject of tuning.
