We've walked through how SQL gets processed — by now you should have a basic feel for it. Now it's time to look at **"where data is actually read from, and how."**

99% of SQL tuning ultimately boils down to **reducing I/O**.

The trick isn't to make things move faster — it's to make them not move at all.

---

## 1. Why does a DBMS read in "block units"?

A row you inserted is not stored on disk as a single row by itself. The DBMS stores and reads data in **blocks**.

- **Block size**: Oracle defaults to 8 KB, MySQL InnoDB defaults to 16 KB
- A single block contains **multiple rows**

### Why not read one row at a time?

- Disks and filesystems only process I/O in **fixed-size units**. HDDs operate at the sector level, SSDs at the page level, and the DBMS bundles these into a unit called a block. Even if you only need 1 byte, the **entire block** that contains that byte is transferred.
- Every I/O request carries a **fixed cost that's independent of block size** (issuing the command, traversing kernel/driver layers, head movement (seek) and rotational delay on HDDs, controller mapping lookups on SSDs). Splitting the same amount of data into **many small requests** pays this fixed cost every time.
- A single block contains multiple rows. Once that block is in the buffer cache, **any subsequent request that needs another row in the same block** can be served from memory with no additional disk I/O.
- The OS also caches files in page units, so when the DBMS block size aligns with the OS page size, **unnecessary copy and split costs** disappear.

---

## 2. The two components that perform I/O

I/O is handled jointly by the execution engine and the storage engine — they split the work.

### ① Execution Engine — the I/O requester

The execution engine receives the execution plan the optimizer built and tells the storage engine **"which block, of which index/table, do I need."** The execution engine **does not access the disk directly** — it just knows the identifier of the block it wants and sends a request.

### ② Storage Engine — the I/O performer

This is the component that actually performs I/O and **manages the buffer cache**.

- **Logical I/O**: The storage engine finds the requested block in the **buffer cache (memory)** it manages. No disk access happens.
- **Physical I/O**: When the buffer cache doesn't have the block, the storage engine reads it from **disk/SSD**, loads it into the buffer cache, then returns it.

---

## 3. Logical I/O vs Physical I/O

This distinction is **the core vocabulary of tuning**. Memorize it.

| Type | Meaning | Where | Speed |
|---|---|---|---|
| **Logical I/O** | Read a block from **memory (buffer cache)** | RAM | Fast |
| **Physical I/O** | Read a block from **disk** and load it into memory | Disk + RAM | Slow (hundreds to thousands of times slower) |

### The flow

```
SQL request
  ↓
"Is this block in the buffer cache?"
  ├─ YES → Logical I/O (Cache Hit)   ✅ Fast
  └─ NO  → Physical I/O (Cache Miss) → Read from disk, load into buffer cache
              ↓
           Subsequent requests for this block hit the cache (Logical I/O)
```

### Buffer Cache Hit Ratio

> **BCHR = the share of logical I/Os that were satisfied without any physical I/O**
>
> ```
> BCHR = (Logical I/O - Physical I/O) / Logical I/O × 100
> ```

Higher is better, but **"BCHR 99% does not automatically mean good tuning"** — the more fundamental tuning goal is to reduce **logical I/O itself**.

### 🔑 The tuning perspective

- Most SQL slowdowns are caused by **too much logical I/O**.
- That is, **"how many blocks are being read"** is what matters most → this is decided by your index/join design.
- Physical I/O is more of a **cache efficiency** and **I/O subsystem** concern.

---

## 4. Single Block I/O vs Multiblock I/O

When reading from disk, **the number of blocks read per request** also dramatically changes speed.

| Mode | Description | When it's used |
|---|---|---|
| **Single Block I/O** | Reads **1 block** at a time | **Index scans**, **table access by ROWID** |
| **Multiblock I/O** | Reads **multiple blocks** (8–128) in one shot | **Table Full Scan**, **Index Fast Full Scan** |

### How the two actually behave

- **Single Block I/O** returns **just 1 block per request**. If you need 100 blocks, you make **100 round trips** to the disk.
- **Multiblock I/O** transfers **multiple consecutive blocks** (Oracle's default max is 128) per request. If you need 100 blocks and you read 32 per call, the round trips drop to about **4**.
- The **fixed cost per request** (system call, command issue, kernel/driver transition, disk preparation) is nearly the same regardless of how many blocks you receive. So the more blocks you receive per call, **the lower the effective per-block cost**.
- Multiblock I/O pays off best when the blocks are **physically contiguous** on disk. If the blocks are scattered across the disk, the system internally splits the request back into separate Single Block reads — the "Multiblock" was only nominal and the real benefit evaporates.

### Performance implications

- If you need to read **20–25% or more** of the entire table?
  → Rather than firing off lots of Single Block I/Os, **a Full Scan that sweeps everything via Multiblock I/O is faster.**
  → This is exactly what "the **index breakeven point**" refers to.

- A good **Clustering Factor (CF)** raises Single Block I/O efficiency and pushes the breakeven point further out.

### Vendor-specific knobs

```
-- Oracle
DB_BLOCK_SIZE                   -- block size (default 8 KB)
DB_FILE_MULTIBLOCK_READ_COUNT   -- blocks per Multiblock I/O read (default 128)

-- MySQL InnoDB
innodb_page_size                -- block (page) size (default 16 KB)
innodb_read_ahead_threshold     -- triggers async sequential read-ahead
                                -- (the equivalent of Multiblock I/O in MySQL is
                                --  "linear/random read-ahead", not a simple
                                --  blocks-per-call knob)
```

---

## 5. Sequential vs Random Access

Another important axis that decides I/O efficiency.

| Type | Description | Examples |
|---|---|---|
| **Sequential Access** | Reads **consecutive blocks** in order | Table Full Scan, Index Leaf Scan |
| **Random Access** | Reads **scattered blocks** here and there | Looking up a table block via an index ROWID |

### Still relevant in the SSD era?

> On old HDDs, random access was tens to hundreds of times slower because of **disk head seek time**.
> On SSDs/NVMe the gap has shrunk a lot, but **sequential is still faster**, and DBMS optimizers still calculate costs under that assumption.

---

## 6. 🧠 Why does this matter for tuning?

The goal of every SQL tuning effort converges on this one line.

> **"Reduce logical I/O. Especially unnecessary Single Block random access."**

Keep that in mind, and the techniques you'll learn later immediately make sense.

| Tuning technique | What it does in I/O terms |
|---|---|
| **Adding columns to an index (Covered Index)** | Eliminates random access to the table entirely |
| **Partition Pruning** | Reduces the number of blocks that need to be read at all |
| **Hash Join** | Converts random access into Full Scan (sequential) |
| **Bind Variables** | Saves hard-parse CPU cost (a different axis from I/O) |
| **Direct Path I/O** | Bypasses the buffer cache + uses Multiblock I/O |

---

## 📝 Summary

- A DBMS does I/O in **block units** (Oracle 8 KB, MySQL 16 KB)
- **Logical I/O (memory)** ≫ **Physical I/O (disk)** — hundreds to thousands of times faster
- **Single Block** (Index, ROWID) vs **Multiblock** (Full Scan) — pick the right one for the situation
- **Sequential** is still faster than **Random** (yes, even on SSDs)
- Tuning = **"reducing the number of blocks you have to read"**
