> The easiest SQL Tuning primer in the world — for junior developers.
>
> Think of this part as an **"index user manual."** The point isn't to memorize a list of rules — it's to understand them **from the underlying principle** we built up in Part 2.1.

## 1. The crux of using an index: "can we find the starting point?"

Using an index efficiently ultimately comes down to one question: can we do an **Index Range Scan**?

- **Index Range Scan (the best case):** Descend from the root, land on exactly the **starting point** you need, then stop at the end point. Very fast.
- **Index Full Scan (the fallback):** We couldn't find a starting point, so we sweep the index from beginning to end. Better than reading the whole table (Table Full Scan), but far from efficient.

> **💡 Hold on — a random-I/O warning**
> When you take a ROWID found in the index and go read the actual data file, that's a **random I/O**. So once the rows you need exceed roughly **20–25%** of the table, it can actually be *faster* to skip the index and just read the whole table.

---

## 2. Conditions that break an Index Range Scan

Even when an index exists, the following WHERE-clause shapes leave the optimizer **unable to decide a starting point**, so it can't use the index.

- **Missing leading column:** Index is on `(team, name, age)` but you search by `name` only? No starting point → full scan. **The first column of the index must appear in the WHERE clause, no exceptions.**
- **OR condition:** "Find A *or* B" makes it ambiguous where the scan should begin, so the index is hard to use. (Splitting it with `UNION ALL` can let each branch use the index.)
- **IN condition:** An `IN` is essentially several `=` conditions combined, so internally it's broken into pieces, each of which can use the index.

---

## 3. The bonus an index gives you: "skipping the sort"

An index is already stored in **sorted order**. Use that well and you get a huge win.

- **Skip ORDER BY:** If you pull data in index order, there's nothing left to sort — you save CPU.
- **MIN/MAX optimization:** Just glance at the leftmost (minimum) or rightmost (maximum) value and you're done — practically instant.

---

## 4. The "thou shalt nots" of using an index

These are extremely common mistakes — burn them into memory.

1. **Don't transform the column.** Wrapping a column in a function like `WHERE YEAR(birth_date) = 1994` kills the index, because the index stores the **original** values, not the transformed ones.
2. **Watch for implicit type conversion.** Comparing a string column to a number (`WHERE string_col = 123`) makes the database internally convert the *column* to a number, which abandons the index. **Always match the types.**
3. **Don't transform the ORDER BY column either.** If you ask the database to sort on a transformed index column, you lose the index's sort benefit.

---

## 5. More condition types to watch out for

Beyond the "thou shalt nots" above, here are a few more condition shapes that **frequently disable an index in real-world code**.

### 5.1 NULL conditions

```sql
-- index: (status)
SELECT * FROM orders WHERE status IS NULL;     -- ❌ Oracle B-Tree → index unusable
SELECT * FROM orders WHERE status IS NOT NULL; -- ⚠️ usually pushed toward a full scan
```

- **An Oracle B-Tree index does not store NULLs** → an `IS NULL` condition can't use the index.
- **MySQL InnoDB does store NULLs**, but when the NULL ratio is high the optimizer often drops the index anyway.
- Fixes:
  - Use a **default value** (`'N'`, `0`, etc.) instead of NULL.
  - If you truly must search for NULLs, consider a **function-based index** (`NVL(status, 'X')`).
  - In MySQL, designing with a **NOT NULL constraint + default value** is generally the safer bet.

### 5.2 Prefix `LIKE` vs. suffix `LIKE`

```sql
-- index: (name)
SELECT * FROM users WHERE name LIKE 'Smith%';    -- ✅ prefix match → Index Range Scan
SELECT * FROM users WHERE name LIKE '%Smith';    -- ❌ suffix match → no starting point → full scan
SELECT * FROM users WHERE name LIKE '%Smith%';   -- ❌ contains match → naturally a full scan
```

- An index is **sorted from the left**, so a Range Scan needs the **leading characters to be fixed**.
- `'%abc'` or `'%abc%'` gives no starting point, so the index is useless.
- If suffix/contains matching is a hard requirement:
  - **Reverse index** (index the *reversed* `name`) → `'%Smith'` behaves like `'Smith%'`.
  - **Full-Text Search** — MySQL's FULLTEXT index, Oracle Text.
  - **A dedicated search engine** — a separate stack like Elasticsearch.

### 5.3 Negation conditions

```sql
-- index: (status)
SELECT * FROM orders WHERE status != 'DONE';     -- ❌ hard for the index to help
SELECT * FROM orders WHERE status NOT IN (...);  -- ❌ same story
SELECT * FROM orders WHERE NOT EXISTS (...);     -- ⚠️ depends on the situation
```

- **Negation conditions** tend to match *most* of the rows, so the **optimizer prefers a full scan**.
- Example: `status != 'DONE'` includes **every status that isn't DONE** → narrowing it with an index buys nothing.
- Fixes:
  - **Rewrite as a positive condition** — spell out the remaining values, e.g. `status IN ('PENDING', 'SHIPPING')`.
  - If the set of possible values is small, **unfold them into an IN list**.

### 5.4 How a range condition affects the trailing columns

```sql
-- index: (dept, hiredate, sal)
SELECT * FROM emp
WHERE dept = 10
  AND hiredate BETWEEN '2020-01-01' AND '2020-12-31'
  AND sal >= 3000;
```

- `dept = 10` → **access condition** (equality)
- `hiredate BETWEEN ...` → **access condition** (range)
- `sal >= 3000` → **filter condition** (since `hiredate` is a range, anything after it can't help decide the scan range)

> ⚠️ **Once a range condition appears in the middle, every column after it becomes a filter condition, not an access condition.** This is exactly why, when designing a composite index, you keep the order **`=` columns → range columns**.

---

### 📝 One-line takeaway

> **"To get an Index Range Scan, put an *untransformed leading column* in the WHERE clause so the database knows exactly where to start."**

Lock in just this one principle and you're already halfway to SQL tuning.
