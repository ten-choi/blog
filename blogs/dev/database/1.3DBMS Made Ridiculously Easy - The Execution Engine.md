## 1. The execution engine's role

All the earlier stages (parser → optimizer → row source generator) only **decided "how to run"** the query. The component that actually pulls data into memory and performs operations is the **execution engine**.

The execution engine walks the **Row Source Tree** produced by the row source generator from the root downward, calling each operator (index scan, table access, join, sort, etc.) in turn and passing the result of each child operator up to its parent — ultimately producing the final result.

---

## 2. 🏗️ Three main responsibilities of the execution engine

### ① Request blocks from the storage engine

The execution engine **does not touch the disk directly**. It passes the address of the block it needs (table/index identifier + block number) to the storage engine and says, **"please put this block into the buffer cache."**

- The block is already in the buffer cache → the storage engine returns it immediately (**logical I/O**)
- The block is not in the buffer cache → the storage engine reads it from disk, loads it into the buffer cache, and then returns it (**physical I/O**)

---

### ② Interpret each block row by row

A block in the buffer cache is just a **fixed-size chunk of bytes**. The execution engine reads the block header and row directory to figure out how many rows are stored inside, where each row's starting offset is, then pulls rows out one at a time and hands them off to the **predicate evaluation and column extraction** step.

---

### ③ Perform operations (filtering, sorting, joining)

It actually carries out each operator in the Row Source Tree.

| Operation | What the execution engine does |
|---|---|
| **Filtering** | Evaluates the WHERE predicate per row and forwards only matching rows to the parent operator |
| **Sorting** | Loads the input set into PGA's Sort Area and sorts by the designated sort key. If the Sort Area is exceeded, it spills to the Temp tablespace |
| **Joining** | Matches the join keys from both inputs using whichever method was chosen — NL / Hash / Sort Merge |

---

## 3. Two memory areas the execution engine uses

The execution engine alternates between two memory areas. Their **sharing scope** and **purpose** are completely different.

| Area | Sharing Scope | Purpose |
|---|---|---|
| **Buffer Cache** | **Shared by all sessions** within the SGA | Caches data blocks read from disk |
| **PGA (Program Global Area)** | **Per-session, isolated** | Holds the Sort Area, the Hash Area for hash joins, and cursor state |

Up through predicate evaluation, **directly referencing blocks in the buffer cache** is enough. But operations that **restructure the input set** — sorting and hash joins — allocate their own space in the PGA to do the work.

### General flow of block requests and operations

```
Is the required block in the buffer cache?
  ├─ Yes  → Reference the block, extract rows → run the operator
  └─ No   → Storage engine reads from disk and loads into the buffer cache
              → Same flow: extract rows → run the operator

If a restructuring operation (sort, hash join) is required
  → Copy the extracted rows from the buffer cache into the PGA (Sort/Hash Area)
    and process them there
```

## 4. 🏁 The complete life of a SQL statement (final review)

Let's tie together everything we've covered from parts 1 through 3.

| Step | Component | What it does |
|---|---|---|
| **1** | Parser | Syntax / semantic / permission checks; look up the plan cache by hash (attempt soft parse) |
| **2** | Optimizer | On cache miss: query transformation → cost estimation → generate candidate plans → pick the lowest-cost one (hard parse) |
| **3** | Row Source Generator | Convert the selected execution plan into a Row Source Tree the execution engine can call |
| **4** | **Execution Engine** | Walk the Row Source Tree, request blocks from the storage engine, extract rows, run filtering / sorting / joining, return results |

---

## 5. ⚠️ This is where tuning matters!

> When people say **"my SQL is slow"**, most of the time is being burned right here — in the **execution engine stage**.

### The main causes of slowdown

- **Sending too many block requests to the storage engine** → excessive logical / physical I/O
- **PGA Sort Area is too small, causing spills to the Temp tablespace** → disk I/O during sort, response time jumps

### So what comes next?

This is exactly why the next topic — **I/O (Input/Output)** — matters so much.

> **Reducing the number of blocks requested from the storage engine** = the heart of tuning
