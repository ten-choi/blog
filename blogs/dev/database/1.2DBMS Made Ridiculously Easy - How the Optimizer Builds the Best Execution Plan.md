## 1. Recap: The Internal Components of a DBMS

A DBMS processes SQL through the cooperation of the following components. Excluding the first (input) and the last (storage), **the three components in the middle are the ones doing the actual "work".**

| Component | Role |
|---|---|
| Interface | Receives the user's SQL command |
| **Parser** | Checks whether the command is syntactically valid |
| **Optimizer** | Builds a strategy to retrieve the data as fast as possible |
| **Execution Engine** | Actually fetches the data according to the optimizer's strategy |
| Storage Engine | Writes data to and reads data from disk/SSD |

---

## 2. When does the optimizer kick in?

If parsing ends with a **soft parse**, **the optimizer does not run at all.** The execution plan already sitting in the plan cache is reused as-is.

The optimizer is only invoked when no plan with a matching hash value is found in the plan cache — i.e. a **hard parse**. Then, based on statistics and query conditions, it **computes a brand-new execution plan**. This computation is CPU-heavy, so in real-world SQL performance, **hard-parse frequency is a meaningful indicator**.

---

## 3. 🛠️ The Optimizer's Three-Stage Pipeline

The core work the optimizer performs during a hard parse breaks down into roughly **three stages**.

> **Order of operations**: Query transformation → Cost estimation → Plan generation

### ① Query Transformer

Rather than executing the SQL exactly as written, the optimizer **rewrites the SQL itself into a form that allows for a better execution plan**.

**Why is rewriting necessary?**
- With the user's original SQL shape, the set of execution paths the optimizer can choose from is often quite limited.
- Transforming it into a logically equivalent but structurally different SQL lets the optimizer explore **many more join orderings, join methods, and index-usage options**.

**Common transformations**
- Subquery → JOIN conversion
- Eliminating unnecessary predicates
- View merging

---

### ② Estimator

**Using statistics**, the optimizer calculates the resource cost of each candidate execution plan.

- **Cardinality**: The **estimated number of rows returned** at each step. If this estimate is wrong, subsequent decisions like join-method selection cascade into being wrong too.
- **Cost**: A single integer value that summarizes the **logical I/O + physical I/O + CPU** required to run that step. The optimizer ultimately picks the **candidate with the lowest cost**.

The cost calculation itself is grounded in **system statistics** like Single Block I/O time, Multiblock I/O time, and CPU speed — it is a quantitative computation.

**Data the estimator consults**
- Row counts of tables
- Block counts
- Column value distributions (histograms)
- Index presence

---

### ③ Plan Generator

By combining the many possible options, the optimizer **finalizes the optimal path**.

| Decision | Options |
|---|---|
| **Join order** | Read table A first? Or table B first? |
| **Access path** | Index scan? Or full scan? |
| **Join method** | NL Join / Hash Join / Merge Join |

---

### ④ Row Source Generator

The execution plan the optimizer chooses is still **an abstract tree of operations**. The Row Source Generator converts that operation tree into a **set of operators the execution engine can invoke directly (Row Source Tree)**.

- Each node is expressed as a **concrete operator** such as `TABLE ACCESS BY INDEX ROWID`, `INDEX RANGE SCAN`, `HASH JOIN`, or `SORT`.
- The `Operation`, `Object Name`, and `Options` columns you see in `EXPLAIN PLAN` output are exactly what comes out of this stage.
- The execution engine walks this tree starting from the root, calling each operator and passing its result up to the parent operator.

---

## 4. 🧐 Why is it said to "burn a lot of CPU"?

Because the set of execution-plan candidates the optimizer has to evaluate **grows exponentially**.

### Example: a 5-table join

- Number of orderings for reading the tables: **5! = 120**
- Multiply that by index-use × join-method combinations per table
- → The candidate set quickly explodes into **thousands or tens of thousands**

The **cost calculation logic runs repeatedly** for every one of these candidates, which is why a single hard parse's CPU consumption is typically **tens to hundreds of times higher** than a normal SELECT execution.

---

## 5. 💡 Summary: so what does the optimizer actually produce?

In the end, the optimizer does only one thing.

> **Pick the Execution Plan with the lowest cost.**

The selected plan is then handed off to the Row Source Generator, which turns it into an **operator tree** that the execution engine can call.

---

## 6. 📝 Side note: where does the Row Source Generator live?

When you run a SQL statement, the `Operation`, `Object Name`, `Options`, and so on that appear in the execution plan output are **exactly the artifacts the Row Source Generator produces**.

Strictly speaking, the Row Source Generator is not part of the "optimizer", but **since soft parses also reuse information produced by this stage**, it's easier to bundle it together with the parsing/optimization flow when learning.
