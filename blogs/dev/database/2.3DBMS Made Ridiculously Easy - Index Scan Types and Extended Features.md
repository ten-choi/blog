## 1. 🚀 DB index scan methods at a glance

How the database walks the index makes an enormous difference in query speed. Let's see which method shows up in which situation.

### Bread-and-butter scans

- **Index Range Scan (the standard)**
  - The most common method. Descend vertically from root to leaf, then sweep up **only the range** you need.
  - **Watch out:** transforming the leading column (`WHERE SUBSTR(name, 1, 1) = ...` and the like) prevents this method.
- **Index Full Scan (the fallback)**
  - No Vertical Traversal — it reads the **leaf blocks from start to finish**. Usually chosen when there's no suitable index, on the logic "at least it beats a table full scan."
- **Index Unique Scan (lightning fast)**
  - Happens when you look up a PK or unique index with an `=` condition. Since "exactly one" match ends the search, there's no scan range at all — top performance.

### Scans that squeeze out extra efficiency

- **Index Skip Scan (the jump scan)**
  - Lets you use an index *even without a leading-column condition*. When the leading column has few distinct values (low cardinality) and the trailing column has many, it **skips through, reading only the leaf blocks that can match**.
- **Index Fast Full Scan (speed first)**
  - Much faster than an Index Full Scan. It ignores the index's logical order and reads everything **physically, all at once (Multiblock I/O)**. The catch: **the result is not sorted.**
- **Index Range Scan Descending (reverse scan)**
  - Identical to a Range Scan but reads **from the back**. With a condition like `ORDER BY ... DESC`, you get the result directly without a separate sort.

## 2. 📊 MySQL execution-plan checkpoints

To confirm a query runs the way you intended, always open up `EXPLAIN`. These four items alone usually tell the story.

| Field | Meaning | Checkpoint ✅ |
| :--- | :--- | :--- |
| **type** | Join / scan method | Better in the order `system` > `const` > `range`; avoid `ALL` (full scan) |
| **key** | The index actually used | Confirm the index you intended is the one in play |
| **rows** | Estimated rows examined | If this number is huge, the index isn't really doing its job |
| **Extra** | Extra info | `Using filesort` or `Using temporary` is a performance red flag 🚨 |

---

### 💡 Tip in one line

Using an index doesn't automatically make a query fast. The real skill is **"how little you scan, and how seldom you touch the table."** The key is to write queries so the execution plan's `type` lands on `range` or `const`.

## 3. 🧩 The other "extended index" features

Everything above was about **"how you scan."** From here, it's about **"what kind of index you can even build."**

### ① Function-Based Index

A technique that makes the **result of transforming a column** the indexed value itself.

```sql
-- ❌ a plain index can't be used here
SELECT * FROM users WHERE UPPER(email) = 'FOO@BAR.COM';

-- ✅ create a function-based index
CREATE INDEX idx_users_upper_email ON users (UPPER(email));

-- now the optimizer can use the index
SELECT * FROM users WHERE UPPER(email) = 'FOO@BAR.COM';
```

- **When to use:** for **function results you frequently use in the WHERE clause** — `UPPER`, `LOWER`, `SUBSTR`, date formatting, and so on.
- **Watch out:** because the indexed value is the **"function result"** rather than a real stored column, the function is evaluated on every DML, so **its DML cost is higher than a plain index**.
- **MySQL:** supports the `CREATE INDEX ... ((UPPER(email)))` syntax (8.0.13+), or the **generated column + index** combo.

### ② Invisible Index

A feature that **hides an index from the optimizer only**. The index physically exists, but query optimization ignores it.

```sql
-- Oracle
ALTER INDEX idx_foo INVISIBLE;
ALTER INDEX idx_foo VISIBLE;

-- MySQL 8.0+
ALTER TABLE users ALTER INDEX idx_foo INVISIBLE;
ALTER TABLE users ALTER INDEX idx_foo VISIBLE;
```

- **When to use:**
  - **Before dropping an index, to test "do we really not need this?"** — flip it to invisible, monitor, and if nothing breaks, DROP it.
  - **When you want it used only in specific sessions** — a session parameter can make invisible indexes visible again.
- **Upside:** instant rollback. Dropping and rebuilding an index is a very expensive operation on a large table.

### ③ Descending Index

A feature that **stores the index in DESC order**.

```sql
-- store a column that's queried newest-first directly in DESC order
CREATE INDEX idx_logs_created_desc ON logs (created_at DESC);
```

- **MySQL:** real DESC indexes since 8.0. Before that, the DESC keyword was ignored and the index was built ASC anyway.
- **Oracle:** supported for a long time.
- **When it helps:** when the **`ORDER BY ... DESC` + `LIMIT`** pattern is frequent. Since it's already stored DESC, even a reverse scan is unnecessary.

### ④ Partial / Filtered Index

A feature that **indexes only some rows** of a table. It shrinks index size and lowers maintenance cost.

```sql
-- PostgreSQL / SQLite
CREATE INDEX idx_active_users ON users (email) WHERE status = 'ACTIVE';

-- SQL Server (Filtered Index)
CREATE INDEX idx_active_users ON users (email) WHERE status = 'ACTIVE';
```

- **When to use:**
  - 100 million rows in the table, but only **1 million are "active users,"** and queries almost always look for active users only.
  - → the index shrinks to **1/100 the size** and the DML burden drops accordingly.
- **MySQL:** no direct support, but you can work around it with a **generated column + index**.
- **Oracle:** no official partial index, but a **function-based index gives a similar effect** (`NVL(CASE WHEN status='ACTIVE' THEN email END, ...)`).

### ⑤ Multi-Value Index (MySQL 8.0.17+)

A feature that indexes the values *inside* a JSON array.

```sql
CREATE INDEX idx_tags ON articles ( (CAST(tags AS CHAR(20) ARRAY)) );

-- usage
SELECT * FROM articles WHERE 'javascript' MEMBER OF (tags);
```

- Useful for modern applications that frequently search JSON fields.

### ⑥ Spatial Index — R-Tree

An index specialized for coordinate / region search.

```sql
CREATE SPATIAL INDEX idx_location ON places (geom);
SELECT * FROM places WHERE ST_Contains(ST_GeomFromText('POLYGON(...)'), geom);
```

- A different structure from a B-Tree — an **R-Tree**.
- Essential for location-based services and GIS.
