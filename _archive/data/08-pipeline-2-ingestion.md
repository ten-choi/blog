**데이터 플랫폼 시리즈 · 08/15** · Moving Data(데이터 이동) 2/5

> 1부에서는 파이프라인의 큰 그림, 즉 extract(추출), transform(변환), load(적재)를 살펴봤습니다. 이번 편에서는 그 첫 단계인 **data ingestion(데이터 수집)**, 곧 데이터를 소스에서 *꺼내어* 플랫폼 안으로 들여오는 작업을 자세히 들여다봅니다. 간단해 보이지만, 전혀 그렇지 않습니다.

## 1. 수집은 정문입니다

모든 파이프라인은 같은 문제에서 출발합니다. 우리가 원하는 데이터는 누군가 다른 사람의 시스템 안에 있습니다. 우리 앱의 프로덕션 데이터베이스, 서드파티 API, 사용자 이벤트 스트림, 서버에 쌓인 파일 더미 같은 곳이죠. data ingestion(데이터 수집)은 그 데이터를 모아 우리 데이터 플랫폼으로 들여오는 과정입니다.

> **data ingestion(데이터 수집) = 데이터를 그 소스에서 추출하여 저장 또는 처리 시스템으로 받아들이는 일.**

이것은 플랫폼 전체의 정문입니다. 그리고 정문이 그렇듯, 이 부분이 불안정하면 그 뒤에 있는 모든 것이 영향을 받습니다. 애초에 도착하지 않았거나 잘못 도착한 데이터는 아무리 영리한 다운스트림(downstream) 처리로도 고칠 수 없습니다.

함정은 여기 있습니다. 수집은 "그냥 데이터를 읽으면 되지"처럼 *간단하게 느껴지지만*, 소스는 수십 가지 방식으로 비협조적입니다. 다운되기도 하고, 속도 제한(rate limit)을 걸기도 하며, 예고 없이 포맷을 바꾸기도 합니다. 양이 어마어마하기도 하고, 라이브 상태라 속도를 떨어뜨리면 안 되기도 합니다. 좋은 수집이란 대부분 이런 현실을 다루는 일입니다.

## 2. 다시 등장하는 두 가지 리듬: batch와 streaming 수집

1부에서 다룬 batch(배치) 대 streaming(스트리밍) 구분은 수집 단계에서 가장 먼저 모습을 드러냅니다.

- **Batch ingestion(배치 수집)** — 일정에 맞춰 데이터를 덩어리 단위로 가져옵니다. "매일 밤 2시에 오늘의 주문을 전부 가져와라" 같은 식입니다. 단순하고 효율적이며, 데이터가 분 단위로 신선할 필요가 없을 때 충분합니다.
- **Streaming ingestion(스트리밍 수집)** — 데이터가 생성되는 즉시 각 이벤트를 연속적으로 캡처합니다. "주문이 발생하는 그 순간 수집하라"는 식입니다. 신선도가 중요한 경우(실시간 대시보드, 사기 탐지, 실시간 개인화)에 필요합니다. 보통 Kafka(카프카) 같은 메시지 스트림 위에서 동작합니다.

대부분의 수집은 배치입니다. 대부분의 분석이 몇 시간의 지연을 견딜 수 있고, 배치가 운영하기 훨씬 단순하기 때문입니다. 더 신선한 데이터가 주는 *비즈니스 가치*가 추가되는 복잡성을 분명히 정당화할 때 스트리밍을 선택하세요. 이 주제는 5부에서 다시 다룹니다.

## 3. 핵심 질문: full load인가, incremental인가?

이제 수집 작업의 성격을 결정짓는 의사결정이자, 초심자가 가장 자주 틀리는 지점을 살펴봅니다. 한 소스에서 반복적으로 수집할 때, 매번 *전부*를 가져올 것인가, 아니면 *새로 생긴 것*만 가져올 것인가?

### Full load(전체 적재)

매 실행마다 전체 데이터셋을 가져옵니다.

- **장점:** 더없이 단순합니다. 목적지는 항상 완전하고 정확한 사본입니다.
- **단점:** 데이터가 커지면 낭비가 심하고 느립니다. 수천 건의 변경을 잡으려고 매일 밤 5억 행짜리 테이블을 다시 복사하는 것은 터무니없는 일입니다.

전체 적재는 작은 테이블이나, 데이터를 처음 채워 넣는 "seed(시드)" 단계에 적합합니다.

### Incremental load(증분 적재)

지난번 이후로 변경된 데이터만 가져옵니다.

- **장점:** 효율적입니다. 전체가 아니라 적은 양의 데이터만 옮깁니다. 규모가 커지면 필수입니다.
- **단점:** 더 어렵습니다. *무엇이* 변경되었는지 신뢰성 있게 알아내는 방법이 필요합니다. 보통 `updated_at` 같은 컬럼을 기준으로 필터링하는 방식("지난 실행 이후 변경된 행을 달라")을 쓰는데, 이는 소스가 그 컬럼을 정직하게 유지해 준다는 전제를 요구합니다.

실무에서 흔히 쓰는 방식은 마지막 실행 시각을 **watermark(워터마크)**로 저장해 두고, 다음 실행에서 그 이후의 행만 긁어오는 것입니다.

```sql
-- 직전 실행 시각(watermark)을 매개변수로 주입
SELECT * FROM orders
WHERE updated_at > '{{ last_run_ts }}'
```

```
Full load:        [■■■■■■■■■■]  copy everything, every time
Incremental:      [········▣▣]  copy only what's new since last run
```

깔끔해 보이지만, 여기에 현업 함정이 숨어 있습니다.

- **늦게 도착하거나 나중에 수정된 행** — 트랜잭션이 늦게 커밋되거나 과거 레코드가 수정되면, 워터마크 경계를 살짝 넘어 누락될 수 있습니다.
- **clock skew(시계 오차)** — 소스 DB와 수집 작업의 시계가 미세하게 어긋나면 경계 부근의 행을 통째로 놓치기도 합니다.
- 그래서 실무에서는 `updated_at > '{{ last_run_ts }}' - INTERVAL '5 minutes'`처럼 **약간의 overlap window(중첩 구간)**를 두고 다시 긁어옵니다. 대신 같은 행이 두 번 들어올 수 있으니, 멱등(idempotent) 적재가 전제되어야 합니다(8절 체크리스트 참고).
- **삭제는 잡히지 않습니다** — 삭제된 행에는 "갱신" 타임스탬프가 남지 않으므로 타임스탬프 기반 증분은 delete를 영영 보지 못합니다. 이것이 바로 다음 절의 CDC가 필요한 이유입니다.

경험칙은 이렇습니다. **작거나 정적인 데이터에는 전체 적재로 시작하고, 양이 늘어나 전체 적재가 부담스러워지는 순간 증분 적재로 옮겨가라.** 더 구체적인 결정 규칙은 이렇습니다.

> **결정 규칙:** 한 번에 다 복사해도 부담이 없을 만큼 작으면 → full load. 데이터가 커서 매번 다 복사하기 아깝고 신뢰할 만한 `updated_at`이 있으면 → 증분 적재. 삭제까지 정확히 반영해야 하거나 거의 실시간이 필요하면 → CDC(다음 절).

증분 적재는 실무 데이터 엔지니어링을 규정하는 핵심 기술 중 하나입니다.

## 4. Change Data Capture(CDC): 증분의 정석

`updated_at` 컬럼을 이용한 증분 적재에는 빈틈이 있습니다. 삭제(delete)를 놓칠 수 있고(삭제된 행에는 "갱신" 타임스탬프가 없습니다), 소스가 그 컬럼을 완벽하게 유지한다는 데 의존합니다. 더 견고한 기법이 바로 **Change Data Capture(변경 데이터 캡처, CDC)**입니다.

CDC는 데이터베이스 자체의 **transaction log(트랜잭션 로그)**를 읽는 방식으로 동작합니다. 트랜잭션 로그란 모든 데이터베이스가 모든 insert, update, delete를 기록해 두는 내부 장부입니다. CDC는 데이터베이스에게 "무엇이 바뀌었나?"라고 *묻는* 대신, 로그를 *지켜보며* 모든 변경을 발생 순서대로 캡처합니다.

```
  source DB writes  ──►  transaction log  ──►  CDC reads the log  ──►  your platform
  (insert/update/delete)  (DB's own record)     (captures every change)
```

강력한 이유는 다음과 같습니다.

- **모든 것을 잡아냅니다** — 타임스탬프 기반 방식이 놓치는 삭제까지 포함됩니다.
- **거의 실시간입니다** — 변경이 커밋되는 대로 흘러들어와 스트리밍 수집을 가능하게 합니다.
- **소스에 미치는 영향이 적습니다** — 로그를 읽는 것은 라이브 데이터베이스에 쿼리를 퍼붓지 않습니다.

실무에서 가장 자주 만나는 도구는 **Debezium(디비지움)**입니다. 데이터베이스의 트랜잭션 로그를 읽어 변경 이벤트를 Kafka(카프카)로 흘려보내며, 보통 **Kafka Connect** 위에서 커넥터 형태로 운영합니다. 다만 이 스택을 직접 구축·운영하는 부담이 만만치 않아서, 많은 팀은 **Fivetran(파이브트랜)**이나 **Airbyte(에어바이트)** 같은 관리형 커넥터를 *사서* 빠르게 시작하기도 합니다. 어느 쪽이든 핵심 개념은 동일합니다 — 로그를 읽어 변경을 그대로 흘려보낸다.

실무에서 알아 둘 포인트가 두 가지 있습니다. 첫째, CDC는 보통 **2단계로 시작합니다.** 트랜잭션 로그에는 최근 변경만 남아 있으므로, 먼저 기존 데이터 전체를 한 번 복사하는 **initial snapshot(초기 스냅샷)**을 뜨고, 그 시점 이후의 로그를 이어서 스트리밍합니다. 둘째, **소스의 스키마 변경이 변경 이벤트에도 그대로 흘러옵니다.** 소스 테이블에 컬럼이 추가되면 어느 순간부터 이벤트의 모양이 달라지므로, 하류가 새 컬럼을 만나도 조용히 깨지지 않도록 설계해 두어야 합니다(7절 체크리스트의 "스키마 변화 대비"가 CDC에서는 선택이 아니라 필수입니다).

CDC는 현대 플랫폼이 운영 데이터베이스의 분석용 사본을 끊김 없이 동기화 상태로 유지하는 방식입니다. 야간 쿼리 한 번보다 설정은 복잡하지만, 데이터 웨어하우스나 레이크를 라이브 데이터베이스와 거울처럼 맞춰 두는 데에는 이것이 정석적인 답입니다. 끊임없이 등장하는 용어이니 꼭 알아두세요.

## 5. 다양한 소스들

"수집"은 폭넓은 소스 유형을 아우르며, 각 유형마다 고유한 특이점이 있습니다.

| 소스 유형 | 수집 방식 | 주의할 점 |
|-------------|----------------|---------------|
| **데이터베이스** | 쿼리, 또는 로그에서 CDC | 라이브 DB에 과부하를 주지 말 것 |
| **API** | HTTP 요청, 페이지네이션 | rate limit, 인증, 페이지네이션 |
| **파일** | 오브젝트 스토리지 / FTP에서 읽기 | 포맷 변화, 부분 업로드 |
| **이벤트 스트림** | 스트림 구독 | 순서, 중복, backpressure |

반복되는 주제가 하나 있습니다. 우리는 소스를 *통제하는* 경우가 드뭅니다. 소스는 자기 나름의 규칙, 신뢰성, 돌발 상황을 가진 남의 시스템입니다. 견고한 수집은 소스가 오작동할 것이라고 가정합니다. 즉 일시적으로 다운되고, 잘못된 형식의 레코드를 반환하며, 스키마(schema)를 바꿀 것이라고 전제하고, 정상 경로(happy path)만 믿는 대신 그런 경우들을 처리합니다.

## 6. 원본 데이터 착지시키기: 먼저 수집하고, 변환은 나중에

체득해 둘 가치가 있는 모범 사례가 있습니다. 수집할 때는 **변환하기 전에, 도착한 그대로의 원본 데이터를 먼저 착지(land)시켜라**는 것입니다. 이것은 데이터 레이크 시리즈의 "raw zone(원본 영역)"과 곧바로 연결됩니다.

왜일까요?

- 변환에 버그가 있어도 원본이 그대로 있으니 다시 처리할 수 있습니다. 잃은 것이 없습니다.
- 수집과 변환이 독립적인 단계가 되어, 각각 따로 실패하고 따로 재시도할 수 있습니다.
- 소스가 실제로 무엇을 보냈는지에 대한 충실한 이력 기록을 얻습니다.

이 "원본을 먼저 착지시키고 나중에 변환한다"는 본능은, 현대 플랫폼에서 **ELT**가 ETL을 밀어낸 이유와도 정확히 맞닿아 있습니다. 그리고 그것이 바로 3부의 주제입니다.

## 7. 수집 체크리스트

새 수집 작업을 만들 때 머릿속에 넣어 둘 짧은 점검표입니다.

- [ ] **원본 먼저 적재** — 변환하기 전에 도착한 그대로를 raw로 착지시켰는가?
- [ ] **멱등 적재(idempotent)** — 같은 데이터를 두 번 수집해도 중복이나 손상이 생기지 않는가? (overlap window·재시도를 고려하면 필수)
- [ ] **스키마 변화 대비** — 소스가 컬럼을 추가·삭제·변경해도 파이프라인이 조용히 깨지지 않는가?
- [ ] **소스 과부하 방지** — 라이브 DB·API에 부하를 주지 않는가? (rate limit 준수, 읽기 전용 사본 또는 로그 기반 CDC 활용)

## 8. 핵심 정리

수집은 데이터 플랫폼의 정문입니다. 비협조적인 소스에서 데이터를 꺼내 우리 시스템으로, 안정적으로 들여오는 일이죠. 이를 규정하는 결정은 **batch 대 streaming**(얼마나 신선하게?)과 **full 대 incremental**(매번 얼마나?)이며, 라이브 데이터베이스와 분석용 사본을 끊김 없이 동기화 상태로 유지하는 견고한 방법으로 **Change Data Capture**가 있습니다. 그리고 황금률은 **원본 데이터를 먼저 착지시킨 뒤** 변환하는 것입니다.

다음은 **3부: ETL 대 ELT**입니다. 데이터가 들어왔으니, 이제 *언제* 변환해야 할까요? 적재하기 전일까요, 후일까요? 지난 10년 사이 이 답은 뒤집혔고, 그 이유를 이해하면 현대 데이터 플랫폼이 어떻게 만들어지는지에 대해 많은 것이 설명됩니다.
