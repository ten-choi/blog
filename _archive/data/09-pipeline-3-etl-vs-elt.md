**데이터 플랫폼 시리즈 · 09/15** · Moving Data(데이터 이동) 3/5

> 데이터 분야에서 가장 자주 입에 오르내리는 두 약어, 그 차이는 정확히 글자 하나의 위치가 바뀐 것뿐입니다. 이번 편에서는 ETL과 ELT가 무엇인지, 업계가 왜 한쪽에서 다른 한쪽으로 대거 옮겨갔는지, 그리고 어떻게 선택해야 하는지를 설명합니다.

## 1. 똑같은 세 단계, 다른 순서

1부에서 다룬 파이프라인의 뼈대를 떠올려 봅시다. **Extract, Transform, Load.** ETL과 ELT는 바로 이 세 단계로 만들어집니다. *유일한* 차이는 **Transform이 Load를 기준으로 어디에 위치하느냐**입니다. 그리고 이 한 번의 자리 바꿈이 아키텍처 전체를 바꿔 놓습니다.

```
ETL:   Extract ──► Transform ──► Load        (transform BEFORE loading)
                   ▲
                   on a separate processing system

ELT:   Extract ──► Load ──► Transform        (transform AFTER loading)
                            ▲
                            inside the destination itself
```

- **ETL (Extract → Transform → Load):** 데이터를 꺼낸 뒤, *별도의* 처리 시스템에서 변환하고, 완성된 깨끗한 결과만 목적지에 적재합니다.
- **ELT (Extract → Load → Transform):** 데이터를 꺼낸 뒤, *원본 그대로* 목적지에 곧바로 적재하고, 목적지 자체의 compute(컴퓨트/연산)를 사용해 *거기서* 변환합니다.

말 그대로 읽어 보면 이렇습니다. ETL은 별도의 주방에서 요리를 완성해 식탁으로 가져오는 방식이고, ELT는 식재료를 식탁으로 가져와 먹을 자리에서 요리하는 방식입니다.

---

## 2. ETL: 전통적인 방식, 그리고 그게 합리적이었던 이유

ETL이 먼저 나왔고, 거기에는 그럴 만한 역사적 이유가 있었습니다. 수십 년 전, data warehouse(데이터 웨어하우스)는 compute가 **비싸고 제한적**이었습니다. 그 귀한 처리 능력을 데이터를 정제하고 재구성하는 무거운 작업에 낭비하고 싶지 *않았던* 것이죠. 그래서 데이터를 웨어하우스 *바깥*의, 전용 ETL 서버에서 변환한 뒤, 다듬어진 최종 결과만 적재했습니다.

```
sources ──► [ ETL server: clean, join, aggregate ] ──► warehouse (only clean data lands)
```

여기에는 실질적인 장점이 있었고, 그중 일부는 지금도 유효합니다.

- **웨어하우스에는 깨끗하고 모델링된 데이터만 담깁니다** — 깔끔하고 storage(스토리지) 효율도 좋습니다.
- **민감한 데이터를 적재되기 전에 걸러내거나 마스킹할 수 있습니다** — 프라이버시 측면에서 유리합니다(뒤의 데이터 거버넌스 시리즈에서 자세히 다룹니다).
- **변환 부하를 그 귀한 웨어하우스에서 떼어 놓을 수 있습니다.**

그리고 실질적인 단점도 있었습니다.

- **유연하지 않습니다** — 저장하기 *전에* 데이터를 미리 정해진 하나의 형태로 변환합니다. 원본 데이터가 필요한 새로운 질문을 던지고 싶다고요? 이미 사라졌습니다. 변환된 버전만 남겨 두었으니까요.
- **별도의 ETL 시스템**은 구축하고, 확장하고, 유지보수해야 할 또 하나의 대상입니다.

## 3. ELT: 무엇이 바뀌었고, 왜 대세가 되었나

그러다 지형이 바뀌었습니다. 현대의 클라우드 웨어하우스와 lakehouse(레이크하우스)는 **막대하게 강력해졌고 확장 비용도 저렴**해졌습니다. 스토리지는 값이 싸졌고(이것이 데이터 레이크 시리즈의 대전제죠), 그러자 "웨어하우스 compute를 낭비하지 말라"는 옛 제약이 갑자기 증발했습니다. 이제 목적지가 별도 서버보다 빠르게, 대규모로 변환을 *스스로* 처리할 수 있게 된 것입니다.

그래서 논리가 뒤집혔습니다. **원본 데이터를 먼저 적재하고, 필요할 때 목적지 안에서 변환한다.**

```
sources ──► warehouse/lake (raw data lands first) ──► transform inside it, as needed
```

이 방식이 대부분의 현대 스택에서 승리한 이유는 다음과 같습니다.

- **유연성** — 원본 데이터를 그대로 보관하므로, 나중에 새로운 질문이 생기면 다르게 변환할 수 있습니다. 버려지는 게 없습니다. (2부에서 다룬 "원본을 먼저 적재하라"는 습관을 아키텍처 수준으로 끌어올린 것입니다.)
- **빠른 적재** — 데이터를 즉시 사용할 수 있고, 변환은 나중에 합니다.
- **저렴하고 확장 가능한 목적지 compute 활용** — 별도로 굴려야 할 ETL 서버 군단이 없습니다.
- **레이크/레이크하우스 모델과 완벽하게 들어맞습니다** — raw 영역에서 시작해, 변환을 거쳐 정제(cleansed) 영역과 큐레이션(curated) 영역을 쌓아 올립니다.

대신 받아들여야 할 트레이드오프도 있습니다.

- **원본(잠재적으로 민감한) 데이터가 변환 전에 목적지에 적재됩니다** — 따라서 목적지의 접근 제어와 거버넌스가 더 중요해집니다.
- **스토리지 비용이 더 듭니다** — 원본 *과* 변환된 데이터를 모두 보관하기 때문입니다. 다만 저렴한 스토리지 덕분에 충분히 감수할 만한 거래입니다.

### 실무에서 ELT는 "EL + dbt"로 굳어졌습니다

추상적인 ELT가 현업에서 실제로 어떤 모습인지 구체적으로 말하면, 오늘날 새로 시작하는 대부분의 팀은 **EL은 관리형 커넥터, T는 dbt**로 나눠 구성합니다.

- **EL (Extract + Load):** Fivetran이나 Airbyte 같은 managed connector(관리형 커넥터)가 소스에서 데이터를 꺼내 Snowflake/BigQuery 같은 웨어하우스에 *원본 그대로* 떨어뜨립니다. 직접 추출 스크립트를 짜고 유지보수하는 부담이 사라집니다.
- **T (Transform):** 그 위에서 dbt가 변환을 담당합니다. 핵심은 **변환 = 웨어하우스 안에서 돌아가는 SQL일 뿐**이라는 점입니다. 별도 처리 엔진 없이, 웨어하우스의 compute로 SQL을 실행해 모델을 쌓아 올립니다.

dbt에서 model(모델) 하나는 결국 `SELECT` 한 덩어리입니다.

```sql
-- models/staging/stg_orders.sql
select
    order_id,
    customer_id,
    cast(amount as numeric)        as amount,
    cast(created_at as timestamp)  as created_at
from {{ source('raw', 'orders') }}
where amount is not null
```

이런 SQL 모델들이 **버전 관리(Git)** 되고, **테스트**(not null, unique 등)가 붙으며, 의존성에 따라 순서대로 실행됩니다. 추출/적재는 커넥터에, 변환은 SQL과 dbt에 맡기는 이 분업이 지금의 사실상 표준 스택입니다.

> **현업 함정 ①** — ELT에서는 원본(잠재적으로 PII가 섞인) 데이터가 *변환 전에* 목적지에 그대로 내려앉습니다. 즉 마스킹이나 필터링으로 거를 기회가 적재 이후로 미뤄지는 셈입니다. 그래서 ETL 시절보다 **목적지의 접근 제어와 마스킹이 훨씬 더 중요**해집니다 — raw 스키마는 권한을 좁히고, 민감 컬럼은 마스킹 정책을 거는 식으로요(데이터 거버넌스 시리즈와 직접 연결되는 지점입니다).
>
> **현업 함정 ②** — ELT 비용은 스토리지가 아니라 *변환 쿼리의 compute*에서 나옵니다. 스토리지는 싸지만, 매번 전체 데이터를 다시 훑는 무분별한 full-refresh는 의외로 비쌉니다. 자주 도는 큰 모델은 **incremental(증분) 모델**로 바꿔 새/변경된 행만 처리하는 것을 고려하세요.

## 4. 나란히 비교하기

| | ETL | ELT |
|---|-----|-----|
| **변환 시점** | 적재 전, 별도 시스템에서 | 적재 후, 목적지 안에서 |
| **목적지에 담기는 것** | 깨끗하게 변환된 데이터만 | 원본 데이터 + 변환 결과 |
| **유연성** | 낮음 (형태를 미리 결정) | 높음 (언제든 원본을 다시 변환) |
| **추동 요인** | 부족하고 비싼 compute | 저렴하고 확장 가능한 클라우드 compute & 스토리지 |
| **가장 잘 맞는 경우** | 적재 전 엄격한 정제/프라이버시 요구 | 현대 클라우드 웨어하우스, 레이크, 레이크하우스 |
| **시대** | 전통적인 기본값 | 현대의 기본값 |

## 5. 그래서 무엇을 써야 할까요?

오늘날 대부분의 새 플랫폼에서는 **ELT가 합리적인 기본값**입니다. 저렴한 클라우드 스토리지·compute와 잘 맞고, 미래의 질문을 위해 원본 데이터를 그대로 남겨 두며, 대부분의 팀이 그 위에 구축하고 있는 레이크/레이크하우스 아키텍처와도 부합합니다. 새로 시작한다면 ELT로 — 구체적으로는 **EL(Fivetran/Airbyte) + dbt** 조합으로 시작하세요.

**ETL이 여전히 제 몫을 하는** 경우는 다음과 같습니다.

- 원본 민감 데이터가 목적지에 적재되어서는 **안 되는** 경우(엄격한 프라이버시나 컴플라이언스 — 적재 *전에* 변환하거나 마스킹합니다).
- 목적지가 실제로 변환 부하를 감당하지 못하는 경우(지금은 드물지만, 일부 on-prem 환경이나 제약이 큰 시스템에서는 현실적인 문제입니다).
- 데이터가 적재되기 전에 완벽하게 모델링되어야 하는, 고전적이고 정형화된 웨어하우스에 데이터를 공급하는 경우.

선택에 앞서 빠르게 점검할 체크리스트입니다.

- 원본 PII가 목적지에 *내려앉아도* 접근 제어/마스킹으로 통제할 수 있는가? → 가능하면 ELT.
- 변환을 SQL로 표현할 수 있고, 웨어하우스가 그 compute를 감당하는가? → 그렇다면 ELT(+dbt).
- 적재 *전에* 반드시 걸러야 할 컴플라이언스 제약이 있는가? → 그렇다면 ETL을 남겨 두세요.

> 이것을 종교처럼 떠받들지는 마세요. 더 깊은 핵심은, 선택이 *compute와 스토리지의 경제성*을 따라간다는 점입니다. compute가 부족했을 때는 ETL이 합리적이었고, 그것이 저렴하고 탄력적으로 변하자 ELT가 합리적이 되었습니다. 정답은 여러분의 제약에 맞는 쪽이며, 오늘날 대부분의 사람에게는 그 제약이 ELT 쪽으로 기웁니다.

## 6. 핵심 정리

ETL과 ELT는 똑같은 세 단계 — Extract, Transform, Load — 이며, Transform을 Load의 앞으로 두느냐 뒤로 두느냐의 차이일 뿐입니다. ETL은 목적지 *바깥*에서 변환해 깨끗한 데이터만 적재하고(비싼 compute에서 태어난 전통적 방식), ELT는 원본 데이터를 *먼저* 적재한 뒤 저렴하고 강력한 현대의 목적지 *안*에서 변환합니다(오늘날의 기본값이자 레이크와 레이크하우스에 자연스럽게 들어맞는 방식). 실무에서 ELT는 보통 **EL은 Fivetran/Airbyte 같은 커넥터로, T는 Snowflake/BigQuery 안에서 도는 dbt SQL로** 구현됩니다.

두 방식 모두 변환을 *한다는* 사실 자체를 설명할 뿐입니다. 다음 **4부 — data integration(데이터 통합)** 에서는 변환의 가장 어려운 부분, 즉 지저분한 여러 소스를 하나의 일관된 뷰로 결합하는 과제를 다룹니다. (데이터가 최종적으로 가져야 할 *형태*를 설계하는 데이터 모델링은 저장과 아키텍처 트랙의 data warehouse(데이터 웨어하우스) 1·2부에서 다룹니다.)
