**데이터 플랫폼 시리즈 · 11/15** · Moving Data(데이터 이동) 5/5

> 시리즈의 마지막 편이자, 앞선 모든 내용을 가로지르는 축입니다. 모든 파이프라인은 두 가지 리듬 중 하나로 동작합니다. 데이터를 정해진 단위로 묶어서 처리하거나, 각 이벤트가 도착하는 즉시 처리하거나입니다. 이것이 batch(배치) 대 streaming(스트리밍)의 선택이며, 이 선택을 제대로(또는 잘못) 하는지에 따라 비용, 복잡도, 그리고 데이터의 신선도가 결정됩니다.

## 1. 두 가지 리듬

이 구분은 1부부터 언급해 왔는데, 이제 제대로 다뤄 보겠습니다. 질문은 단순합니다. **여러분의 파이프라인은 데이터를 주기적인 단위로 처리합니까, 아니면 도착하는 대로 연속적으로 처리합니까?**

- **batch processing(배치 처리)** — 일정 기간 동안 데이터를 모았다가, 정해진 일정에 따라 그룹 전체를 한 번에 처리합니다. "매일 밤 2시에 그날의 주문을 처리한다"가 그 예입니다.
- **stream processing(스트림 처리)** — 각 이벤트를 생성되는 그 순간에 개별적으로 처리합니다. "주문이 발생하는 즉시 처리한다"가 그 예입니다.

```
BATCH:    [····collect····] ──► process chunk ──► (wait) ──► [····collect····] ──► process
          runs on a schedule; data is hours/a day old

STREAM:   event─►process  event─►process  event─►process  event─►process
          runs continuously; data is seconds old
```

어느 한쪽이 "더 낫다"고 할 수는 없습니다. 둘은 서로 다른 요구에 맞춰 최적화되어 있으며, 실제 대부분의 플랫폼은 **둘 다** 사용합니다. 분석의 대부분은 배치로, 정말로 신선해야 하는 몇 가지는 스트리밍으로 처리하는 식입니다.

## 2. 배치: 든든한 일꾼

배치는 대부분의 데이터 처리가 늘 작동해 온 방식이며, 지금도 대다수 분석 업무의 기본값으로 적절합니다. 야간 리포트, 일일 웨어하우스 적재, 월별 청구 같은 작업을 떠올리면 됩니다.

**배치가 합리적인 기본값인 이유:**

- **더 단순합니다** — 잡(job)을 실행하고, 정해진 데이터 집합을 처리하면 끝입니다. 이해하기 쉽고, 테스트하기 쉽고, 다시 실행하기도 쉽습니다.
- **더 저렴합니다** — 24시간 내내 돌리는 대신, 연산 자원을 집중된 시간대에 몰아서 사용합니다.
- **더 효율적입니다** — 백만 건의 행을 한꺼번에 처리하는 것이 한 건씩 처리하는 것보다 행당 효율이 훨씬 좋습니다.
- **복구가 쉽습니다** — 배치가 실패했나요? 다시 실행하면 됩니다. 경계가 깔끔하게 나뉘어 있습니다.

**대가는 바로 *latency(지연/레이턴시)*입니다.** 데이터는 마지막으로 실행된 시점만큼만 신선합니다. 파이프라인이 매일 밤 한 번 돈다면, 대시보드는 최대 하루까지 뒤처질 수 있습니다. 하지만 "지난달 우리 실적이 어땠지?" 같은 대부분의 리포팅에는 전혀 문제가 없습니다. 지난달 매출을 초 단위까지 계산해야 하는 사람은 아무도 없으니까요.

> 체화해 둘 만한 직관은 이것입니다. **비즈니스가 그 지연을 감내할 수 있다면, 배치가 거의 항상 옳은 선택입니다.** 더 단순하고 더 저렴하며, 단순함 그 자체가 하나의 장점입니다.

## 3. 스트리밍: 신선한 데이터가 핵심일 때

어떤 사례는 데이터가 낡으면 그 가치를 잃어버립니다. 그런 경우에는 스트리밍이 추가되는 복잡도를 감수할 만합니다.

- **fraud detection(이상 거래 탐지)** — 사기 거래를 *내일* 잡아내 봐야 소용없습니다. *지금* 잡아야 합니다.
- **실시간 대시보드와 모니터링** — 운영 팀이 시스템이나 비즈니스 지표를 실시간으로 지켜보는 경우입니다.
- **real-time personalization(실시간 개인화)** — 사용자가 몇 초 전에 한 행동을 바탕으로 무언가를 추천하는 경우입니다.
- **alerting(알림)** — 센서 임계값 초과나 오류 급증 같은 이벤트에 그 순간 반응하는 경우입니다.

스트리밍에서는 데이터가 생성된 지 **수 초 또는 수 밀리초** 안에 처리됩니다. 이를 가능하게 하는 기술은 보통 **message/event streaming platform(메시지/이벤트 스트리밍 플랫폼)**인데, Apache Kafka가 가장 잘 알려진 예입니다. 이러한 플랫폼은 프로세서가 구독하는, 연속적이고 순서가 보장된 event stream(이벤트 스트림)의 파이프 역할을 합니다.

현업에서 스택은 대개 두 조각으로 나뉩니다. 이벤트는 Apache Kafka 같은 플랫폼을 통해 흐르고(저장과 전달 담당), 실제 처리는 그 위에 올라가는 **stream processor(스트림 프로세서)**가 맡습니다. Apache Flink, Spark Structured Streaming, Kafka Streams가 대표적인 선택지입니다. Kafka가 "데이터를 나르는 컨베이어 벨트"라면, 스트림 프로세서는 그 위에서 집계·조인·필터링을 수행하는 "작업대"라고 보면 됩니다.

```
producers ──► [ event stream (e.g., Kafka) ] ──► stream processors ──► destinations
(apps emit     a continuous, ordered log          (Flink / Spark /        (live dashboards,
 events)       of events                            Kafka Streams)          alerts, stores)
```

여기서 **Change Data Capture**(2부)가 빛을 발합니다. CDC는 데이터베이스의 변경 사항을 바로 이런 이벤트 스트림으로 바꿔 주며, 분석 시스템이 운영 데이터의 변화에 거의 실시간으로 반응할 수 있게 합니다.

## 4. 윈도와 워터마크: 무한 스트림을 다루는 법

스트림은 끝이 없기 때문에, "시간당 집계" 같은 것을 계산하려면 끝없는 흐름을 시간 조각으로 잘라 내는 **window(윈도)**가 필요합니다. 윈도에는 크게 두 종류가 있습니다.

- **tumbling window(텀블링 윈도)** — 겹치지 않는 고정 구간입니다. "매시 정각부터 1시간"처럼 각 이벤트가 정확히 한 윈도에만 속합니다.
- **sliding window(슬라이딩 윈도)** — 겹치는 구간입니다. "지난 1시간을 5분마다 다시 계산"처럼 하나의 이벤트가 여러 윈도에 걸쳐 집계됩니다.

"시간당 주문 건수"를 텀블링 윈도로 집계하는 모습은 SQL 기반 스트림 프로세서(예: Flink SQL)에서 대략 이렇게 보입니다.

```sql
SELECT TUMBLE_START(order_time, INTERVAL '1' HOUR) AS hour,
       COUNT(*) AS orders
FROM order_events
GROUP BY TUMBLE(order_time, INTERVAL '1' HOUR);
```

여기서 핵심 질문이 하나 남습니다. **"한 윈도를 언제 닫고 결과를 내보낼 것인가?"** 이벤트가 늘 제시간에, 순서대로 도착하지는 않기 때문입니다. 이를 다루는 장치가 **watermark(워터마크)**입니다. 워터마크는 "이 시점까지의 이벤트는 (대체로) 다 도착했다고 본다"는 시스템의 판단선이며, 워터마크가 윈도 끝을 지나면 그 윈도를 닫고 집계를 확정합니다. 늦게 도착한 이벤트(late event)는 버리거나, 별도로 보정 처리할지를 정책으로 정해 둬야 합니다.

전달 보장(delivery guarantee)도 함께 정해야 합니다. **at-least-once(최소 한 번)**는 장애 시 같은 이벤트가 중복 처리될 수 있어 집계가 부풀 수 있고, **exactly-once(정확히 한 번)**는 중복을 막아 주지만 상태 저장과 체크포인트 비용이 따릅니다. "건수를 정확히 세야 하는가, 약간의 오차를 감내할 수 있는가"가 둘 사이의 실질적 선택 기준입니다.

## 5. 스트리밍이 더 어려운 이유

스트리밍의 신선함은 복잡도라는 분명한 대가를 치릅니다. 왜 그런지 솔직하게 짚어 볼 가치가 있습니다. 이것이 바로 스트리밍을 기본으로 선택해서는 안 되는 이유이기도 합니다.

- **항상 켜져 있어야 합니다** — 스트리밍 시스템은 24시간 내내 돌아가며 지속적으로 건강한 상태를 유지해야 합니다. 배치 잡은 새벽 2시에 실패해도 아침 8시에 다시 돌리면 되지만, 뒤처진 스트림 프로세서는 곧바로 현재 진행 중인 장애입니다. 즉 누군가의 **온콜(on-call)** 부담으로 직결됩니다.
- **재처리(backfill)가 훨씬 까다롭습니다** — 로직 버그를 고친 뒤 "지난 한 달치를 다시 계산"하는 일은 배치에선 잡을 한 번 더 돌리면 끝이지만, 스트리밍에선 과거 이벤트를 다시 흘려보내면서 워터마크·윈도·상태를 모두 일관되게 맞춰야 합니다. 운영 트래픽과 backfill 트래픽이 섞이지 않게 하는 것만으로도 별도의 설계가 필요합니다.
- **out-of-order data(순서가 뒤바뀐 데이터)와 지연 도착 데이터** — 이벤트가 항상 발생한 순서대로 도착하는 것은 아닙니다. "이 이벤트는 10분 전 것인데, 방금 도착했다"를 처리하는 것은 (앞서 본 워터마크로 일부 완화되지만) 그 자체로 하나의 난제 영역입니다.
- **깔끔한 경계가 없습니다** — 배치에는 "오늘의 데이터"처럼 단정한 경계가 있지만, 스트림은 무한합니다. 어디까지가 "오늘"인지조차 윈도와 워터마크로 직접 정의해 줘야 합니다.
- **추론과 복구가 더 어렵습니다** — "스트림의 지난 한 시간"을 다시 처리하는 일은 배치 잡을 다시 돌리는 것보다 훨씬 지저분합니다.

> 솔직한 규칙은 이것입니다. **스트리밍은 "더 나은 배치"가 아닙니다. 더 높은 운영 비용을 가진, 다른 도구입니다.** 실시간이 멋있어 보여서가 아니라, *신선함이 주는 비즈니스 가치*가 그 비용을 분명히 정당화할 때 도입하십시오.

## 6. 배치와 스트리밍, 나란히 비교

| | 배치 | 스트리밍 |
|---|-------|-----------|
| **처리 단위** | 일정에 따른 묶음 | 도착하는 각 이벤트 |
| **데이터 신선도** | 수 분에서 하루 전 | 수 초 이내 |
| **복잡도** | 낮음 | 높음 |
| **비용 패턴** | 집중 처리(burst) | 항상 가동 + 온콜 |
| **재처리/복구** | 배치를 재실행 | 더 지저분함(backfill, 윈도, 순서 처리) |
| **대표 스택** | 스케줄러 + SQL/웨어하우스 | Kafka + Flink / Spark Structured Streaming / Kafka Streams |
| **적합한 용도** | 리포팅, 대부분의 분석 | 이상 거래, 알림, 실시간 대시보드, 개인화 |
| **기본값으로?** | 그렇습니다, 대부분의 경우 | 신선함이 정말 중요할 때만 |

## 7. 현대의 현실: 둘을 함께

실무에서는 플랫폼 전체에 한쪽만 선택하지 않습니다. 사례별로 선택하며, 성숙한 플랫폼 대부분은 둘을 섞어서 운영합니다. 분석의 대부분은 신뢰할 수 있고 저렴한 배치 파이프라인을 거치고, 진짜로 시간에 민감한 소수의 사례만 스트리밍을 사용합니다. 게다가 점점 *같은* 도구와 저장소(특히 lakehouse(레이크하우스))가 둘 다를 처리할 수 있게 되면서, "배치 시스템"과 "스트리밍 시스템" 사이의 오래된 경계가 흐려지고 있습니다.

그래도 의사 결정의 틀은 단순하게 유지됩니다. 어떤 파이프라인이든 **"이 데이터가 한 시간 낡았을 때의 비용은 얼마인가?"**라고 물어보십시오. 답이 "별것 아니다"라면 배치를 쓰면 됩니다. 답이 "돈을 잃는다 / 사기를 놓친다 / 잘못된 실시간 의사 결정을 내린다"라면, 바로 그 지점에서 스트리밍이 제값을 합니다. 현실적인 결론은 이렇습니다. **대부분은 배치로 가고, 신선함이 실제로 돈이 되는 소수의 케이스에만 스트리밍을 얹으십시오.**

**선택 체크리스트:**

- [ ] 이 데이터가 1시간(혹은 하루) 늦으면 실제로 무엇을, 얼마나 잃는가?
- [ ] 그 비용이 24/7 운영·온콜 부담을 정당화하는가?
- [ ] backfill(과거 재처리) 시나리오가 현실적으로 가능한 설계인가?
- [ ] 집계가 정확해야 하는가(exactly-once), 약간의 오차를 감내할 수 있는가(at-least-once)?
- [ ] 늦거나 순서가 뒤바뀐 이벤트를 어떻게 처리할지 정했는가(워터마크·late event 정책)?

## 8. 핵심 정리 — 그리고 시리즈 마무리

배치는 데이터를 정해진 일정의 묶음 단위로 처리합니다(더 단순하고 더 저렴하며, 대부분의 분석에서 옳은 기본값입니다). 스트리밍은 각 이벤트가 도착하는 대로 처리합니다(더 신선하지만, 항상 가동되어야 하고 더 어렵습니다). Kafka로 이벤트를 나르고 Flink·Spark Structured Streaming·Kafka Streams 같은 프로세서로 윈도와 워터마크를 다루되, 데이터 신선도가 실제로 얼마나 가치 있는지 물어 사례별로 선택하고, 둘 다 운영하게 되리라 예상하십시오.

이것으로 **Moving Data(데이터 이동) 시리즈**를 마칩니다. 한 걸음 물러서서, 우리가 따라온 여정 전체를 살펴봅시다.

```
Part 1  Pipeline overview  — extract, transform, load; the skeleton
Part 2  Ingestion          — getting data in (batch/stream, full/incremental, CDC)
Part 3  ETL vs. ELT        — transform before or after loading
Part 4  Integration        — making many sources agree on one truth
```
