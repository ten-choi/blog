**데이터 플랫폼 시리즈 · 10/15** · Moving Data(데이터 이동) 4/5

> 단일 소스는 쉽습니다. 하지만 실제 조직에는 수십 개의 소스가 있고, 저마다 고유한 형식과 ID를 가지고 있으며 "고객"이 무엇인지에 대한 정의조차 제각각입니다. **data integration(데이터 통합)**은 이들을 서로 일치시키는 작업입니다. 변환(transform) 단계에서 가장 어렵고, 가장 빛이 안 나면서도, 가장 가치 있는 부분입니다.

## 1. 문제: 데이터는 흩어져 있고 일관성이 없습니다

데이터를 한곳에 모아 두는 실제 비즈니스는 없습니다. 영업 데이터는 CRM에, 결제는 빌링 시스템에, 고객 지원은 티켓팅 도구에, 웹 활동은 분석 플랫폼에, 핵심 레코드는 앱 데이터베이스에 들어 있습니다. *"이 고객의 생애 가치(lifetime value)는 얼마인가?"* 같은 단순한 질문 하나에 답하려고 해도, 이 여러 소스의 데이터를 조합해야 합니다.

그리고 막상 조합을 시도하는 순간, 이들이 서로 일치하지 않는다는 사실을 발견하게 됩니다.

- CRM은 고객을 `customer_id`로 부르고, 빌링은 `account_no`로 부르며, 고객 지원은 이메일을 사용합니다.
- 어떤 시스템은 날짜를 `2026-06-09`로 저장하고, 다른 시스템은 `06/09/2026`으로, 또 다른 시스템은 Unix 타임스탬프로 저장합니다.
- CRM은 고객이 "USA"에 있다고 하고, 빌링은 "United States", 고객 지원은 "US"라고 합니다.
- 같은 사람이 세 시스템에 걸쳐 조금씩 다른 세 개의 레코드로 존재합니다.

> **데이터 통합이란 여러 개의 서로 다른 소스에서 나온 데이터를 단일하고 일관되며 통합된 뷰로 결합하는 과정입니다.**

이것은 플랫폼 전체를 잇는 결합 조직(connective tissue)입니다. 통합이 없으면 data silo(데이터 사일로)가 생깁니다. 각각이 이야기의 일부만 들려주는 섬들이고, 그중 어느 것도 전체 이야기를 믿고 맡길 수 없는 상태가 되는 것입니다.

## 2. 왜 진짜로 어려운가

통합은 충돌이 끝도 없고 미묘하기 때문에 보기보다 훨씬 까다롭습니다. 주요 범주는 다음과 같습니다.

**① 서로 다른 식별자.** 동일한 실세계 엔터티가 시스템마다 다른 키를 가집니다. 조인 키가 일치하지 않으면 그냥 테이블을 조인할 수 없습니다. "이 세 레코드는 같은 고객이다"라고 판정하는 것은 그 자체로 하나의 분야입니다(§4 참고).

**② 서로 다른 형식과 단위.** 날짜, 통화, 단위, 인코딩이 전부 다릅니다. 통합이란 모든 것을 공통 표준으로 변환해서 값들이 실제로 비교 가능하도록 만드는 일입니다.

**③ 서로 다른 의미(semantics).** 이것이 가장 깊은 문제입니다. 두 시스템이 같은 *단어*를 서로 다른 *대상*에 사용합니다. "매출(revenue)"에 세금이 포함될까요? 환불은요? "활성 사용자(active user)"는 이번 달에 로그인한 사용자를 뜻할까요, 아니면 한 번이라도 구매한 적이 있는 사용자를 뜻할까요? 이름만 같고 의미는 다른 필드를 무작정 결합하면, 확신에 차서 틀린 숫자를 얻게 됩니다.

**④ 서로 다른 품질과 신선도(freshness).** 한 소스는 실시간으로 갱신되고, 다른 소스는 야간 배치로 갱신됩니다. 하나는 깨끗하고, 다른 하나는 null과 오타투성이입니다. 통합된 데이터는 그 가장 지저분한 입력만큼만 신뢰할 수 있습니다.

> 가장 어려운 통합 문제는 기술적인 것이 아니라 *의미에 대한 합의*입니다. 바로 이 지점에서 데이터 통합은 데이터 거버넌스와 만납니다. 조직이 "매출"이 무엇을 의미하는지 합의하기 전까지는 다섯 시스템의 "매출"을 통합할 수 없습니다. 뒤의 데이터 거버넌스 시리즈에서 다룰 비즈니스 용어집(business glossary)이 바로 통합을 가능하게 만드는 것입니다.

**현업 함정 — 진짜 사고는 의미 불일치에서 납니다.** 형식 차이는 눈에 보이고 결국 고쳐집니다. 하지만 의미 불일치는 조용히 잘못된 숫자를 만들어 냅니다. 한 팀이 `revenue`를 세금 포함·환불 차감 후로 계산하고, 다른 팀이 세금 제외·환불 미반영으로 계산하면, 두 숫자는 둘 다 "맞지만" 결코 일치하지 않습니다. 이건 SQL로 해결할 수 없습니다. 통합에 들어가기 *전에* 조직이 정의에 합의해야 하고(비즈니스 용어집, 거버넌스), 그 합의를 컬럼 정의로 못 박아 두어야 합니다.

또 하나 흔한 함정은 **잘못된 조인 키로 통합하는 것**입니다. 조인 키가 미묘하게 어긋나면 (예: 한쪽은 정규화된 이메일, 다른 쪽은 대문자 섞인 원본 이메일) 에러 없이 조용히 행이 누락되거나 중복됩니다. 통합 후에는 항상 입력 대비 출력 행 수를 확인하세요. 갑자기 줄거나 부풀었다면 키가 어긋난 것입니다.

## 3. 통합 작업의 모습

소스를 결합한다는 것은 실무에서는 변환 단계에서 수행하는 일련의 조정(reconciliation) 작업입니다.

```
  source A ─┐
  source B ─┼─► [ standardize formats ]
  source C ─┘        │
                     ▼
            [ match & resolve entities ]   ← same customer across sources
                     │
                     ▼
            [ reconcile definitions ]      ← one agreed meaning per field
                     │
                     ▼
            [ unified, consistent dataset ]
```

1. **표준화(Standardize)** — 모든 소스를 공통 형식, 단위, 인코딩으로 맞춥니다.
2. **엔터티 매칭(Match entities)** — 소스 전반에서 어떤 레코드들이 동일한 실세계 대상을 가리키는지 알아냅니다.
3. **정의 조정(Reconcile definitions)** — 결합되는 각 필드에 대해 합의된 하나의 의미를 적용합니다.
4. **통합 데이터셋 생성(Produce a unified dataset)** — 다운스트림 소비자가 단일 뷰로 신뢰할 수 있는 데이터셋을 만듭니다.

표준화는 추상적으로 들리지만 실제로는 매우 구체적입니다. 앞서 본 "USA"/"United States"/"US" 문제는 하나의 canonical(표준) 값으로 매핑하는 일입니다.

```sql
SELECT
  CASE upper(trim(country))
    WHEN 'USA'           THEN 'US'
    WHEN 'UNITED STATES' THEN 'US'
    WHEN 'US'            THEN 'US'
    ELSE upper(trim(country))   -- 미처리 값은 그대로 두고 나중에 점검
  END AS country_code
FROM raw_customers;
```

ELSE 분기에서 알 수 없는 값을 버리지 않고 남겨 두는 것이 핵심입니다. 그래야 새로운 표기("U.S.A.", "America")가 들어왔을 때 조용히 사라지지 않고 눈에 띕니다. 매핑이 늘어나면 이런 코드는 보통 별도의 reference 테이블이나 dbt seed로 빼서 관리합니다.

## 4. 엔터티 해소와 마스터 데이터

가장 까다로운 한 조각, 즉 동일한 실세계 대상을 나타내는 레코드들을 매칭하는 작업에는 이름이 있습니다. 바로 **entity resolution(엔터티 해소)**(record linkage 또는 deduplication이라고도 함)입니다. CRM의 "John Smith, jsmith@example.com"과 빌링의 "J. Smith, 같은 이메일"은 *아마도* 같은 사람일 것입니다. 오타와 누락된 필드가 섞인 수백만 건의 레코드에 걸쳐 이를 신뢰성 있게 판정하는 것은 어렵습니다.

실무에서는 보통 두 단계로 접근합니다.

1. **결정론적 매칭(deterministic matching)을 먼저** — 정규화한 이메일이나 전화번호처럼 신뢰할 수 있는 키로 정확히 조인합니다. 소문자화·공백 제거 후 이메일이 같으면 같은 사람으로 봅니다. 대부분의 레코드는 여기서 깔끔하게 해소됩니다. 빠르고, 설명 가능하며, 재현 가능합니다.
2. **남은 것에 대해 퍼지/확률적 매칭(fuzzy/probabilistic matching)** — 결정론적 키가 없거나 어긋난 나머지에 대해 이름 유사도, 주소, 생년월일 등을 점수화해서 같은 엔터티일 확률을 추정합니다. 임계값(threshold)을 정해 자동 병합·보류·수동 검토로 나눕니다.

매칭한 레코드들에는 **안정적인 dedup 키**를 하나 부여합니다. 예를 들어 정규화 이메일에 대한 해시처럼, 재실행해도 같은 그룹에 같은 키가 붙는 값이어야 합니다. 매번 바뀌는 키를 쓰면 다운스트림 조인이 깨집니다. 이 결정론적→확률적 흐름과 안정적 키 부여가 바로 MDM 도구들이 자동화해 주는 부분입니다.

이는 거버넌스 시리즈에서 다시 만나게 될 용어 하나와 연결됩니다. 바로 **master data management(마스터 데이터 관리), 즉 MDM**입니다. MDM은 고객, 제품, 위치 같은 핵심 엔터티에 대해 단일하고 권위 있는 버전을 확립하는 분야로, "이 고객은 누구인가?"에 대해 조직 전체가 공유하는 하나의 표준 답을 갖게 합니다. 통합은 MDM이 *왜* 필요한지를 보여 줍니다. 합의된 마스터 레코드가 하나 없으면, 모든 통합이 "이것들이 같은 것인가?"를 매번 처음부터 다시 따지게 됩니다.

## 5. 흔한 통합 패턴

항상 같은 방식으로 통합하는 것은 아닙니다. 주요 접근법은 다음과 같습니다.

- **consolidation(통합/병합) — 가장 흔한 방식** — 모든 소스의 데이터를 한곳(웨어하우스나 레이크)으로 물리적으로 복사해 와서 그곳에서 결합합니다. 여러분의 파이프라인이 하는 일이 바로 이것이며, 분석(analytics)의 기본값입니다.
- **virtualization(가상화) / federation(페더레이션)** — 데이터를 소스에 그대로 두고 필요할 때 소스들을 가로질러 쿼리하여, 복사 없이 통합된 뷰를 제공합니다. 중복과 오래된 데이터(staleness)를 피할 수 있지만, 소스의 성능과 가용성에 제약을 받습니다. 복사가 비현실적일 때 유용합니다.
- **propagation(전파)** — 데이터가 변경될 때마다 한 시스템에서 다른 시스템으로 밀어 넣어(흔히 2부의 streaming/CDC 기법을 통해) 시스템들을 동기화 상태로 유지합니다.

분석 플랫폼의 경우, **웨어하우스나 레이크로의 병합이 압도적으로 표준**입니다. 소스 시스템에 부담을 주지 않으면서 분석할 수 있는, 빠르고 안정적이며 통합된 복사본을 제공해 주기 때문입니다.

## 6. 보상: 단일 진실 공급원

이 모든 어려움을 견디는 이유는 무엇일까요? 그 목표가 모든 데이터 조직이 좇고 있는 바로 그것이기 때문입니다. 바로 **single source of truth(단일 진실 공급원)**, 즉 모두가 사용하는 단일하고 통합되며 일관되고 신뢰할 수 있는 비즈니스 뷰입니다.

실무에서 이 단일 뷰는 보통 한 번 만들어 둔 **conformed(정합) 모델**로 물질화(materialize)됩니다. 즉 표준화·엔터티 해소·정의 합의를 모두 거친 결과를 `dim_customer` 같은 모델 하나로 굳혀 두고, 모든 팀이 자기 나름의 조인을 다시 짜는 대신 그것을 가져다 씁니다. dbt 같은 도구를 쓰면 이 정합 모델을 버전 관리되는 코드로 정의하고, 테스트로 키 유일성·null·행 수를 지켜낼 수 있습니다. 핵심은 도구가 아니라, 통합 로직이 사람들 머릿속 임시 쿼리가 아니라 *한 곳에* 있다는 점입니다.

통합이 잘 되어 있으면, 재무·마케팅·제품 팀이 모두 같은 고객 수와 같은 매출 수치를 가져오고, 서로 *일치*합니다. 통합이 부실하거나 아예 생략되면, 이 커리큘럼 전체에서 반복적으로 등장하는 악몽이 펼쳐집니다. 다섯 개의 대시보드, 다섯 개의 서로 다른 숫자, 그리고 어느 것이 맞는지 아무도 말할 수 없는 상황 말입니다.

통합은 화려하지 않습니다. 데모에 등장하는 일도 드뭅니다. 하지만 어떤 단일 도구보다도, 사람들이 신뢰하는 데이터 플랫폼과 사람들이 조용히 우회해 버리는 플랫폼을 가르는 것이 바로 통합입니다.

**통합 체크리스트**

1. **표준화** — 날짜·통화·국가 코드 등을 canonical 값으로 매핑했는가? 미처리 값은 버리지 않고 드러나게 두었는가?
2. **엔터티 매칭** — 결정론적 키로 먼저 조인하고, 나머지를 퍼지 매칭했는가? 재실행해도 안정적인 dedup 키를 부여했는가?
3. **정의 합의** — 결합하는 각 지표의 의미를 조직이 합의하고 용어집/컬럼 정의로 못 박았는가?
4. **단일 뷰** — 결과를 conformed 모델 하나로 굳히고, 통합 전후 행 수를 검증했는가?

## 7. 핵심 정리

데이터 통합은 서로 다르고 일관성 없는 여러 소스를 하나의 통합되고 신뢰할 수 있는 뷰로 결합하는 작업입니다. 불일치하는 ID, 형식, 중복 레코드 같은 기술적 충돌(엔터티 해소와 마스터 데이터로 해결)은 실재하지만, 가장 깊은 과제는 **의미에 대한 합의**이며, 바로 이 지점에서 통합은 거버넌스에 기댑니다. 대부분의 분석 플랫폼은 모든 것을 웨어하우스나 레이크로 **병합(consolidation)**하는 방식으로 통합하며, 그 보상은 conformed 모델로 물질화되는 **단일 진실 공급원**입니다.

지금까지 데이터를 들여오는 일(2부), 언제 변환할지(3부), 그리고 소스를 결합하는 일(4부)을 다뤘습니다. (통합된 데이터가 어떤 *형태*를 가져야 하는지 설계하는 데이터 모델링은 저장과 아키텍처 트랙의 data warehouse(데이터 웨어하우스) 1·2부에서 다룹니다.) 다음 **5부: 배치 대 스트리밍(batch vs. streaming)**에서는 이 모든 일이 정해진 일정에 따라 일어나는지, 아니면 끊임없이 일어나는지를 다룹니다.
