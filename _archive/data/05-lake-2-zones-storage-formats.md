> 1부에서는 데이터 레이크(data lake)가 왜 존재하는지를 설명했습니다. 이번 글에서는 그 뚜껑을 열어 봅니다. 레이크가 내부적으로 실제 어떻게 구성되는지, 무엇 위에 세워지는지, 그리고 어떤 파일 포맷(file format)을 선택하느냐에 따라 쿼리가 어떻게 열 배 빨라지거나 느려질 수 있는지를 다룹니다.

## 1. 레이크에는 zone(존)이 필요합니다. 그렇지 않으면 늪이 됩니다

1부는 "data swamp(데이터 늪)" — 아무도 무엇을 찾거나 신뢰할 수 없는 레이크 — 에 대한 경고로 끝맺었습니다. 이를 막는 첫 번째 방어선은 구조이며, 표준적인 구조는 레이크를 여러 개의 **zone(존)** 으로 나누는 것입니다(layer(레이어)라고 부르기도 하고, 금속이나 온도에 빗대어 이름을 붙이기도 합니다). 데이터는 한 zone에서 다음 zone으로 흐르면서 각 단계마다 점점 더 깨끗하고 정제된 형태가 됩니다.

흔히 쓰이는 세 개의 zone 모델은 다음과 같습니다.

| Zone | 다른 이름 | 무엇이 담기는가 | 누가 사용하는가 |
|------|-------------|--------------|-------------|
| **raw**(원본) | landing, bronze | 도착한 그대로, 손대지 않은 데이터 | 파이프라인, 드물게 이뤄지는 심층 조사 |
| **cleansed**(정제) | staging, silver | 검증·중복 제거·표준화가 끝난 데이터 | 엔지니어, 데이터 사이언티스트 |
| **curated**(가공) | serving, gold | 비즈니스에 바로 쓸 수 있는, 집계·모델링된 데이터 | 분석가, 대시보드, ML 피처 |

흐름은 다음과 같은 모습입니다.

```
sources ──► [ RAW ] ──► [ CLEANSED ] ──► [ CURATED ] ──► dashboards / ML
            untouched    cleaned          business-ready
```

이를 작동하게 만드는 두 가지 원칙이 있습니다.

**raw zone은 절대 버리지 않습니다.** 스토리지는 저렴하고(그것이 레이크의 핵심 명분입니다), 원본 데이터를 보관해 두면 버그를 발견하거나 새로운 용도를 떠올렸을 때 언제든 다시 처리할 수 있습니다. raw zone은 여러분의 안전망이자 단일 진실 공급원(single source of truth)입니다.

**각 zone은 바로 앞 zone으로부터 파생됩니다.** curated 데이터는 cleansed로부터 만들어지고, cleansed는 raw로부터 만들어집니다. 하류(downstream)에서 무언가 잘못된 것처럼 보인다면, zone을 거슬러 올라가며 어디서 어긋났는지 찾아낼 수 있습니다. lineage(리니지)가 물리적인 형태로 구현된 셈입니다.

실무에서는 이 두 원칙을 버킷(bucket) 경계로 못박아 둡니다. raw zone에는 쓰기 전용·읽기 제한 권한을 걸어 두고(아무도 직접 손대지 못하게), 정제 로직은 전부 cleansed/curated를 *재생성하는* 잡(job)으로만 만드는 식입니다. 흔한 함정 하나: 누군가 "급하다"는 이유로 raw 데이터를 직접 수정하기 시작하면, 그 순간 단일 진실 공급원은 무너지고 레이크는 다시 늪을 향해 갑니다.

## 2. 레이크가 실제로 무엇 위에 세워지는가: object storage(오브젝트 스토리지)

1부에서는 레이크를 "거대하고 저렴한 폴더"라고 표현했습니다. 이제 이를 좀 더 정확하게 짚어 보겠습니다. 현대의 데이터 레이크는 **object storage(오브젝트 스토리지)** 위에 세워집니다. Amazon S3, Google Cloud Storage, Azure Data Lake Storage 같은 시스템들이죠. (개념은 어느 제공자든 동일하며, 특정 한 곳에 종속되는 이야기가 아닙니다.)

object storage는 여러분에게 익숙한 파일 시스템과 다음과 같은 점에서 다릅니다. 그리고 그 차이는 레이크에 중요합니다.

- **계층형이 아니라 평면형(flat)입니다.** 진짜 폴더라는 것이 없습니다. 여러분은 `raw/orders/2026/06/09/data.parquet` 같은 키(key) 아래에 *오브젝트*(파일)를 저장합니다. "폴더"란 키에 들어간 명명 규칙일 뿐입니다. 다만 도구들이 그 규칙에 크게 의존한다는 점이 중요합니다.
- **저렴하고 사실상 무제한입니다.** 저장한 기가바이트당, 그리고 요청당 비용을 지불하는데, 데이터베이스 스토리지 비용의 극히 일부에 불과합니다. 페타바이트 규모로 확장하는 일은 아키텍처 프로젝트가 아니라 청구서상의 사건일 뿐입니다.
- **스토리지와 컴퓨팅이 분리되어 있습니다.** 데이터는 그저 그 자리에 놓여 있고, 얼마든지 많은 별도의 컴퓨팅 엔진이 이를 읽을 수 있습니다. 이 분리(decoupling)야말로 스토리지와 컴퓨팅이 한 몸으로 붙어 있는 웨어하우스(warehouse)에 대한 핵심적인 경제적 이점입니다.

이 마지막 항목은 좀 더 곱씹어 볼 가치가 있습니다. 스토리지와 컴퓨팅이 분리되어 있기 때문에, SQL 엔진, Spark 잡, ML 학습 실행을 *같은 파일들*에 복사 없이 동시에 겨눌 수 있고, 무언가가 실제로 돌아갈 때만 컴퓨팅 비용을 지불하면 됩니다.

다만 "평면형"이라는 점이 실무에서 자주 함정이 됩니다. 요청당 과금이라는 특성 때문에, 평면 키 공간(key space)에 수많은 자잘한 오브젝트가 쌓이면 단순히 "무엇이 있는지" 나열(listing)하는 비용조차 무시할 수 없게 됩니다. 다음 절의 파티셔닝과 파일 크기 이야기가 바로 이 비용과 직결됩니다.

## 3. partitioning(파티셔닝): 가장 값싼 성능 향상

레이크는 결국 키 아래에 놓인 파일들일 뿐이므로, *파일을 어떻게 배치하느냐*가 쿼리 속도에 직접적인 영향을 줍니다. 핵심 기법은 **partitioning(파티셔닝)** 으로, 자주 필터링하는 컬럼(가장 흔하게는 날짜)을 기준으로 파일을 하위 폴더로 정리하는 것입니다.

```
raw/orders/year=2026/month=06/day=09/data.parquet
raw/orders/year=2026/month=06/day=10/data.parquet
```

이제 "2026-06-09의 주문"을 찾는 쿼리는 다른 모든 폴더를 통째로 건너뛸 수 있습니다. 수년치 데이터를 훑는 대신 하루치 파일만 읽는 것이죠. 이를 **partition pruning(파티션 프루닝)** 이라고 하며, 레이크에서 단일 항목으로는 가장 큰 성능 지렛대인 경우가 많습니다.

주의할 점이 있습니다. 과도하게 파티셔닝하지 마세요. 카디널리티(cardinality)가 높은 컬럼(예: `user_id`)으로 쪼개면 수백만 개의 자잘한 파일이 생기고, 자잘한 파일은 나열하고 읽는 데 느립니다. 실제로 필터링에 쓰면서 카디널리티가 적당한 값으로 파티셔닝하세요. 날짜가 전형적인 안전한 선택입니다.

### 작은 파일 문제(small files problem)

실무에서 가장 자주 부딪히는 함정이 바로 **작은 파일 문제(small files problem)** 입니다. 레이크의 엔진들은 파일 하나하나마다 메타데이터를 읽고, 열고, 닫고, 스케줄링하는 고정 비용을 치릅니다. 그래서 같은 1GB라도 1~10KB짜리 파일 수만 개로 흩어져 있으면, 적당한 크기의 파일 몇 개로 묶여 있을 때보다 훨씬 느리고 비쌉니다 — 정작 데이터는 한 줄도 더 읽지 않았는데 말이죠. object storage의 나열(listing)·요청 비용까지 폭증합니다.

경험칙은 단순합니다. **목표 파일 크기를 대략 수백 MB** 정도로 잡으세요(엔진의 블록 단위와 잘 맞습니다). 그리고 앞서 본 `user_id`처럼 카디널리티가 높은 컬럼으로 파티셔닝하면, 파티션마다 레코드 몇 개씩 들어간 수백만 개의 초소형 파일이 만들어진다는 점을 기억하세요. 이것이 작은 파일 문제를 일으키는 가장 흔한 원인입니다.

이미 자잘한 파일이 쌓였다면 실무적 처방은 **compaction(컴팩션, 작은 파일 병합)** 입니다. 주기적으로 잡을 돌려 한 파티션의 작은 파일 여러 개를 적정 크기 파일 하나로 다시 써 주는 것이죠. 스트리밍처럼 작은 파일이 계속 흘러드는 파이프라인에서는 이 컴팩션을 정기 작업으로 아예 일정에 넣어 두는 경우가 많습니다.

## 4. 파일 포맷: 쿼리의 승패가 갈리는 지점

이 부분이야말로 엔지니어들이 가장 자주 잘못 다루는 지점입니다. 기본값 — 그냥 CSV나 JSON으로 쏟아붓는 것 — 이 조용히 큰 대가를 치르게 하기 때문입니다. 파일 포맷은 디스크에서 얼마나 많은 데이터를 읽어 들이는지, 그리고 얼마나 빠르게 처리할 수 있는지를 결정합니다. 여기서 중요한 갈림길은 **row-based(행 기반) 대 columnar(컬럼 기반)** 입니다.

### row-based(행 기반): CSV, JSON, Avro

이들은 데이터를 한 번에 한 레코드씩, 한 행의 모든 필드를 함께 저장합니다. 레코드를 쓰거나 행 전체를 통째로 옮기는 데에는 훌륭합니다. 분석 측면에서의 문제는, `SUM(amount)`를 계산하려면 엔진이 필요한 단 하나의 컬럼에 접근하기 위해 *모든 행의 모든 컬럼*을 읽어야 한다는 점입니다.

- **CSV / JSON** — 사람이 읽을 수 있고 보편적이지만, 부피가 크고 쿼리가 느립니다. 작은 데이터와 교환(interchange) 용도로는 괜찮지만, 큰 분석용 테이블에는 좋지 않은 선택입니다.
- **Avro** — schema evolution(스키마 진화)에 강점을 가진 컴팩트한 바이너리 행 포맷입니다. 레코드를 끊임없이 통째로 기록하는 스트리밍 및 메시지 파이프라인에서 흔히 쓰입니다.

### columnar(컬럼 기반): Parquet, ORC

이들은 데이터를 한 번에 한 *컬럼*씩 저장합니다. 모든 `amount` 값을 함께, 모든 `country` 값을 함께 모아 두는 식이죠. 이는 분석에 있어 판도를 바꿉니다.

- **필요한 컬럼만 읽습니다.** `SUM(amount)`는 `amount` 컬럼만 읽고 나머지는 무시합니다. 폭이 넓은 테이블에서는 이것만으로도 읽는 데이터가 10배 줄어들 수 있습니다.
- **압축이 훨씬 뛰어납니다.** 한 컬럼 안의 값들은 서로 비슷하고(모두 금액, 모두 날짜), 비슷한 데이터는 뒤섞인 행보다 극적으로 잘 압축됩니다.
- **통계 정보가 내장되어 있습니다.** 각 청크(chunk)가 최소/최대 값을 저장하므로, 엔진은 필터와 맞을 수 없는 블록 전체를 건너뛸 수 있습니다. 이 또한 읽는 양을 크게 줄여 줍니다.

**Apache Parquet** 는 레이크의 분석용 데이터에서 사실상의 표준이며, **ORC** 는 이에 견줄 만하고 Hadoop/Hive 생태계에서 흔합니다. 어느 정도 규모가 있는 분석용 테이블이라면 columnar가 거의 항상 옳은 선택입니다.

| 포맷 | 레이아웃 | 적합한 용도 | 피해야 할 용도 |
|--------|--------|----------|-----------|
| CSV / JSON | row | 교환, 작은 데이터, 디버깅 | 큰 분석용 테이블 |
| Avro | row | 스트리밍, 레코드 단위 쓰기 | 컬럼 선택적 분석 |
| Parquet / ORC | columnar | 분석, 큰 테이블, 집계 | 자잘한 파일, 끊임없는 단일 행 읽기 |

### 실무 파이프라인 패턴: raw는 그대로, 변환 단계에서 Parquet으로

실무에서 이를 풀어내는 정석적인 패턴이 있습니다. **raw zone은 도착한 형태 그대로(CSV/JSON) 보존**하고 — 원본을 손대지 않는다는 1절의 원칙 그대로입니다 — 별도의 **변환(transform) 단계**에서 cleansed/curated용으로 Parquet으로 바꿔 쓰는 것입니다. 이렇게 하면 원본을 *그대로 보관하면서도* 하류에서 빠른 쿼리를 얻을 수 있습니다.

그 변환 단계는 보통 이렇게 한 줄로 끝납니다. 컬럼 포맷으로 쓰면서 파티션 키까지 함께 지정하는 식이죠.

```python
# raw(JSON)를 읽어 날짜로 파티셔닝된 Parquet으로 기록
df = spark.read.json("s3://lake/raw/orders/2026/06/09/")
df.write.partitionBy("dt").parquet("s3://lake/cleansed/orders/")
```

여기서 `partitionBy("dt")`가 3절의 날짜 파티셔닝을, `.parquet(...)`이 이번 절의 columnar 저장을 한 번에 담당합니다. 단, 이 한 줄이 자칫 작은 파일 문제를 만들 수도 있으니(파티션·태스크 수가 많으면 조각이 잘게 쪼개집니다) 쓰기 전에 적정 크기로 묶어 주는 것을 잊지 마세요.

### 존(zone) 설계 체크리스트

새 레이크나 새 데이터셋을 설계할 때 빠르게 점검할 항목입니다.

- [ ] **raw 보존** — raw zone은 도착한 형태 그대로 두고, 직접 수정은 막아 두었는가?
- [ ] **각 존은 파생** — cleansed/curated는 앞 존을 *재생성하는* 잡으로만 만들어지는가?
- [ ] **Parquet 변환** — 분석용(cleansed/curated)은 columnar(Parquet/ORC)로 저장하는가?
- [ ] **적절한 파티션 키** — 실제 필터링에 쓰는 적당한 카디널리티 컬럼(보통 날짜)으로 파티셔닝하고, 목표 파일 크기를 수백 MB로 맞췄는가?

## 5. 그림을 하나로 맞춰 보기

잘 운영되는 레이크는 파일 더미가 아닙니다. 그것은 다음과 같습니다.

1. **zone으로 나뉘어 있습니다(zoned)** — raw → cleansed → curated. 그래서 데이터는 점진적으로 더 신뢰할 만해지고, raw 원본은 결코 사라지지 않습니다.
2. **object storage 위에 세워집니다** — 저렴하고, 거의 무한하며, 스토리지가 컴퓨팅으로부터 분리되어 있습니다.
3. **partitioning되어 있습니다** — 쿼리가 필요 없는 데이터를 건너뛸 수 있도록 배치되고, 작은 파일 문제를 피하도록 적정 크기로 유지됩니다.
4. **columnar로 저장됩니다** — 분석용이라면 무엇이든 Parquet/ORC로 저장하여, 쿼리가 더 적게 읽고 더 빠르게 실행됩니다.

이 네 가지를 제대로 갖추면 빠르고 저렴하며 확장 가능한 토대를 얻게 됩니다. 그러나 여전히 빠진 것이 무엇인지 주목해 보세요. 여기에는 *transaction(트랜잭션)* 이라는 개념이 없습니다. 두 잡이 같은 데이터에 동시에 쓰거나, 잡이 중간에 실패하면, 손상되거나 절반만 쓰인 결과가 나올 수 있습니다. 단순한 파일들은 그런 상황으로부터 여러분을 지켜 주지 못합니다.

**바로 그 빈틈이 3부의 주제입니다** — **lakehouse(레이크하우스)** 와 오픈 테이블 포맷(Delta Lake, Iceberg, Hudi)입니다. 이들은 방금 설명한 바로 그 저렴한 object storage 위에 웨어하우스급 신뢰성(ACID 트랜잭션, 스키마 강제, time travel(타임 트래블))을 더해 줍니다.
