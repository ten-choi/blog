**데이터 플랫폼 시리즈 · 07/15** · Moving Data(데이터 이동) 1/5

> 데이터는 한 곳에서 태어나 다른 곳에서 분석됩니다. 그 간극을 가로질러 데이터를 실어 나르고, 그 과정에서 정제하고 형태를 바꾸는 시스템이 바로 data pipeline(데이터 파이프라인)입니다. 이 시리즈는 그 여정에 관한 이야기입니다. 1부에서는 전체 그림을 먼저 그려 봅니다.

## 1. 파이프라인이 존재하는 이유, 그 간극

플랫폼 개요에서 우리는 데이터가 *생성되는* 곳(앱, API, 이벤트)에서 *분석되는* 곳(warehouse, 대시보드, 모델)까지 한 조각의 데이터를 따라가 보았습니다. 이 두 장소는 거의 언제나 서로 다른 시스템이며, 데이터가 곧바로 쓸 수 있는 형태로 도착하는 일도 거의 없습니다. 그래서 무언가가 다음을 해 줘야 합니다.

- source(소스)에서 데이터를 꺼내고,
- 정제하고, 형태를 바꾸고, 다른 데이터와 결합하고,
- 목적지로 전달하고,
- 그것도 일정에 맞춰 혹은 끊임없이, 사람이 손대지 않고 안정적으로.

그 "무언가"가 바로 **data pipeline(데이터 파이프라인)**입니다.

> **data pipeline(데이터 파이프라인)**은 하나 이상의 source에서 목적지로 데이터를 옮기면서 그 과정에서 데이터를 변환하는, 자동화된 일련의 단계입니다.

핵심은 *자동화*라는 단어에 있습니다. CSV를 내보내 한 번 손으로 정리하는 일은 누구나 할 수 있습니다. 파이프라인은 바로 그 작업을 **반복적이고 안정적으로** 해냅니다. 매시간이든, 매일 밤이든, 매초든 사람의 손을 거치지 않고 수행하며, 무언가 잘못되면 알려 줍니다. 실무에서는 "손으로 한 번 돌리던 스크립트"가 "매일 아침 8시에 무조건 돌아 있어야 하는 무언가"로 승격되는 순간, 그것을 파이프라인으로 옮기게 됩니다.

---

## 2. 파이프라인의 해부학

아무리 화려한 파이프라인이라도 거의 대부분은 동일한 세 종류의 단계로 이루어져 있습니다.

```
 SOURCE ──►  [ extract ] ──► [ transform ] ──► [ load ]  ──► DESTINATION
              pull data       clean & reshape   deliver       (warehouse,
              out             it                it            lake, etc.)
```

- **Extract(추출)** — source(데이터베이스, API, 파일, 이벤트 스트림)에서 데이터를 읽어 냅니다.
- **Transform(변환)** — 원시 데이터를 쓸모 있는 데이터로 바꿉니다. 잘못된 값을 정제하고, 포맷을 변환하고, 다른 source와 join하고, 집계하고, 비즈니스 규칙을 적용합니다.
- **Load(적재)** — 그 결과를 실제로 사용될 목적지에 씁니다. 목적지는 보통 Snowflake나 BigQuery 같은 warehouse, 혹은 data lake가 됩니다.

눈치채셨겠지만 이 세 단어의 머리글자가 바로 **ETL**(Extract, Transform, Load)입니다. ETL은 이 전체 형태를 일컫는 고전적인 이름입니다. 변환이 적재 전에 일어나느냐 후에 일어나느냐, 즉 ETL이냐 **ELT**냐는 이 시리즈에서 별도의 글로 다룰 만큼 중요한 변형입니다(3부). 지금은 그저 이 세 단계 골격만 머릿속에 담아 두시면 됩니다.

---

## 3. 두 가지 리듬: batch와 streaming

파이프라인은 근본적으로 두 가지 리듬 중 하나로 동작하며, 이 선택이 파이프라인의 나머지 모든 것을 결정합니다.

- **Batch(배치)** — 일정에 따라 동작하며, 한 번에 데이터를 *덩어리* 단위로 처리합니다(예: "매일 밤, 오늘의 주문을 처리한다"). 더 단순하고 더 저렴하며, 대부분의 리포팅에 적합한 기본 선택지입니다.
- **Streaming(스트리밍)** — 끊임없이 동작하며, 각 이벤트를 *도착하는 즉시* 처리합니다(예: "주문이 발생하는 순간 실시간 대시보드를 갱신한다"). 데이터가 더 신선하지만, 복잡도와 비용이 더 큽니다.

세상의 대부분은 batch로 돌아가며, streaming은 신선도가 정말로 중요한 경우를 위해 아껴 둡니다. 실무에서 흔한 함정은 "실시간이면 좋겠다"는 막연한 바람만으로 streaming을 고르는 것입니다. 정작 대시보드는 하루 한 번만 보면서 말이지요. 이 구분은 시리즈의 5부를 통째로 할애할 만큼 중요합니다. 지금은 "이 파이프라인은 얼마나 자주 동작하는가?"라는 질문에 크게 두 가지 답이 있고, 그 답이 서로 매우 다른 설계로 이어진다는 점만 알아 두시면 됩니다.

## 4. 그냥 동작하는 것을 넘어, *좋은* 파이프라인이란

데이터를 한 번 옮기는 파이프라인은 쉽습니다. 운영 환경에서 *신뢰할 수 있는* 파이프라인은 데이터의 지저분한 현실을 감당해 내야 하며, 다음 자질들이 장난감과 진짜를 가릅니다.

- **Reliability(신뢰성)** — 일정에 맞춰 동작하고 실패에서 복구합니다. 어젯밤 실행이 죽었다면, 오늘 밤 실행이 하루치 데이터를 조용히 건너뛰어서는 안 됩니다.
- **Idempotency(멱등성)** — 두 번 실행해도 중복 행이 아니라 동일한 결과가 나옵니다. 재실행은 끊임없이 일어나므로, 파이프라인은 반복해도 안전해야 합니다. 실무에서 이를 보장하는 방법은 대체로 두 가지입니다. 하나는 **partition overwrite(파티션 덮어쓰기)** — 그날 파티션을 통째로 다시 쓰는 방식이고, 다른 하나는 key 기준의 **MERGE/upsert** 입니다.

  ```sql
  -- 키가 같으면 갱신, 없으면 삽입 (재실행해도 중복 없음)
  MERGE INTO orders t USING staging s ON t.order_id = s.order_id
  WHEN MATCHED THEN UPDATE SET ...
  WHEN NOT MATCHED THEN INSERT ...
  ```

  단순히 `INSERT INTO ... SELECT`로 적재하면, 재실행 한 번에 어제 행이 그대로 한 벌 더 쌓입니다.
- **Monitoring(모니터링) 및 alerting(알림)** — 무언가 잘못되면, 이해관계자가 대시보드가 틀렸음을 알아채기 *전에* 담당자가 먼저 알게 됩니다. 실무에서 실제로 알림을 거는 대상은 대체로 정해져 있습니다. **실행 실패**(잡이 죽거나 timeout), **신선도 SLA 초과**(예: "어제 데이터가 아침 9시까지 안 들어옴"), 그리고 **행수 급변**(평소 100만 행인데 오늘 0행, 혹은 10배)입니다.
- **Data quality(데이터 품질) 검증** — 잘못된 데이터는 리포트를 오염시킨 뒤가 아니라 이동 중에 걸러집니다. (이는 뒤의 데이터 거버넌스 시리즈에서 자세히 다룰 데이터 품질 규율을 파이프라인 내부에 적용한 것입니다.)
- **Scalability(확장성)** — 데이터 양이 10배로 늘어나도 계속 동작합니다.

> 유용한 직관 하나: 파이프라인은 정상 경로(happy path)에서가 아니라 가장자리에서 실패합니다. 잠깐 다운된 source, 형식이 깨진 레코드, 이전 실행과 겹쳐 버린 실행 같은 것들 말입니다. 좋은 파이프라인 설계의 대부분은 바로 그런 경우를 우아하게 처리하는 일에 관한 것입니다.

**신뢰할 수 있는 파이프라인 체크리스트**

- [ ] 재실행해도 안전한가? (partition overwrite 또는 key 기준 MERGE/upsert)
- [ ] 어젯밤 실행이 죽었으면 자동으로 따라잡거나, 적어도 빈 구멍을 남기지 않는가?
- [ ] 실패·신선도 SLA 초과·행수 급변에 알림이 걸려 있는가?
- [ ] 입력 데이터의 기본 품질(필수 컬럼, 행수 범위)을 적재 *전에* 검증하는가?
- [ ] 데이터가 10배가 되어도 같은 방식으로 동작하는가?

## 5. Orchestration(오케스트레이션): 파이프라인에도 지휘자가 필요하다

파이프라인 하나는 단순합니다. 그러나 실제 플랫폼에는 의존 관계를 가진 *수십 개*의 파이프라인이 있습니다. "주문을 적재하되, *고객이 적재된 후에만*; 그다음 매출 리포트를 만들되, 둘 다 끝난 후에만."

이러한 의존 관계, 일정, 재시도, 실패 처리를 관리하는 것이 바로 **orchestrator(오케스트레이터)**의 일입니다. 무엇이 어떤 순서로 실행되어야 하는지 알고, 파이프라인이라는 거대한 망 전체를 정상 궤도에 유지해 주는 도구이지요. 대표적인 도구가 **Apache Airflow**입니다. Airflow에서는 작업들 사이의 의존 관계를 **DAG**(directed acyclic graph, 방향성 비순환 그래프)로 선언합니다.

```python
# "고객 → 주문 → 매출" 순서를 선언하면 나머지는 orchestrator가 챙긴다
load_customers >> load_orders >> build_revenue
```

이렇게 무엇이 무엇에 의존하는지만 선언하면, 시스템이 순서를 잡고 실패 시 재시도하며 부분 실행을 추적해 줍니다. (Airflow 외에도 **Dagster**, **Prefect** 같은 대안이 있고, 발상은 모두 동일합니다.) 이 개념을 이해하는 데 특정 제품이 필요하지는 않습니다. 무언가가 지휘자 역할을 해 주어야 하며, 그러지 않으면 파이프라인은 아무도 머릿속에서 따라갈 수 없는 cron job 뭉치가 되어 버립니다. 실무에서 "왜 이 리포트가 비어 있지?"의 범인은 의외로 cron 두 개의 실행 시각이 어긋난 경우가 많습니다. orchestrator는 바로 그 시각 의존성을 시각이 아니라 *완료 여부*로 바꿔 줍니다.

## 6. 이 시리즈가 향하는 곳

이제 골격이 손에 잡혔습니다. 파이프라인은 source에서 목적지로 데이터를 **추출하고, 변환하고, 적재하며(extract, transform, load)**, **batch 또는 streaming** 리듬으로 동작하고, **신뢰성, 멱등성, 모니터링, 품질 검증**을 통해 신뢰를 얻습니다. 그리고 이 모든 것을 **orchestrator(오케스트레이터)**가 조율합니다.

시리즈의 나머지는 그 여정의 각 부분을 확대해 들여다봅니다.

- **2부 — data ingestion(데이터 수집).** "추출" 단계를 깊이 다룹니다. source에서 데이터를 실제로 *꺼내는* 방법과 이를 잘 해내기 위한 패턴(change data capture 포함)을 살펴봅니다.
- **3부 — ETL vs. ELT.** 변환 단계의 큰 논쟁입니다. 적재 *전에* 변환할 것인가 *후에* 변환할 것인가, 그리고 지난 10여 년간 그 답이 왜 뒤집혔는가를 다룹니다.
- **4부 — data integration(데이터 통합).** 지저분한 여러 source를 하나의 일관된 뷰로 결합하는, "변환" 단계에서 가장 어려운 부분입니다.
- **5부 — Batch vs. streaming.** 두 리듬을 깊이 들여다보고, 어떻게 선택할지 다룹니다.

> 데이터가 안착하는 *형태*를 설계하는 데이터 모델링(정규화/역정규화, 차원 모델링, star schema)은 저장과 아키텍처 트랙의 data warehouse(데이터 웨어하우스) 1·2부에서 다룹니다.

이 시리즈가 끝날 무렵이면, 파이프라인을 블랙박스가 아니라 일련의 의도적인 선택들로 보게 될 것입니다. 그리고 그 선택 하나하나를 어떻게 내릴지 알게 될 것입니다.
